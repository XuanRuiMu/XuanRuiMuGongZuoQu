# 方案审查

在方案/架构/设计实施前进行对抗性审查，提前发现致命风险、边界遗漏与兼容性问题。内置快速模式与深度模式，由触发关键词自动选择。

## 使用前提

调用本技能前必须满足：

- 已有成形方案：包含目标、主干设计、关键选型理由。
- 主代理已形成初步结论（深度模式必需，供独立审计者审视推理质量）。
- 方案可被独立阅读；若散落在对话中，先固化成文档或结构化文本。

不满足时停止并说明缺什么，不要硬启动。

## 模式选择

流程开始时检查用户措辞：

- 快速模式关键词：验证方案、方案审查、架构验证、实施前验证、找漏洞、方案前置验证、反对者审查。
- 深度模式关键词：多视角审查、对抗审查、交叉验证、关键决策审查、多视角对抗、四角色审查、peer review、独立审计。

匹配快速模式关键词 → 使用快速模式；匹配深度模式关键词 → 使用深度模式。两者皆匹配或皆不匹配时，用 AskUserQuestion 询问用户选择。

## 快速模式

适用于中小型方案或常规实施前排雷。启动 1 个反对者子代理，从可行性、边界条件、兼容性、性能、安全五维度逐项审视。

### 审查维度

| 维度     | 审查重点                                                                                             |
| -------- | ---------------------------------------------------------------------------------------------------- |
| 可行性   | 核心假设是否成立？依赖的外部能力是否真实可用？是否有未验证的技术前提？                               |
| 边界条件 | 极端输入、空值、超时、并发竞争、资源耗尽、部分失败等边界场景                                         |
| 兼容性   | 与现有系统集成点是否对齐？数据格式/协议/版本是否兼容？是否破坏既有契约？向后兼容与迁移路径是否可行？ |
| 性能     | 关键路径吞吐/延迟/资源占用；热点是否会成为瓶颈；是否会随数据量增长劣化                               |
| 安全     | 输入验证、权限边界、凭据处理、日志脱敏、SQL注入、越权访问                                            |

### 快速模式流程

1. 固化方案：确认满足使用前提。
2. 启动反对者：使用 Task 子代理，subagent_type="general_purpose_task"。
3. 收集风险报告。
4. 主代理去重合并、标注处置建议、给出最终裁决。
5. 用 AskUserQuestion 报告结果并询问下一步。

### 快速模式提示词

```text
先读 AGENTS.md。

你是一个反对者，审查以下尚未实施的方案。先用自己的话复述方案核心思路（1-2句），确认你准确理解了方案，而非攻击错误靶子。

然后从以下维度逐项审视：可行性、边界条件、兼容性、性能、安全。

对每个发现的风险，给出具体的触发场景（在什么输入/什么并发条件/什么数据规模下会出问题）、影响范围与缓解建议，禁止泛泛而谈。

【方案】
{方案全文或文档路径}

【输出结构】
1. 方案概要：用1-2句话概括方案核心思路
2. 风险清单：
   - 致命风险（会导致方案完全不可行）
   - 重大风险（会导致方案部分失败或需要大幅调整）
   - 一般风险（需要额外处理但可控）
   每条风险必须包含：风险描述、触发场景、影响、缓解建议。
3. 遗漏场景：方案可能没有考虑到的边界情况或使用场景
4. 改进建议：针对发现的风险，给出具体改进方向
5. 总体裁决：推进 / 有条件推进（列出前置条件） / 不推进（列出致命风险），并说明理由

【关键原则】
目标是帮方案提前排雷，而非证明方案不可行。遵循 AGENTS.md'批判而非全盘否定'原则。对每个风险给出具体触发场景，禁止抽象风险。

你可以使用 Read/Grep/Glob 探索代码库以验证方案与现有系统的兼容性。
```

### 快速模式最终报告

1. 方案概要
2. 风险清单（按致命/重大/一般分级，每条含触发场景、影响、缓解建议）
3. 遗漏场景
4. 改进建议
5. 总体裁决（推进 / 有条件推进 / 不推进）

## 深度模式

适用于关键决策、架构选型、高风险方案验证。启动 4 个角色并行审查：严谨审查者、反对者、独立审计者、合成者。前 3 个角色并行运行，合成者在前 3 个返回后启动。

### 角色定义

#### 1. 严谨审查者

审查方案/代码本身的对错，按严重程度分级找问题。重点：正确性、安全性、可维护性、边界条件、错误处理。

输出结构：

```text
## 改动/方案意图复述
（一句话）

## 发现问题
🔴 严重（必须修复才能推进）：
- [问题] [触发场景] [建议]

🟠 重要（应修复，可附条件推进）：
- [问题] [触发场景] [建议]

🟡 次要（建议修复）：
- [问题] [建议]

🟢 建议（可选优化）：
- [建议]

## 总体评价
（一句话：是否可推进 / 需要修改 / 需要重做）
```

#### 2. 反对者

对方案/架构找漏洞。重点：可行性、边界条件、兼容性、长期维护成本、替代方案。

输出结构：

```text
## 方案核心思路复述
（一句话）

## 风险清单
🔴 致命风险（方案在此场景下会失败）：
- [风险] [触发场景：当……时] [影响] [缓解建议]

🟠 重大风险（需额外措施才能规避）：
- [风险] [触发场景] [影响] [缓解建议]

🟡 一般风险（需关注但不阻塞）：
- [风险] [触发场景] [缓解建议]

## 整体可行性判断
（一句话：方案整体可行 / 需重大调整 / 不可行）
```

#### 3. 独立审计者

批判性审视主代理初步结论的推理质量，找盲区、逻辑漏洞、过度推断。

输出结构：

```text
## 原结论概要
（主代理结论的核心观点）

## 共识（原结论中可靠的部分）
- [观点] [为何可靠]

## 分歧（原结论中存疑的部分）
- [存疑点] [原结论怎么说] [为何存疑] [更合理的判断]

## 盲区（原结论完全没考虑的维度）
- [遗漏维度] [为何重要] [应如何纳入考量]

## 总体评价
（一句话：原结论整体可靠 / 部分可靠需修正 / 重大缺陷需重做）
```

#### 4. 合成者

综合前三方发现，裁决分歧、融合出单一高质量结论。

输出结构：

```text
## 综合结论
（一段话：融合三方发现后的最终判断）

## 共识点（三方一致的部分）
- [共识] [三方分别怎么说]

## 分歧与裁决
- [分歧点] [甲方观点] [乙方观点] [裁决] [裁决理由]

## 采用理由
（为何采用此结论而非其他——必须可追溯到某方的具体发现）

## 行动建议
（基于综合结论，下一步应做什么）
```

### 深度模式流程

1. 准备审查对象摘要：类型、完整内容、主代理初步结论、审查重点。
2. 并行启动前 3 个角色：一条消息内 3 个 Task 调用，独立上下文，互不泄露彼此结论。
3. 启动合成者：等待前 3 个角色全部返回后，传入完整输出。
4. 聚合报告：主代理综合合成者结论、分歧点、自身判断，用 AskUserQuestion 报告。

### 深度模式最终报告

主代理在合成者输出基础上，向用户报告：

1. 综合结论
2. 风险清单（按致命/重大/一般分级，每条含触发场景、影响、缓解建议）
3. 分歧点与裁决
4. 主代理最终判断（采纳 / 修正 / 拒绝合成者结论，并给理由）
5. 总体裁决（推进 / 有条件推进 / 不推进）
6. 是否需要用户裁决

## 与其他审查技能的区别

| 技能               | 时机        | 对象                     | 目的               |
| ------------------ | ----------- | ------------------------ | ------------------ |
| 方案审查（本技能） | 实施前      | 尚未实施的方案/架构/设计 | 提前排雷           |
| 三轴审查           | 实施后      | 已写出的代码差异         | 规范/规格/盲区检查 |
| 决策追问           | 设计阶段    | 决策分支                 | 压力测试决策       |
| 纾困复盘           | 卡顿/失败后 | 已发生问题               | 复盘根因           |

## 关键原则

- 已有成形方案：不是让子代理替你设计。
- 批判而非全盘否定：目标是排雷，不是证明方案不可行。
- 具体触发场景：每个风险必须说明在什么输入/并发条件/数据规模下会出问题。
- 子代理独立上下文：避免确认偏差累积。
- 主代理裁决不转嫁：主代理必须给出最终推进/有条件推进/不推进的裁决。
- 工具结论是参考：合成者结论供参考，最终判断由主代理负责。
