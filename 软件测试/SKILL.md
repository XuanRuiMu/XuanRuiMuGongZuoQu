# 软件测试技能

## 铁律

|规则|内容|
|---|---|
|测试代码与实现代码完全统一一致|测试文件的包路径必须镜像实现文件的包路径；实现文件移动时测试文件必须同步移动；测试类名=实现类名+测试后缀|
|三层镜像|实现代码、测试代码、测试报告三者目录结构完全一致，仅根目录不同；实现代码`XRM/src/main/java/暮澜纪元/命令/玩家指令.java`→测试代码`XRM/src/test/java/暮澜纪元/命令/玩家指令测试.java`→测试报告`开发需求文档/测试报告/XRM/src/test/java/暮澜纪元/命令/玩家指令测试报告.md`；路径从src开始完全镜像|
|独立测试文件|测试代码必须放在src/test/java/下，绝不嵌入实现代码中|
|全部模块全覆盖|每个模块、每个类、每个公共方法都必须有对应测试|
|极限全覆盖|每个测试目标必须覆盖：正常流程+边界值+异常输入+空值/null+权限检查+玩家可见输出|
|玩家可见输出为必测项|任何向玩家发送消息的代码（命令、消息管理器、事件监听器等）必须有对应的玩家可见输出测试；格式：输入操作→预期玩家看到→实际玩家看到→一致/不一致|
|必须验证最终文本|测试必须加载翻译文件，将翻译键解析为最终中文文本并验证内容；仅验证翻译键被调用毫无意义——玩家看不到翻译键；不验证最终文本的测试视为无效测试|
|禁止跳过测试|任何代码修改后必须运行测试验证；不得仅靠编译通过作为验证|
|禁止交付未测试产物|未通过测试的代码不得交付|
|内容哈希清单必用|编写测试前必须运行`python sync_manifest.py check --base-dir <插件目录>`检测变更文件；只读取和更新变更部分对应的测试代码和测试报告；禁止全量读取所有代码来确认一致性；测试完成后必须运行`python sync_manifest.py update --base-dir <插件目录> --source-file <源文件相对路径>`更新清单|

## 适用场景

- 为新功能编写测试
- 为现有代码补充测试
- Bug修复后编写验证测试
- 重构后运行回归测试
- 任何代码修改后的验证

## 核心工作流

### 阶段0：哈希清单检测（增量定位）

**目的**：用内容哈希清单快速定位需要更新测试的文件，避免全量读取所有代码。

1. **运行检测命令**：`python <项目根>/sync_manifest.py check --base-dir <插件目录>`
2. **读取变更列表**：读取生成的`.sync-changes.json`，获取需要更新的文件清单
3. **分类处理**：
   - `source_changed`：源代码已变更，需重新读取源代码+更新测试代码+更新测试报告
   - `source_changed_and_missing_test`：源代码已变更且无测试，需重新读取源代码+新建测试代码+新建测试报告
   - `source_changed_and_missing_report`：源代码已变更且无报告，需重新读取源代码+更新测试代码+新建测试报告
   - `missing_test`：源代码无对应测试，需新建测试代码+新建测试报告
   - `missing_report`：源代码有测试但无报告，需新建测试报告
   - `new`：新增源代码，需新建测试代码+新建测试报告
4. **只读取变更文件**：根据变更列表，只读取对应的源代码、测试代码、测试报告，禁止全量读取
5. **首次使用**：如果插件目录下没有`.sync-manifest.json`，
   先运行`python <项目根>/sync_manifest.py scan --base-dir <插件目录>`生成初始清单

**清单文件说明**：

- `.sync-manifest.json`：记录每个源文件的SHA-256哈希值及对应的测试/报告路径
- `.sync-changes.json`：`check`命令的输出，记录当前哪些文件需要更新
- 脚本位置：`<项目根>/sync_manifest.py`

### 阶段1：测试规划

1. **扫描实现代码**：使用搜索工具扫描src/main/java/下所有包，列出所有需要测试的类
2. **排除规则**：用户明确排除的模块不编写测试（如用户说"技能文件夹先不测"）
3. **生成测试计划**：用TodoWrite记录，每个模块一个todo项，标注：
   - 实现文件路径
   - 对应测试文件路径
   - 需要测试的公共方法列表
   - 需要覆盖的极限场景列表
4. **测试文件路径规则**：
   - 实现文件：`src/main/java/暮澜纪元/修饰器/修饰器.java`
   - 测试文件：`src/test/java/暮澜纪元/修饰器/修饰器测试.java`
   - 规则：包路径完全镜像，类名加"测试"后缀

### 阶段2：极限全覆盖测试编写

#### 2.2 极限覆盖维度（每个公共方法必须覆盖以下全部维度）

|维度|覆盖内容|示例|
|---|---|---|
|正常流程|典型输入下的正确行为|`new 修饰器(修饰器类型.攻击力, 100)` 返回正确对象|
|边界值|零值、最大值、最小值、临界值|数值=0、数值=Double.MAX_VALUE、数值=-1|
|异常输入|null参数、空字符串、非法类型|`new 修饰器(null, 100)` 抛出异常|
|空值/null|所有可null的参数传入null|`修饰器.应用(null)` 不崩溃|
|权限检查|需要权限的操作，无权限时的行为|非管理员执行管理员命令返回权限不足消息|
|玩家可见输出|聊天栏/标题栏/记分板/动作栏的输出内容|`/战斗日志 开` 后玩家收到正确的开启消息|
|并发安全|多线程同时访问时的行为（如适用）|多个线程同时修改玩家属性不产生数据竞争|
|状态转换|对象状态变更后的行为|技能冷却中再次释放返回冷却中消息|

#### 2.3 玩家可见输出测试规范

**核心原则**：测试必须验证玩家最终收到的文本内容。仅验证翻译键被调用毫无意义——必须验证翻译键解析后的最终文本。

**铁律**：

|规则|内容|
|---|---|
|禁止仅验证翻译键|verify(mock消息管理器).发送(player, "技能系统", "日志.设置成功") 这种写法毫无意义，玩家看不到翻译键|
|必须验证最终文本|必须加载翻译文件，将翻译键解析为中文文本，验证玩家实际看到的文字|
|测试即文档|测试报告中的"预期玩家看到"和"实际玩家看到"必须是中文文本，不是翻译键|

**必须测试的输出类型**：

- 聊天栏消息（sendMessage）
- 动作栏消息（sendActionBar）
- BossBar消息
- 标题/副标题（showTitle）
- 计分板内容
- 物品Tooltip

**测试格式**：

```text
输入：玩家执行 /战斗日志 开
预期玩家看到：战斗日志已开启。
实际玩家看到：战斗日志已开启。
结果：一致 ✅
```

**测试代码模式——必须加载翻译文件**：

```java
@Nested
@DisplayName("玩家可见输出")
class 玩家可见输出 {

    private YamlConfiguration 翻译文件;

    @BeforeEach
    void 加载翻译() throws Exception {
        翻译文件 = YamlConfiguration.loadConfiguration(
            new InputStreamReader(Files.newInputStream(
                new File("绝对路径/src/main/resources/文本消息/技能系统/zh.yml").toPath()),
                StandardCharsets.UTF_8));
    }

    @Test
    @DisplayName("执行/战斗日志 开后玩家应看到'战斗日志已开启'")
    void 执行战斗日志开后玩家应看到开启消息() {
        // 1. 执行命令
        指令.onCommand(mock玩家, mock命令, "战斗日志", new String[]{"开"});

        // 2. 验证翻译键被调用
        verify(mock消息管理器).发送(
            any(CommandSender.class),
            eq("技能系统"),
            eq("日志.设置成功"),
            any(),
            any()
        );

        // 3. 用翻译键查翻译文件，获取最终文本（注意：YAML顶级键必须包含在路径中）
        String 最终文本 = 翻译文件.getString("技能系统.日志.设置成功");
        assertNotNull(最终文本, "翻译键'技能系统.日志.设置成功'在翻译文件中不存在");

        // 4. 验证最终文本包含玩家应看到的关键词
        assertTrue(去格式标签(最终文本).contains("已"),
            "最终文本应包含'已'，实际为: " + 最终文本);
    }
}
```

**翻译键路径关键规则**：

|规则|内容|
|---|---|
|YAML顶级键必须包含|翻译文件有顶级键前缀，`技能系统/zh.yml`中键`日志.设置成功`的完整路径是`技能系统.日志.设置成功`；`通用消息/zh.yml`中键`通用.仅玩家`的完整路径是`通用消息.通用.仅玩家`|
|必须用UTF-8编码加载|`YamlConfiguration.loadConfiguration(new InputStreamReader(Files.newInputStream(...), StandardCharsets.UTF_8))`，否则中文键无法读取|
|禁止用File直接加载|`YamlConfiguration.loadConfiguration(new File(...))`不保证UTF-8编码，中文键可能返回null|
|Mockito重载方法注意|`发送(Player,...)`和`发送(CommandSender,...)`是两个重载，verify时用`any(CommandSender.class)`避免匹配错误|

**消息内容验证规则**：

1. 必须加载翻译文件（YamlConfiguration + UTF-8），用翻译键查最终文本
2. 必须验证最终中文文本包含预期的关键词
3. 禁止仅验证翻译键被调用而不验证最终文本
4. 验证消息发送到了正确的输出目标（聊天栏/动作栏/BossBar）
5. 验证消息使用了翻译键而非硬编码文本
6. 测试报告中"预期玩家看到"和"实际玩家看到"必须是中文文本
7. getString()的键路径必须包含YAML顶级键前缀

## 测试覆盖审计

### 功能定位

**审计测试是否够全**，不是"写测试"。静态分析源码的逻辑分支与现有测试，找出未覆盖的路径、边界条件与异常场景。

与核心工作流（阶段0-5，"写测试"）的关系：

| 维度 | 核心工作流（写测试） | 测试覆盖审计 |
| --- | --- | --- |
| 目标 | 编写测试代码让被测代码通过 | 评估现有测试是否覆盖了所有逻辑路径 |
| 时机 | 实现代码前后 | 测试已编写完成后，交付前 |
| 输入 | 实现代码、需求 | 实现代码、现有测试代码 |
| 输出 | 测试文件、测试报告 | 审计报告（缺失场景清单+建议用例） |
| 互补 | 提供被审计的测试资产 | 反馈测试缺口，指导下一轮补充测试 |

典型使用时机：阶段5交付确认前；重大重构后；接手遗留代码时；用户问"测试够不够"时。

### 审计流程

#### 步骤1：梳理核心逻辑路径

读取被审计的源码，列出：

- 主要逻辑路径（正常流程的每条分支）
- 关键分支（if/else、switch、循环、提前return）
- 异常路径（throw、catch、降级处理）
- 状态转换（对象状态变更点）

**目的**：确认理解了代码要做什么，为后续逐条比对做准备。

#### 步骤2：逐条对照现有测试

对步骤1列出的每条逻辑路径，检查现有测试是否有覆盖：

- 有覆盖 → 记入"已覆盖"
- 无覆盖 → 记入"缺失场景"，标注风险等级
- 覆盖但不充分（如只测了正常输入，没测边界）→ 记入"缺失场景"，标注为边界/异常缺口

#### 步骤3：评估风险并给出建议用例

对每个缺失场景：

- 评估风险等级（🔴关键/🟠重要/🟡次要）
- 给出具体测试用例描述（输入、预期输出、验证点）

### 输出结构

审计报告按以下5部分组织：

```text
## 测试覆盖审计报告：<模块/类名>

### 1. 核心逻辑
<源码的主要逻辑路径和关键分支，帮助确认理解了代码>

### 2. 已覆盖
<现有测试已经覆盖了哪些场景。无现有测试时跳过此项>

### 3. 缺失场景
#### 🔴 关键缺口
<核心逻辑或易错路径未测>
- [路径描述] 风险：<为什么关键>

#### 🟠 重要缺口
<边界值、异常处理未测>
- [路径描述] 风险：<为什么重要>

#### 🟡 次要缺口
<锦上添花的场景>
- [路径描述]

### 4. 建议用例
| 缺失场景 | 输入 | 预期输出 | 验证点 |
|---|---|---|---|
```

### 风险等级判定

| 等级 | 判定标准 |
| --- | --- |
| 🔴 关键缺口 | 核心业务逻辑路径未测；易错路径（并发、状态转换、权限判断）未测；已有Bug历史但无回归测试 |
| 🟠 重要缺口 | 边界值（零值、最大值、临界值）未测；异常输入（null、空字符串、非法类型）未测；错误处理分支未测 |
| 🟡 次要缺口 | 罕见但合法的输入组合；性能边界场景；非玩家可见的内部状态 |

### 用 Task 子代理做独立审计（可选）

当审计目标较大（多模块、多文件）时，使用 Task 工具启动子代理并行审计：

- subagent_type="general_purpose_task"
- 每个子代理聚焦一个模块/类，独立读取源码和测试，输出该模块的审计报告
- 最多并行5个子代理
- 子代理任务描述必须包含：
  - 被审计的源码路径
  - 对应的测试路径
  - 输出格式（按上述5部分结构）
  - 风险等级判定标准

**子代理审计优势**：独立上下文，不受主代理对实现代码的既有理解影响，更容易发现"想当然"的测试缺口。

### 与 peer_review 的区别

| 维度 | peer_review | 测试覆盖审计 |
| --- | --- | --- |
| 审查对象 | 代码改动本身 | 测试资产是否覆盖源码逻辑 |
| 核心问题 | 代码改对了吗 | 测试够全吗 |

## TDD模式

当用户要求TDD开发时，遵循红-绿-重构循环：

```text
    红灯                绿灯                重构
 写一个失败    ──→  写最少代码     ──→  优化代码     ──→  (重复)
 的测试             让测试通过          保持测试通过
```

### Prove-It模式（Bug修复时）

Bug报告到达 → 编写复现测试(红灯) → 实现修复 → 测试通过(绿灯) → 运行全量测试(无回归)

## 垂直切片TDD方法论（源自mattpocock/tdd）

### 水平切片是反模式

**水平切片**：先写完所有测试 → 再写所有实现 = 糟糕的测试

为什么？因为测试在写的时候你对问题的理解最浅。先写完所有测试意味着所有测试都基于不完整的理解，后续实现时发现理解有误，测试要么全要改，要么变成束缚。

**垂直切片（追踪子弹）**：一个测试 → 一个实现 → 重复

```text
    测试1 ──→ 实现1 ──→ 测试2 ──→ 实现2 ──→ 测试3 ──→ 实现3 ──→ ...
    ↑                                    │
    └──── 每个测试回应上一个循环中学到的东西 ←─┘
```

**核心原则**：

1. 每次只写一个测试，让它失败（红灯）
2. 写最少代码让这个测试通过（绿灯）
3. 从上一个循环中学到的东西指导下一个测试
4. 后续测试可以推翻或修正前面的假设——这完全正常
5. 全部测试通过后再重构

### 测试应验证行为而非实现

**好测试** = 集成风格，通过公共API验证真实代码路径

- 测试用户能观察到的行为
- 使用真实实现而非Mock（除非外部系统不可控）
- 测试输入→输出的映射关系
- 重构时测试不会断

**坏测试** = 耦合实现，mock内部协作者、测试私有方法

- Mock了被测系统内部的协作者
- 测试私有方法或内部状态
- 验证方法调用顺序而非结果
- 重构时测试断了但行为没变

**警告信号**：重构时测试断了但行为没变 → 说明测试耦合了实现细节，必须重写测试使其验证行为。

### 规划阶段的用户确认

写代码前，向用户确认：

1. 接口变更——新增/修改了哪些公共API
2. 要测试的行为——这个接口应该表现出什么行为
3. 预期的输入输出——关键场景的期望结果

确认后再开始红-绿-重构循环。

### 重构阶段

所有测试通过后，寻找重构候选：

1. **提取重复**——多个测试中重复的setup/断言逻辑
2. **深化模块**——将通用逻辑提取为独立模块
3. **SOLID原则**——检查是否违反单一职责、开闭原则等
4. **消除死代码**——重构后不再需要的代码

重构时保持所有测试通过。如果重构导致测试失败但行为未变，说明测试耦合了实现——先修复测试再继续重构。

## 测试反模式

|反模式|问题|正确做法|
|---|---|---|
|测试实现细节|重构时测试会断|测试输入和输出，不测内部结构|
|过度Mock|测试通过但生产环境崩溃|优先使用真实实现>Fake>Stub>Mock|
|测试间共享状态|单独通过但一起失败|每个测试独立设置和清理状态|
|快照滥用|无人审查，任何变更都断|谨慎使用，每次变更都审查|
|测试框架代码|浪费时间测试第三方行为|只测试自己的代码|
|模糊命名|无法理解测试意图|测试名应像规格说明一样可读|
|跳过失败测试|隐藏问题|失败必须修复，不得跳过|

## 测试命名规范

|场景|命名格式|示例|
|---|---|---|
|测试类|实现类名+测试|`修饰器测试.java`|
|正常流程|方法名_条件_期望结果|`创建修饰器_正确参数_返回正确对象`|
|异常场景|方法名_异常条件_期望行为|`创建修饰器_null类型_抛出异常`|
|边界场景|方法名_边界条件_期望行为|`获取数值_零值_返回零`|
|输出验证|方法名_操作_期望输出|`执行命令_战斗日志开_玩家收到开启消息`|

## 测试文件与实现文件的同步规则

|操作|实现文件|测试文件|
|---|---|---|
|新建类|创建实现文件|同步创建测试文件|
|移动类|移动到新包路径|同步移动到对应包路径|
|重命名类|重命名实现类|同步重命名测试类|
|删除类|删除实现文件|同步删除测试文件|
|新增公共方法|添加方法实现|同步添加该方法的极限全覆盖测试|
|修改方法签名|修改签名|同步修改测试中的调用|
|修改方法行为|修改逻辑|同步修改测试断言|

```java
private String readFileContent(String relativePath) {
    java.io.File file = new java.io.File(源码根 + relativePath);
    return new String(java.nio.file.Files.readAllBytes(file.toPath()), StandardCharsets.UTF_8);
}

private String extractMethodBody(String source, int methodStart) {
    int braceCount = 0;
    int braceStart = source.indexOf('{', methodStart);
    StringBuilder methodBody = new StringBuilder();
    for (int i = braceStart; i < source.length(); i++) {
        char c = source.charAt(i);
        if (c == '{') braceCount++;
        if (braceCount > 0) methodBody.append(c);
        if (c == '}') { braceCount--; if (braceCount == 0) break; }
    }
    return methodBody.toString();
}
```

## 常见问题处理

|问题|处理方式|
|---|---|
|无法Mock的Bukkit类|使用反射或提取接口间接测试|
|测试依赖数据库|使用内存数据库或Mock数据层|
|测试间相互影响|确保每个测试独立setup/teardown|
|测试运行太慢|拆分为单元测试(快)+集成测试(慢)|
|实现代码频繁变动|先锁定公共API，基于API写测试|
|翻译键找不到|检查YAML文件路径和键名层级|
