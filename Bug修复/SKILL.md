# Bug修复技能

## 适用场景

- 用户报告代码bug、错误、警告或异常信息
- 用户需要排查和解决运行时/编译时问题
- 用户反馈功能不按预期工作
- 用户要求修复日志中的错误或警告

## 核心工作流

> **回溯机制说明**：本工作流非严格线性。当出现核心工作流后的"回溯机制"章节中列出的7种情况时，按对应规则处理：多数情况需回溯到指定阶段重新执行；情况4（阶段5发现新独立根因）不立即回溯，而是先完成原问题修复流程再对新根因单独走完整流程。回溯时不丢弃已获得的正确结论，但根因相关结论（如由错误根因推导出的需求文档对照结论、修复计划）应作废。详见"回溯机制"章节。

### 阶段1：问题整理与确认

1. **提取关键信息**：从用户描述中提取以下要素
   - 错误类型（bug/错误/警告/信息）
   - 错误表现（报错信息、异常行为、日志输出）
   - 发生场景（触发条件、复现步骤）
   - 期望行为（用户期望正确结果）

2. **问题初步分类**：根据用户描述，对问题做初步分类（此分类在阶段2看完代码后需确认或修正）
   - **语法/编译错误**：代码无法编译或加载，报错信息明确指向某行
   - **运行时错误**：代码能编译但运行时抛异常或崩溃
   - **逻辑错误**：代码不报错但行为不符合预期
   - **性能/资源问题**：功能正确但存在卡顿、内存泄漏、CPU占用过高等
   - **多类型混合**：一个问题可能同时涉及多个类型（如运行时错误导致逻辑错误），此时按主要表现分类，次要类型在阶段2分类确认时补充

3. **歧义消除**：如果用户描述存在以下任何一种情况，必须使用AskUserQuestion询问
   - 错误信息不完整或模糊
   - 无法确定问题发生的具体位置
   - 存在多种可能的理解方式
   - 缺少复现步骤或触发条件
   - 期望行为不明确
   - **注意**：若歧义消除获得的新信息影响初步分类，需回头更新初步分类

4. **问题确认**：整理后向用户确认理解是否正确（仅在歧义消除步骤中已询问时可跳过此确认）

### 阶段2：代码审查与根因深挖

1. **完整阅读相关代码**：修复前必须完整查看相关代码文件，不得只看片段
   - 使用搜索工具定位相关代码
   - 阅读完整的函数/类/模块，理解上下文
   - 追踪调用链，理解数据流向
   - 检查配置文件和翻译文件（如适用）

2. **分类确认**：看完代码后，确认或细化阶段1的初步分类
   - 如果代码实际表现与初步分类一致 → 确认分类，继续根因深挖
   - 如果代码实际表现需要细化（如从"运行时错误"细化为"运行时错误+逻辑错误"，或补充次要类型）→ 更新分类，记录调整原因，继续根因深挖（正常迭代，不触发回溯）
   - 如果阶段1初步分类存在根本性错误（如把运行时错误误判为纯语法错误）→ 按回溯机制情况1处理

3. **根因深挖（铁律）**：
   - **禁止只修表面症状**：用户报告的问题往往是表象，必须追问"这个问题到底是因为什么才会错误出现"，找到根本原因
   - **5问法**：对每个表面症状连续追问"为什么"，至少追问到能解释"为什么代码会进入错误状态"的根因层
   - **参考常见bug模式清单**：可对照"常见bug模式清单"章节加速识别根因（如NullPointerException对照"空指针/未判空"模式）
   - **区分根因与表现**：
     - 表现 = 用户看到的现象（如"技能伤害为0"）
     - 根因 = 导致现象的代码层面原因（如"伤害计算公式中读取了错误的属性字段"）
   - **检查连锁问题**：一个根因可能引发多个表现，必须确认所有表现都源自同一根因，或识别出多个独立根因
   - **确认修复范围**：哪些文件需要修改，哪些不能动

4. **并行分析代理（复杂bug可选）**：当bug涉及多个模块、调用链复杂、或5问法识别出多个可能的根因方向时，遵循CLAUDE.md并行优先规则，派发并行子代理（Task工具）加速根因调查
   - 每个子代理负责调查一个独立的根因方向（如"方向1：数据库连接池是否耗尽"、"方向2：参数校验是否缺失"）
   - 每个子代理独立阅读代码、收集证据、返回结论
   - 并行数不超过5个
   - 子代理返回后，主代理汇总证据确定根因
   - 简单bug无需并行，直接顺序调查即可

5. **根因结论记录**：在进入阶段3前，用TodoWrite记录根因分析结论（非修复计划）
   - 记录确认的根因、影响的文件范围、初步修复思路（此思路在阶段3需求文档对照后可能调整）
   - 修复计划的正式制定在阶段4步骤1进行，此处不重复

### 阶段3：需求文档对照（铁律）

根因确定后，**必须**去需求文档中查找对应模块的描述，判断代码行为是否符合需求。**禁止跳过此步骤直接修复。**

**例外**：当阶段1/阶段2分类确认为纯语法错误（如拼写错误、缺少分号、括号不匹配）时，无需对照需求文档，因为语法错误不可能"符合需求"。执行阶段4（修复语法错误）后，跳过阶段5同类问题搜索（编译器会一次性报出所有语法错误，无需搜索同类），继续阶段6（执行编译验证确认语法错误已消除；内容哈希清单检查、调用软件测试skill、逻辑模拟、内容哈希清单更新均按流程执行）和阶段7交付确认。交付报告中"需求文档对照结论"填写"纯语法错误，跳过需求文档对照"，"同类问题处理"填写"纯语法错误，跳过同类问题搜索"。

1. **定位需求文档**：
   - 主需求文档：`开发需求文档/需求.md`
   - 子模块文档：`开发需求文档/` 下按类别组织（如
     `技能库/`、`BOSS设计/`、`数值模拟/`、`天赋树.md`、`职业，专精与属性.md` 等）
   - 其他项目需求文档：根据bug所在项目定位（如 `和我恋爱吧/需求文档.md`）
   - 使用Grep在需求文档目录下搜索bug相关模块名/功能名/关键词

2. **对照分析**：将bug代码的当前行为与需求文档描述进行对比，得出以下三种结论之一：

   | 结论             | 含义                                                                                                                                                                                              | 处理方式                                                                          |
   | ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
   | **代码偏离需求** | 需求文档要求代码应表现出某行为，但实际代码未实现或实现错误（需求文档正确，代码错）                                                                                                                | **修复代码**，让代码符合需求文档的描述                                            |
   | **需求存疑**     | 代码当前行为与需求文档不一致，或代码符合需求文档但用户期望的行为与需求文档不一致，且需求文档本身可能需要调整（需求变更/需求遗漏/需求矛盾/需求过时）。当无法确定需求文档是否正确时，归入"需求存疑" | **禁止自行决定**，必须用AskUserQuestion询问用户该怎么处理，直到达成一致后再修复   |
   | **需求缺失**     | 需求文档中找不到该模块/功能的描述                                                                                                                                                                 | **禁止臆测**，必须用AskUserQuestion询问用户该模块的预期行为，获得明确答复后再修复 |

3. **跨模块bug处理**：如果bug根因涉及多个模块，需分别对照各模块的需求文档
   - 每个涉及模块都需独立得出对照结论
   - 若不同模块结论冲突（如模块A结论为"代码偏离需求"但模块B结论为"需求存疑"或"需求缺失"），需在AskUserQuestion中一并呈现
   - 跨模块修复需确认模块间的依赖关系和影响范围

4. **询问用户时的必备信息**（当结论为"需求存疑"或"需求缺失"时）：
   - bug的根因是什么
   - 代码当前行为是什么
   - 需求文档原文是什么（引用具体段落）
   - 二者的冲突点在哪里
   - 提供可选处理方案供用户选择

5. **批判性审视（不自动同意用户）**：遵循CLAUDE.md"不自动同意用户"规则，即使用户给出了处理方案，也需批判性审视
   - 用户方案是否真的能解决根因（而非只修表面）
   - 用户方案是否会引入新问题
   - 用户方案是否与需求文档其他部分冲突
   - 若发现用户方案有问题，必须客观指出并提供替代方案，由用户最终决定

6. **后续流程分支**：根据对照结论和用户决定，选择后续路径：

   | 路径                                 | 触发条件                                                           | 后续动作                                                                                                                                                                                                                                                                                                                                                                                                         |
   | ------------------------------------ | ------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
   | **修复代码**                         | 结论为"代码偏离需求"，或用户决定按需求文档修复                     | 继续阶段4执行修复                                                                                                                                                                                                                                                                                                                                                                                                |
   | **修改需求文档+修复代码**            | 用户决定调整需求文档（需求变更/需求遗漏/需求矛盾），且代码也需修复 | 先修改需求文档，再重新执行阶段3需求文档对照确认代码与新需求一致，然后继续阶段4修复代码                                                                                                                                                                                                                                                                                                                           |
   | **只修改需求文档**                   | 用户决定只调整需求文档，代码当前行为已符合新需求                   | 修改需求文档后跳过阶段4-5（因为未修改代码，无需执行修复和同类问题搜索，不违反铁律），但仍需执行阶段6的清理调试仪器步骤，然后走阶段7交付确认。修复状态分类为NOT_ADDRESSED                                                                                                                                                                                                                                         |
   | **删除功能/回滚版本/其他非修复操作** | 用户决定不修复代码，而是删除出问题的功能或回滚到旧版本             | 删除功能/回滚版本必然涉及代码或配置变更，执行用户决定的操作后跳过阶段4的bug修复流程，但仍需走阶段5的同类问题搜索（搜索其他引用被删除功能的代码）和阶段6的完整流程（内容哈希清单检查、测试验证、内容哈希清单更新、逻辑模拟、编译验证、清理调试仪器）。修复状态分类：若操作后bug不再存在则FIXED（需在交付报告"遗留问题或风险"中注明"通过删除功能/回滚版本解决，原功能不再存在"）；若操作后bug仍存在则NOT_ADDRESSED |
   | **暂不处理**                         | 用户决定暂缓（如修复成本过高、依赖第三方问题）                     | 记录决定，执行阶段6的清理调试仪器步骤，走阶段7交付确认，修复状态分类为NOT_ADDRESSED                                                                                                                                                                                                                                                                                                                              |

7. **达成一致后记录**：将需求文档对照结论与用户决定记录到TodoWrite或会话记忆中，作为修复的依据

### 阶段4：执行修复

1. **制定修复计划**：根据根因和需求文档对照结论，制定修复计划。可使用/plan模式（复杂修复推荐），也可直接用TodoWrite记录计划后执行（简单修复）
2. **精确修改**：
   - 只修改与bug直接相关的代码
   - 不重构无关代码
   - 不添加未要求的"改进"
   - 匹配现有代码风格
3. **翻译文件优先（铁律）**：遵循CLAUDE.md翻译文件优先[P0]规则
   - 用户可见文本（错误消息、提示、日志、配置描述）必须走翻译文件，禁止硬编码
   - 修复中若涉及用户可见文本，必须修改翻译文件而非代码中的字符串
4. **零硬编码（铁律）**：遵循CLAUDE.md零硬编码[P0]规则
   - 数值/名称/路径/变量必须可配置
   - 修复中若涉及数值或配置，必须走配置文件而非硬编码到代码中
5. **修改后立即验证**：每完成一个修改步骤，验证该步骤的效果

### 阶段5：同类问题搜索与一并修复（铁律）

修复主问题后，**必须**搜索项目中是否存在同类型的代码模式，排查是否还有相同问题。**禁止只修用户报告的这一处。**

1. **提取搜索特征**：从根因中提取可搜索的代码特征
   - 错误的API调用方式/方法名
   - 错误的字段名/属性名/配置键名
   - 错误的模式（如"未判空就调用"、"资源未关闭"、"硬编码数值"）
   - 相同的代码结构（如同类技能/同类BOSS/同类接口的实现）

2. **全项目搜索**：
   - 使用Grep搜索特征字符串（方法名、字段名、配置键等）
   - 使用SearchCodebase按语义搜索同类模式（如"还有哪些地方用相同方式读取这个属性"）
   - 搜索范围覆盖整个项目相关目录，不限于用户报告的文件

3. **逐一审查命中项**：对每个搜索命中，判断是否存在相同问题
   - 命中项存在相同问题 → 加入修复清单
   - 命中项不存在问题 → 跳过，但记录已审查

4. **一并修复**：对所有存在相同问题的位置一并修复，修复方式与主问题保持一致
   - **必须遵循阶段4的全部铁律**：包括精确修改（只修bug相关代码）、翻译文件优先（用户可见文本走翻译文件）、零硬编码（数值/名称/路径必可配置）、修改后立即验证

5. **记录搜索结论**：在交付报告中说明
   - 搜索了哪些特征
   - 命中多少处，其中多少处存在相同问题
   - 一并修复了哪些位置（列出文件和行号）

> **衔接说明**：阶段5完成后进入阶段6测试验证。阶段5修复的所有位置（主问题+同类问题）都必须在阶段6中验证。

### 阶段6：测试验证

1. **内容哈希清单检查（编写测试前）**：遵循CLAUDE.md内容哈希清单[P0]规则
   - **编写测试前**必须先运行 `python sync_manifest.py check`（在项目根目录运行）检测变更文件
   - 禁止全量读取所有代码来确认一致性
   - **严格诊断法跳过说明**：若使用严格诊断法，此步骤已在S-Phase 5内部完成，跳过

2. **调用软件测试skill**：修复后必须调用软件测试skill进行测试验证，不得仅靠编译通过作为验证
   - 为修复项编写极限全覆盖测试（正常流程+边界值+异常输入+空值/null+权限检查+玩家可见输出）
   - 测试文件路径、命名、结构遵循软件测试skill的规范
   - 运行测试确认全部通过
   - **严格诊断法跳过说明**：若使用严格诊断法，此步骤由S-Phase 5的回归测试替代，跳过

3. **内容哈希清单更新（测试完成后）**：
   - 测试完成后必须运行 `python sync_manifest.py update --source-file <路径>` 更新清单
   - **严格诊断法跳过说明**：若使用严格诊断法，此步骤已在S-Phase 5内部完成，跳过

4. **逻辑模拟**：在脑中模拟修复后的代码执行流程
   - 从入口点开始，逐步跟踪执行路径
   - 检查边界条件和异常情况
   - 验证修复是否引入新问题
   - **严格诊断法跳过说明**：若使用严格诊断法，此步骤由S-Phase 5的回归测试替代，跳过

5. **编译验证**（如适用）：
   - 运行编译/构建命令确认无编译错误
   - 运行lint/typecheck确认无警告
   - **JAR输出验证**（Minecraft插件项目）：编译后必须验证JAR文件已正确输出到目标目录，确认插件可加载

6. **清理调试仪器**：删除调试过程中产生的所有临时内容。执行条件分两种情况：
   - **走完整阶段6的情况**（修复代码路径、修改需求文档+修复代码路径、删除功能/回滚版本路径）：仅在前面所有步骤（哈希清单检查、测试、哈希清单更新、逻辑模拟、编译验证）全部通过后执行。测试/逻辑模拟/编译验证失败触发回溯机制（见回溯机制第5项），不执行清理。哈希清单check检测到变更文件是正常情况（因为修复了代码），按步骤1正常处理即可；哈希清单update失败需排查失败原因（如路径错误、文件权限），不触发回溯但需修复后重跑update
   - **跳过部分步骤的情况**（只修改需求文档路径、暂不处理路径）：直接执行清理，无需等待前面步骤通过（因为这些路径未修改代码，跳过了测试/逻辑模拟/编译验证等步骤）
   - 删除内容：
   - 删除严格诊断法S-Phase 4探测过程中添加的所有 `[DEBUG-DIAG]` 标记的日志
   - 删除阶段2根因深挖过程中产生的临时搜索记录、临时笔记
   - 删除临时测试脚本、临时线束、临时探测代码
   - 删除所有临时文件和中间产物
   - 遵循CLAUDE.md工作区卫生[P1]规则

### 阶段7：交付确认（铁律）

**必须**使用AskUserQuestion向用户交付修复结果，并询问下一步计划。**禁止静默结束。**

**预防分析（必做）**：在生成交付报告前，必须进行预防分析。无论是否使用严格诊断法，所有bug修复都必须进行预防分析。从以下四个维度形成防御策略：

| 维度                    | 含义                            | 示例                                         |
| ----------------------- | ------------------------------- | -------------------------------------------- |
| **Prevention（预防）**  | 如何在开发阶段就防止此类bug产生 | 增加类型约束、添加 lint 规则、修改代码模板   |
| **Detection（检测）**   | 如何更早发现此类bug             | 增加自动化测试、添加运行时断言、完善监控告警 |
| **Containment（遏制）** | 如何限制此类bug的影响范围       | 添加熔断/降级、隔离故障模块、限制错误传播    |
| **Recovery（恢复）**    | 如何更快从此类bug中恢复         | 提供回滚机制、保留旧版本、提供应急开关       |

分析结论填入交付报告的"预防分析结论"项。

交付报告必须包含以下信息：

| 项目             | 内容                                                                                                 |
| ---------------- | ---------------------------------------------------------------------------------------------------- |
| 修复了什么问题   | 一句话概括主问题                                                                                     |
| 根本原因         | 根因深挖的结论（为什么会错误出现）                                                                   |
| 需求文档对照结论 | 代码偏离需求/需求存疑/需求缺失，以及对应的处理方式（纯语法错误时填写"纯语法错误，跳过需求文档对照"） |
| 修改清单         | 列出所有修改的文件和关键变更                                                                         |
| 同类问题处理     | 搜索了什么特征、命中多少、一并修复了哪些位置                                                         |
| 验证结果         | 模拟结果、编译结果、测试结果、sync_manifest.py检查结果                                               |
| 修复状态分类     | FIXED / PARTIALLY_FIXED / NOT_ADDRESSED / CANNOT_DETERMINE（见下表）                                 |
| 预防分析结论     | 预防分析的具体结论（Prevention/Detection/Containment/Recovery四维度的具体建议，见上方四维度表）      |
| 遗留问题或风险   | 如有则列出，无则说明"无遗留问题"                                                                     |

**修复状态分类**（源自majiayu000/claude-skill-registry）：

| 状态                 | 含义     | 适用场景                                                                                                               |
| -------------------- | -------- | ---------------------------------------------------------------------------------------------------------------------- |
| **FIXED**            | 完全修复 | 主问题根因已消除，同类问题已全部修复，测试全部通过                                                                     |
| **PARTIALLY_FIXED**  | 部分修复 | 主问题已修复但存在以下情况之一：同类问题数量过多分批修复中、某些位置因依赖关系暂无法修复、测试部分通过但核心路径已验证 |
| **NOT_ADDRESSED**    | 未修复   | 用户决定不修复、需求文档对照后决定调整需求而非修复代码、修复成本过高用户决定暂缓                                       |
| **CANNOT_DETERMINE** | 无法确定 | 无法复现且无法定位根因、缺少必要环境/数据无法验证、依赖第三方问题无法在本项目内解决                                    |

**AskUserQuestion的下一步计划选项**：根据修复实际情况灵活组织，典型选项包括但不限于：

- 是否需要补充测试
- 是否需要更新需求文档
- 是否需要提交代码
- 是否还有其他相关问题需要处理
- 是否需要落实预防分析中的某项建议

### 回溯机制

以下7种情况触发回溯，回溯到指定阶段重新执行受影响步骤。回溯时，不丢弃已获得的正确结论，但需明确区分哪些结论仍然有效：

- **根因相关结论自动作废**：当回溯原因是根因判断有误（情况2、3、6），原根因推导出的所有结论（包括需求文档对照结论、修复计划、修复范围）均应作废，重新推导
- **与根因无关的结论保留**：如代码阅读理解、分类信息、用户明确提供的事实等，可继续沿用
- **其他回溯情况**：由回溯原因决定是否作废，如情况1（分类错误）需更新分类但保留问题描述；情况4（新根因）需将新根因作为独立问题从阶段1开始走完整流程；情况5（测试失败）主要检查修复实施而非根因；情况7（交付后反馈）以用户反馈为准

| 编号 | 触发情况                                                          | 回溯到                   | 说明                                                                                                                             |
| ---- | ----------------------------------------------------------------- | ------------------------ | -------------------------------------------------------------------------------------------------------------------------------- |
| 1    | 阶段2分类确认发现阶段1初步分类存在根本性错误（非正常的细化/调整） | 阶段1                    | 修正分类，重新确认问题，然后继续阶段2                                                                                            |
| 2    | 阶段3需求文档对照发现根因判断有误                                 | 阶段2                    | 重新根因深挖，确认新根因后再回阶段3                                                                                              |
| 3    | 阶段4修复过程中发现根因判断有误                                   | 阶段2                    | 重新根因深挖，重新走阶段3-4                                                                                                      |
| 4    | 阶段5同类问题搜索发现新的独立根因                                 | 不立即回溯               | 先完成原问题的修复流程（到阶段6测试通过），再对新根因从阶段1开始走完整流程                                                       |
| 5    | 阶段6测试/逻辑模拟/编译验证失败                                   | 阶段4                    | 分析失败原因，修正修复，重新走阶段6全部步骤                                                                                      |
| 6    | 修复引入新bug                                                     | 视情况回溯到阶段2或阶段4 | 修复方式有问题但根因正确→回溯到阶段4重新修复；根因分析有误（未考虑到某些情况）→回溯到阶段2重新根因深挖。两种情况都需立即回退修复 |
| 7    | 交付后用户反馈有问题                                              | 阶段1                    | 按CLAUDE.md迭代规则处理：产物变差先回退，每次只改用户指出问题，迭代后重自检                                                      |

### 卡顿触发纾困复盘

当 Bug修复过程中出现以下信号时，主动调用"纾困复盘"skill 理清整体方向，而非继续在细节中修补：

- **同一问题回溯超2次**——回溯机制被反复触发，说明可能在错误的方向上修补
- **根因深挖5问法无法收敛**——连续追问5个为什么仍找不到根因，可能是方向性问题而非具体bug
- **修复引入新bug超2次**——每次修复都引入新问题，说明可能存在更深层的设计问题
- **严格诊断法S-Phase 1-4无法构建有效反馈回路**——反馈回路构建失败，可能是问题理解有误

触发时：停止当前修复流程，调用"纾困复盘"skill 进行整体方向复盘。复盘完成得出方向判断（继续/调整/局部重构/推倒重来）后，再决定是否继续修复或调整方向。

## 严格诊断法（源自mattpocock/diagnose）

当bug难以定位或复现时，使用以下6阶段严格诊断法替代常规工作流中阶段2-6的部分步骤。严格诊断法不是独立流程，仍需嵌入核心工作流中执行。

**与核心工作流的对应关系**：

- S-Phase
  1-4（构建反馈回路→迭代回路→假设生成→探测）替代阶段2的"根因深挖"步骤（阶段2的"完整阅读相关代码"、"分类确认"、"并行分析代理"、"根因结论记录"等步骤仍需执行）
- 阶段3（需求文档对照）不受严格诊断法影响，仍需正常执行
- S-Phase
  5（修复+回归测试）替代阶段4的执行流程，并在其内部完成阶段6的步骤1（内容哈希清单检查）、步骤2（调用软件测试skill/编写回归测试）、步骤3（内容哈希清单更新）、步骤4（逻辑模拟）。但阶段4的"精确修改"、"翻译文件优先"、"零硬编码"规则和阶段6的步骤5（编译验证/JAR输出验证）仍需执行。阶段6的步骤6（清理调试仪器）由S-Phase
  6替代（见下条对应关系）
- 阶段5（同类问题搜索与一并修复）不受严格诊断法影响，仍需正常执行。执行时机：在S-Phase
  5（修复+回归测试）完成后、阶段6步骤5（编译验证）之前。若阶段5一并修复了新代码，需回到S-Phase
  5重新运行回归测试确认无回归，然后再继续执行阶段6步骤5编译验证和步骤6清理
- S-Phase 6（清理+事后分析）对应阶段6步骤6（清理调试仪器）+ 阶段7（交付确认）

**与sync_manifest.py检查的顺序**：严格诊断法S-Phase 5的回归测试阶段，先运行 `python sync_manifest.py check`
检测变更文件（对应阶段6步骤1），再编写测试，测试完成后运行 `python sync_manifest.py update`
更新清单（对应阶段6步骤3）。即S-Phase 5内部完成哈希清单的check和update，无需在阶段6重复执行。

**严格诊断法下的回溯**：S-Phase 5回归测试失败时，回溯到S-Phase
5的"实施修复"步骤重新修复（对应回溯机制第5项回溯到阶段4）；若发现根因判断有误，回溯到S-Phase
3重新生成假设（对应回溯机制第3项回溯到阶段2）。其他回溯情况（阶段3需求文档对照发现根因有误、阶段5同类问题搜索发现新根因等）仍按核心工作流的回溯机制处理。

### S-Phase 1：构建反馈回路

**核心思想**：没有反馈回路就无法调试。反馈回路 = 一个能让你在几秒/几分钟内观察到bug行为的机制。

**10种构建反馈回路的方式**：

| 方式          | 适用场景         | 做法                                   |
| ------------- | ---------------- | -------------------------------------- |
| 失败测试      | 可自动化的bug    | 编写一个能稳定复现bug的自动化测试      |
| Curl脚本      | HTTP API bug     | 编写curl命令直接调用API，观察响应      |
| CLI调用       | 命令行工具bug    | 编写shell脚本直接调用，观察输出        |
| 无头浏览器    | 前端UI bug       | 用Playwright/Puppeteer脚本自动操作页面 |
| 重放跟踪      | 有日志的bug      | 添加详细日志，重放请求观察日志输出     |
| 临时线束      | 复杂系统bug      | 编写最小化的独立程序隔离触发bug        |
| 属性/模糊循环 | 边界条件bug      | 用属性测试或模糊测试自动探索输入空间   |
| 二分线束      | 版本回归bug      | 编写脚本自动二分查找引入bug的提交      |
| 差分循环      | 性能/输出差异bug | 对比两个版本的输出差异                 |
| HITL脚本      | 需人工判断的bug  | 编写半自动脚本，人工确认关键节点       |

**选择原则**：选择能最快让你观察到bug行为的回路。优先级：失败测试 > Curl/CLI > 无头浏览器 > 其他。

### S-Phase 2：迭代回路本身

构建初始回路后，持续优化它：

1. **更快**——减少从"做出改变"到"看到结果"的时间
2. **更锐利**——让回路更精确地暴露问题（减少噪音）
3. **更确定性**——消除随机性，让每次运行结果一致

如果回路太慢/太吵/太随机，先优化回路再继续调试。

### S-Phase 3：假设生成

基于反馈回路观察到的现象，生成3-5个排名假设：

| 要求               | 内容                                                                        |
| ------------------ | --------------------------------------------------------------------------- |
| 每个假设必须可证伪 | 必须能设计一个实验来否定该假设                                              |
| 排名依据           | 与观察现象的吻合度 + 简单性（奥卡姆剃刀）                                   |
| 格式               | 假设N：[描述]。预测：如果此假设为真，则[可观察的预测]。验证方式：[如何证伪] |

**示例**：

```text
假设1：数据库连接池耗尽导致超时
预测：超时发生时连接池使用率应为100%
验证方式：添加连接池监控日志，观察超时时的使用率

假设2：特定请求参数触发死循环
预测：只有包含特定参数的请求才会超时
验证方式：用Curl发送含/不含该参数的请求，对比响应时间

假设3：GC暂停导致延迟飙升
预测：延迟飙升时应伴随GC日志中的Full GC事件
验证方式：启用GC日志，对比延迟时间线和GC时间线
```

### S-Phase 4：探测

对每个假设执行探测实验：

1. **每个探测映射到特定预测**——不做与假设无关的探测
2. **一次改变一个变量**——同时改变多个变量会让结果无法解读
3. **标记调试日志**——添加的调试日志必须标记（如`[DEBUG-DIAG]`），便于后续清理
4. **记录结果**——每个探测的结果记录为"支持/否定/无关"假设N

**探测顺序**：按假设排名从高到低，先验证最可能的假设。

### S-Phase 5：修复+回归测试

确认根因后：

1. **先运行内容哈希清单检查**——运行 `python sync_manifest.py check` 检测变更文件（对应阶段6步骤1）
2. **编写回归测试**——在修复前编写一个能复现bug的测试（红灯）
3. **实施修复**——修改代码让回归测试通过（绿灯）
4. **运行全量测试**——确认无回归
5. **更新内容哈希清单**——测试完成后运行 `python sync_manifest.py update --source-file <路径>` 更新清单（对应阶段6步骤3）
6. **正确的接缝**——修复应插入在正确的抽象层，而非在调用链中间打补丁

**接缝概念**：修复代码应该插入在"接缝"处——即行为可以改变的边界。好的接缝是公共API、配置入口、策略模式切换点。坏的接缝是直接修改私有方法内部逻辑。

### S-Phase 6：清理+事后分析

1. **移除所有调试仪器**——删除所有`[DEBUG-DIAG]`日志、临时脚本、测试线束
2. **记录正确的假设**——哪个假设是对的，为什么其他假设是错的
3. **记录根因**——用一句话描述根因
4. **记录教训**——如何能在开发阶段避免此类bug
5. **更新文档**——如果根因涉及架构/设计问题，更新相关文档

**预防分析**（源自Fix Issue，避免复发）：

修复完成后，必须分析"如何避免此类bug再次发生"。预防分析的四维度方法论（Prevention/Detection/Containment/Recovery）已在阶段7中详细呈现，此处不重复。使用严格诊断法时，预防分析在S-Phase
6阶段执行，结论记录到阶段7交付报告中。

## 常见bug模式清单（灵感源自ckorhonen/claude-skills的Common Bug Patterns概念）

根因深挖时，对照以下常见bug模式清单，加速识别根因。此清单不穷尽，仅作为排查起点。

### Java/Bukkit/Minecraft插件常见模式

| 模式               | 典型表现                                    | 根因排查方向                                                                                |
| ------------------ | ------------------------------------------- | ------------------------------------------------------------------------------------------- |
| **空指针/未判空**  | NullPointerException                        | 检查所有外部输入、Map.get结果、数据库查询结果、玩家离线状态                                 |
| **资源未关闭**     | 文件句柄耗尽、数据库连接泄漏                | 检查 InputStream/OutputStream/Connection/ResultSet 是否在 finally/try-with-resources 中关闭 |
| **并发修改**       | ConcurrentModificationException、数据不一致 | 检查集合在迭代时是否被修改、多线程访问共享状态是否同步                                      |
| **事件监听器泄漏** | 内存泄漏、重复触发                          | 检查 Listener 是否在插件禁用时注销、是否重复注册                                            |
| **调度器误用**     | 异步操作Bukkit API报错                      | 检查是否在异步线程调用同步API（应用 BukkitScheduler.runTask）                               |
| **硬编码数值**     | 数值不平衡、配置不生效                      | 检查伤害/冷却/概率等数值是否硬编码而非走配置/翻译文件                                       |
| **翻译文件缺失键** | 玩家看到原始键名而非文本                    | 检查 translation key 是否在翻译文件中存在、大小写是否一致                                   |
| **状态机错误**     | 技能/BOSS状态卡死、行为异常                 | 检查状态转换条件是否完整、是否有不可达状态、并发状态切换是否同步                            |
| **数值计算精度**   | 伤害计算偏差、概率不正确                    | 检查 int/long 除法截断、double 精度丢失、Math.random 范围                                   |
| **权限检查缺失**   | 普通玩家可执行管理员操作                    | 检查命令执行前是否校验 sender 权限、是否校验目标玩家权限                                    |

### 通用代码模式

| 模式               | 典型表现             | 根因排查方向                                                 |
| ------------------ | -------------------- | ------------------------------------------------------------ |
| **边界条件错误**   | 仅在特定输入下出错   | 检查 off-by-one、空集合、最大/最小值、首尾元素               |
| **类型转换错误**   | ClassCastException   | 检查 instanceof 前置校验、泛型擦除、自动装箱拆箱             |
| **字符串比较错误** | 相等判断失效         | 检查用 == 而非 equals、大小写敏感、前后空格、Unicode 规范化  |
| **配置加载错误**   | 配置不生效或加载旧值 | 检查配置热重载是否触发、默认值是否正确、配置文件编码         |
| **日志副作用**     | 日志输出改变程序行为 | 检查日志中是否调用了有副作用的方法（如 toString 触发懒加载） |
| **缓存不一致**     | 数据更新后仍读到旧值 | 检查缓存失效策略、多级缓存同步、缓存穿透/雪崩                |

### 使用方式

1. 根据bug表现，在清单中找最接近的模式
2. 按对应"根因排查方向"快速定位代码
3. 找到根因后，仍需走完整工作流（需求文档对照、同类问题搜索等）
4. 清单未覆盖的模式，使用5问法或严格诊断法深挖

## 铁律

| 规则                 | 内容                                                                                                                                                                                        |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 禁止跳过代码审查     | 修复前必须完整阅读相关代码，不得只看报错行就动手                                                                                                                                            |
| 禁止猜测根因         | 不确定根因时必须继续调查，不得基于猜测修复                                                                                                                                                  |
| 禁止只修表面症状     | 必须追问"为什么会错误出现"，找到根本原因后再修复                                                                                                                                            |
| 禁止跳过需求文档对照 | 修复前必须去需求文档中查找对应模块描述，代码偏离需求则修，需求存疑/需求缺失则问用户（纯语法错误除外）                                                                                       |
| 禁止跳过同类问题搜索 | 修复主问题后必须搜索同类型代码，存在相同问题则一并修复（纯语法错误除外，编译器会一次性报出所有语法错误）                                                                                    |
| 禁止跳过测试验证     | 修复后必须调用软件测试skill编写自动化测试验证修复效果，不得仅靠编译通过作为验证（仅修改需求文档、暂不处理时除外，因为未修改代码无需测试。删除功能/回滚版本必然涉及代码变更，不适用此例外）  |
| 禁止跳过模拟验证     | 修复后必须模拟执行流程，确认逻辑正确（仅修改需求文档、暂不处理时除外，因为未修改代码无需模拟。删除功能/回滚版本必然涉及代码变更，不适用此例外）                                             |
| 禁止过度修改         | 只修bug，不顺手重构/优化/美化无关代码                                                                                                                                                       |
| 禁止跳过交付确认     | 修复完成后必须用AskUserQuestion交付并询问下一步计划，不得静默结束                                                                                                                           |
| 翻译文件优先         | 用户可见文本必须走翻译文件，禁止硬编码（遵循CLAUDE.md [P0]规则）                                                                                                                            |
| 零硬编码             | 数值/名称/路径/变量必须可配置，禁止硬编码（遵循CLAUDE.md [P0]规则）                                                                                                                         |
| 停止线规则           | 发现bug时停止添加新功能，先修复bug再继续开发。在bug修复过程中，若发现新bug或用户要求添加新功能，应先完成当前bug修复再处理。禁止在已知bug存在时继续累积技术债（源自addyosmani/agent-skills） |

## 常见问题处理

| 问题类型                 | 处理方式                                                                                                                                                                                                                                       |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 无法复现                 | 询问用户详细的复现步骤和环境信息；考虑使用严格诊断法S-Phase 1构建反馈回路。若仍无法复现，修复状态分类为CANNOT_DETERMINE，走阶段7交付确认                                                                                                       |
| 多个相关bug              | 按依赖关系排序，先修根因bug，再修表现bug                                                                                                                                                                                                       |
| 修复可能引入新问题       | 在修复计划中明确标注风险，模拟时重点验证                                                                                                                                                                                                       |
| 需要修改配置/翻译文件    | 确认修改影响范围，验证其他引用该配置的地方                                                                                                                                                                                                     |
| 第三方库bug              | 确认库版本，搜索已知issue，评估是否需要升级或绕过                                                                                                                                                                                              |
| 需求文档与代码冲突       | 用AskUserQuestion向用户呈现冲突点，提供可选方案，达成一致后再修复                                                                                                                                                                              |
| 需求文档无对应描述       | 用AskUserQuestion询问用户该模块预期行为，禁止臆测                                                                                                                                                                                              |
| 同类问题数量过多         | 先修复主问题并验证，再将同类问题分批修复，每批验证。若阶段5中发现同类问题实际为独立根因，按回溯机制第4项处理                                                                                                                                   |
| 用户决定与需求文档不一致 | 以用户决定为准，但需在交付报告中记录"需求文档需同步更新"                                                                                                                                                                                       |
| 回归bug（新版本引入）    | 使用二分法查找引入bug的提交：git log 定位最近变更范围 → 二分 checkout/bisect → 每个中间点复现测试 → 缩小到具体提交。注意：git bisect涉及切换提交，需用户确认后方可执行（遵循CLAUDE.md Git安全协议）。复杂场景用严格诊断法S-Phase 1的"二分线束" |
| 修复引入新bug            | 立即回退修复，按回溯机制第6项处理：修复方式有问题但根因正确→回溯到阶段4重新修复；根因分析有误→回溯到阶段2重新根因深挖。在修复计划中标注新bug风险                                                                                               |
| 交付后用户反馈有问题     | 按CLAUDE.md迭代规则处理：产物变差先回退，每次只改用户指出问题，迭代后重自检                                                                                                                                                                    |
| 需求变更误触发           | 区分"bug修复"与"需求变更"：若用户描述的是新行为而非修复错误，应提示用户这属于需求变更，走需求实现流程而非bug修复流程                                                                                                                           |
