# 编写智能体简报

智能体简报是在Issue移至 `ready-for-agent` 时发布在GitHub
Issue上的结构化评论。它是AFK智能体工作的权威规范。原始Issue正文和讨论是上下文——智能体简报是契约。

## 原则

### 持久性优于精确性

Issue可能在 `ready-for-agent`
状态停留数天或数周。代码库会在此期间变化。编写简报使其即使文件被重命名、移动或重构也能保持有用。

- **要**描述接口、类型和行为契约
- **要**命名智能体应查找或修改的特定类型、函数签名或配置结构
- **不要**引用文件路径——它们会过时
- **不要**引用行号
- **不要**假设当前实现结构会保持不变

### 行为性，非过程性

描述系统应该做什么，而非如何实现。智能体会重新探索代码库并做出自己的实现决策。

- **好的：**"`SkillConfig` 类型应接受可选的 `schedule` 字段，类型为 `CronExpression`"
- **坏的：**"打开 src/types/skill.ts 在第42行添加 schedule 字段"
- **好的：**"当用户不带参数运行 `/triage` 时，应看到需要关注的Issue摘要"
- **坏的：**"在主处理函数中添加 switch 语句"

### 完整的验收标准

智能体需要知道何时完成。每个智能体简报必须有具体、可测试的验收标准。每条标准应可独立验证。

- **好的：**"运行 `gh issue list --label needs-triage` 返回经过初始分类的Issue"
- **坏的：**"分流应该正常工作"

### 显式的范围边界

声明什么不在范围内。这防止智能体过度设计或对相邻功能做假设。

## 模板

```markdown
## 智能体简报

**分类：** bug / enhancement **摘要：** 一行描述需要做什么

**当前行为：** 描述现在发生了什么。对于Bug，这是损坏的行为。对于增强，这是功能构建于的现状。

**期望行为：** 描述智能体工作完成后应该发生什么。对边界情况和错误条件要具体。

**关键接口：**

- `TypeName` — 需要变更什么及原因
- `functionName()` 返回类型 — 当前返回什么 vs 应该返回什么
- 配置结构 — 需要的任何新配置选项

**验收标准：**

- [ ] 具体、可测试的标准1
- [ ] 具体、可测试的标准2
- [ ] 具体、可测试的标准3

**不在范围内：**

- 不应在此Issue中变更或处理的内容
- 看似相关但独立的功能
```

## 示例

### 好的智能体简报（Bug）

```markdown
## 智能体简报

**分类：** bug **摘要：** 技能描述截断在单词中间断开，产生损坏的输出

**当前行为：**
当技能描述超过1024个字符时，它在恰好1024个字符处截断，不考虑单词边界。这产生在单词中间结束的描述（例如"Use when the user
wants to confi"）。

**期望行为：** 截断应在1024字符之前的最后一个单词边界处断开并追加"..."以指示截断。

**关键接口：**

- `SkillMetadata` 类型的 `description` 字段 — 无需类型变更，但填充它的验证/处理逻辑需要尊重单词边界
- 任何读取SKILL.md前置信息并提取描述的函数

**验收标准：**

- [ ] 1024字符以下的描述不变
- [ ] 超过1024字符的描述在1024字符前的最后一个单词边界处截断
- [ ] 截断的描述以"..."结尾
- [ ] 包含"..."的总长度不超过1024字符

**不在范围内：**

- 更改1024字符限制本身
- 多行描述支持
```

### 好的智能体简报（增强）

```markdown
## 智能体简报

**分类：** enhancement **摘要：** 添加 `.out-of-scope/` 目录支持以跟踪被拒绝的功能请求

**当前行为：** 当功能请求被拒绝时，Issue以 `wontfix`
标签关闭并附带评论。没有决策或理由的持久记录。未来类似的请求需要维护者回忆或搜索之前的讨论。

**期望行为：** 被拒绝的功能请求应记录在 `.out-of-scope/<concept>.md`
文件中，捕获决策、理由和所有请求该功能的Issue链接。分流新Issue时，应检查这些文件以查找匹配项。

**关键接口：**

- `.out-of-scope/` 中的Markdown文件格式 — 每个文件应有 `# 概念名称` 标题、`**决策：**` 行、`**理由：**`
  行、和带Issue链接的 `**先前请求：**` 列表
- 分流工作流应尽早读取所有 `.out-of-scope/*.md` 文件并按概念相似性将传入Issue与它们匹配

**验收标准：**

- [ ] 关闭功能为wontfix时在 `.out-of-scope/` 中创建/更新文件
- [ ] 文件包含决策、理由和关闭Issue的链接
- [ ] 如果匹配的 `.out-of-scope/` 文件已存在，新Issue追加到其"先前请求"列表而非创建重复
- [ ] 分流期间检查现有 `.out-of-scope/` 文件，当新Issue匹配先前拒绝时揭示

**不在范围内：**

- 自动匹配（人工确认匹配）
- 重新打开先前被拒绝的功能
- Bug报告（只有增强拒绝进入 `.out-of-scope/`）
```

### 坏的智能体简报

```markdown
## 智能体简报

**摘要：** 修复分流Bug

**做什么：** 分流的东西坏了。看看主文件并修复它。第150行左右的函数有问题。

**要改的文件：**

- src/triage/handler.ts (第150行)
- src/types.ts (第42行)
```

这很糟糕因为：

- 没有分类
- 模糊描述（"分流的东西坏了"）
- 引用会过时的文件路径和行号
- 没有验收标准
- 没有范围边界
- 没有描述当前行为 vs 期望行为
