# 范围外知识库

仓库中的 `.out-of-scope/` 目录存储被拒绝的功能请求的持久记录。它有两个目的：

1. **机构记忆** — 为什么功能被拒绝，这样理由不会在Issue关闭时丢失
2. **去重** — 当新Issue与先前拒绝匹配时，技能可以揭示之前的决策而非重新争论

## 目录结构

````text
.out-of-scope/
├── dark-mode.md
├── plugin-system.md
└── graphql-api.md
```text
每个**概念**一个文件，不是每个Issue一个。多个请求同一功能的Issue归在一个文件下。

## 文件格式

文件应以轻松、可读的风格编写——更像简短的设计文档而非数据库条目。使用段落、代码示例和示例使理由清晰，对首次遇到它的人有用。

```markdown
# 深色模式

本项目不支持深色模式或面向用户的主题。

## 为什么这不在范围内

渲染管线假设在 `ThemeConfig` 中定义的单一调色板。
支持多主题需要：

- 包裹整个组件树的主题上下文提供者
- 每个组件的主题感知样式解析
- 用户主题偏好的持久化层

这是一个重大的架构变更，与项目专注于内容创作
的方向不一致。主题化是下游消费者在嵌入或
再分发输出时的关注点。

```ts
// 当前的 ThemeConfig 接口不是为运行时切换设计的：
interface ThemeConfig {
  colors: ColorPalette; // 单一调色板，构建时解析
  fonts: FontStack;
}
```text
## 先前请求

- #42 — "添加深色模式支持"
- #87 — "无障碍夜间主题"
- #134 — "深色主题选项"

```text
### 命名文件

使用简短、描述性的kebab-case概念名称：`dark-mode.md`、`plugin-system.md`、`graphql-api.md`。名称应足够可识别，让人浏览目录时无需打开文件就能理解什么被拒绝了。

### 编写理由

理由应是实质性的——不是"我们不想要这个"而是为什么。好的理由引用：

- 项目范围或理念（"本项目专注于X；主题化是下游关注点"）
- 技术约束（"支持这需要Y，与我们的Z架构冲突"）
- 战略决策（"我们选择使用A而非B，因为..."）

理由应是持久的。避免引用临时情况（"我们现在太忙了"）——那些不是真正的拒绝，是推迟。

## 何时检查 `.out-of-scope/`

在分流期间（步骤1：收集上下文），阅读 `.out-of-scope/` 中的所有文件。评估新Issue时：

- 检查请求是否匹配现有的范围外概念
- 匹配按概念相似性，不是关键词——"夜间主题"匹配 `dark-mode.md`
- 如果有匹配，向维护者揭示："这与 `.out-of-scope/dark-mode.md` 相似——我们之前拒绝了，因为[理由]。你还持相同看法吗？"

维护者可以：

- **确认** — 新Issue被添加到现有文件的"先前请求"列表，然后关闭
- **重新考虑** — 范围外文件被删除或更新，Issue进入正常分流
- **不同意** — Issue相关但不同，继续正常分流

## 何时写入 `.out-of-scope/`

只有当**增强**（不是Bug）被拒绝为 `wontfix` 时。流程：

1. 维护者决定功能请求不在范围内
2. 检查是否已存在匹配的 `.out-of-scope/` 文件
3. 如果存在：将新Issue追加到"先前请求"列表
4. 如果不存在：创建新文件，包含概念名称、决策、理由和第一个先前请求
5. 在Issue上发布评论解释决策并提及 `.out-of-scope/` 文件
6. 用 `wontfix` 标签关闭Issue

## 更新或删除范围外文件

如果维护者对先前拒绝的概念改变主意：

- 删除 `.out-of-scope/` 文件
- 技能不需要重新打开旧Issue——它们是历史记录
- 触发重新考虑的新Issue进入正常分流
````
