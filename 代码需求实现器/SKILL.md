# 代码需求实现器

## 定位

本skill是**单功能点/小范围需求**的执行引擎。它的核心职责是**编排调用其他专项skill**，而不是自己实现所有技术细节。

| 场景 | 推荐入口skill | 原因 |
| --- | --- | --- |
| 多功能点、中大型任务、批量修复、完整重构 | 循环工程 | Orchestrator+Headless自主循环，避免上下文墙 |
| 单功能点、小范围修改、用户希望逐步确认 | 代码需求实现器 | 阶段清晰，每步停下确认 |

**循环工程的子代理在开发新功能时，内部调用本skill完成具体实现。**

## 铁律

| 规则 | 内容 |
| --- | --- |
| 禁止跳步 | 每个阶段必须按顺序执行，不得跳过任何步骤 |
| 禁止隐瞒问题 | 遇到任何问题必须立即暂停并用AskUserQuestion向用户报告 |
| 必须决策追问 | 需求不清时必须调用"方案审查"skill（追问模式），不得自行假设 |
| 必须PRD文档化 | 所有需求必须生成PRD，不得口头确认后直接开发 |
| 必须垂直切片TDD | 开发时遵循垂直切片（一个测试→一个实现→重复），禁止水平切片 |
| 必须极限测试 | 调用"软件测试"skill，覆盖正常/边界/异常/null/权限/用户可见输出 |
| 必须三轴审查 | 测试通过后必须调用"三轴审查"skill |
| 必须用AskUserQuestion交付 | 任务完成后必须用AskUserQuestion向用户报告结果 |

## 阶段跳过规则

并非所有需求都需要走完7个阶段。以下情况可跳过特定阶段：

| 情况 | 可跳过的阶段 | 条件 |
| --- | --- | --- |
| 需求非常明确、无歧义 | 阶段1（需求澄清） | 用户描述已包含完整的功能/接口/行为定义 |
| 小型需求（单文件修改） | 阶段2（PRD）+ 阶段3（Issue拆解） | 改动范围≤3个文件，无需拆解 |
| 紧急修复 | 阶段1-3 | 用户明确说"紧急"或"hotfix"，直接进入阶段4 |
| 纯UI/样式修改 | 阶段5（极限测试） | 无逻辑变更时，视觉验证即可 |

**跳过前必须用AskUserQuestion向用户确认**："这个需求比较简单，建议跳过[阶段名]直接进入[下一阶段]，是否同意？"

## 项目自动识别

启动时自动识别当前项目：

1. **查找项目配置文件**：检查 `references/` 目录下是否有匹配当前工作目录的项目配置
2. **如果没有匹配的配置**：自动探测项目结构（构建工具/语言/框架/测试框架/翻译文件位置等），生成配置并保存
3. **项目配置包含**：
   - 项目名和路径
   - 源码/测试/资源目录结构
   - 构建工具和命令
   - 测试框架和命令
   - 翻译/国际化文件位置
   - 哈希清单脚本路径（如有）
   - 项目特有的编码规范

## 完整工作流

```text
阶段1：需求澄清 ──→ 阶段2：PRD生成 ──→ 阶段3：Issue拆解 ──→ 阶段4：TDD开发 ──→ 阶段5：测试验证 ──→ 阶段6：三轴审查 ──→ 阶段7：交付确认
    ↓ 方案审查          ↓ 生成PRD           ↓ 生成PRD        ↓ 软件测试         ↓ 软件测试         ↓ 三轴审查
                      ↓ 方案审查（如需）
```

---

### 阶段1：需求澄清

**目标**：将用户模糊需求转化为精确、无歧义的需求

1. **理解原始需求**：仔细阅读用户描述，提取关键意图

2. **探索代码库**：根据需求涉及的功能模块，搜索并阅读对应源代码
   - 根据项目配置确定源码目录
   - 根据项目配置确定翻译/资源文件目录

3. **调用"方案审查"skill（追问模式）**：
   - 对需求的每个决策分支进行无情追问
   - 每个问题提供推荐答案
   - 一次只问一个问题，等待用户反馈
   - 如果问题可以通过探索代码库回答，先探索再问
   - 如果仓库存在 `CONTEXT.md`、`CONTEXT-MAP.md` 或 `docs/adr/`，自动启用文档感知模式

4. **转化为精确需求**：
   - 功能描述（做什么）
   - 涉及模块（改哪些文件/包/类）
   - 技术方案（怎么改）
   - 验收标准（怎么算完成）
   - 翻译/国际化影响（需要新增/修改哪些翻译键）

5. **用AskUserQuestion向用户确认**

---

### 阶段2：PRD生成

**目标**：将确认的需求文档化为PRD

1. **调用"生成PRD"skill**：
   - 综合阶段1的澄清结果
   - 不面试用户（已在阶段1完成）
   - 生成包含问题陈述、解决方案、用户故事、实现决策、测试决策、范围外、补充说明的PRD

2. **如需求涉及架构/方案选择，调用"方案审查"skill**：
   - 从可行性/边界/兼容性/性能/安全维度找漏洞
   - 给出具体触发场景
   - 修复方案审查发现的问题后再进入下一阶段

3. **PRD中必须包含**：
   - 翻译/国际化键规划（如项目使用翻译文件）
   - 用户可见输出的完整列表
   - 与现有系统的交互点

4. **用AskUserQuestion向用户确认PRD**

---

### 阶段3：Issue拆解

**目标**：将PRD拆分为可独立开发的垂直切片Issue

1. **调用"生成PRD"skill（拆Issue模式）**：
   - 将PRD拆分为追踪子弹式垂直切片
   - 每个切片横跨所有集成层端到端
   - 标记HITL（需人工交互）和AFK（可自动完成）类型
   - 标注依赖关系

2. **Issue拆解原则**：
   - 宁多勿少（多个薄切片优于少数厚切片）
   - 每个切片可独立演示或验证
   - 按依赖顺序发布（阻塞者先行）

3. **用AskUserQuestion向用户确认拆解**

---

### 阶段4：TDD开发

**目标**：按垂直切片逐个开发，每个切片遵循红-绿-重构循环

对每个Issue/切片：

#### 4.1 哈希清单检测（如项目支持）

如果项目有 `sync_manifest.py`，运行检测命令定位变更文件。

#### 4.2 垂直切片TDD循环

**调用"软件测试"skill的垂直切片TDD方法论**：

```text
红灯：写一个测试 → 测试失败
绿灯：写最少代码让测试通过 → 测试通过
重构：优化代码保持测试通过
重复：下一个测试
```

**禁止水平切片**：

```text
错误：先写所有测试 → 再写所有实现
正确：测试1→实现1 → 测试2→实现2 → 测试3→实现3
```

#### 4.3 翻译/国际化文件同步

每个切片涉及用户可见输出时：

1. 在翻译文件中新增/修改翻译键
2. 测试中加载翻译文件验证最终文本
3. 禁止硬编码任何用户可见文本

#### 4.4 编译验证

每完成一个切片后，运行项目的编译命令确认无编译错误。

---

### 阶段5：测试验证

**目标**：极限全覆盖测试，确保代码质量

1. **调用"软件测试"skill**：
   - 极限覆盖维度：正常流程+边界值+异常输入+空值/null+权限检查+用户可见输出
   - 用户可见输出测试必须验证最终文本（加载翻译文件）
   - 测试文件路径镜像实现文件路径
   - 测试类名 = 实现类名 + "测试"（或项目约定的后缀）

2. **运行测试**：使用项目配置中的测试命令

3. **如果发现Bug，调用"Bug修复"skill**：
   - 严格诊断法：构建反馈回路→复现→假设→探测→修复→回归
   - 先写回归测试再修复

4. **更新哈希清单**（如项目支持）

5. **迭代直到所有测试通过**

---

### 阶段6：三轴审查

**目标**：从标准、规格和盲区三个维度审查代码

1. **调用"三轴审查"skill**：
   - **Standards轴**：代码是否符合项目编码规范
   - **Spec轴**：代码是否忠实实现了PRD中的需求
   - **BlindSpot轴**：变更是否存在逻辑漏洞、遗漏维度、过度推断或交互风险

2. **通用审查清单**：
   - [ ] 所有用户可见文本使用翻译文件（如适用）
   - [ ] 无硬编码数值/名称/路径
   - [ ] 异常处理完善（非法输入不崩溃）
   - [ ] 输入验证用白名单
   - [ ] 错误消息不暴露内部实现
   - [ ] 测试覆盖所有极限维度
   - [ ] 所有测试通过

3. **修复审查发现的问题**

---

### 阶段7：交付确认

**目标**：向用户正式交付任务结果

1. **运行最终构建**：使用项目配置中的构建命令

2. **运行全量测试**：使用项目配置中的测试命令

3. **使用AskUserQuestion向用户交付**，包含：
   - 完成了什么需求
   - 生成的PRD
   - 拆解的Issue列表
   - 修改了哪些文件
   - 新增/修改的翻译键（如适用）
   - 测试结果（通过数/失败数）
   - 三轴审查结果
   - 遗留问题（如有）

4. **询问用户下一步计划**

## 技能调用映射

| 阶段 | 必须调用的skill | 说明 |
| --- | --- | --- |
| 阶段1 | 方案审查 | 需求不清时；自动启用文档感知模式 |
| 阶段2 | 生成PRD + 方案审查（如需） | PRD生成；复杂方案必须前置审查 |
| 阶段3 | 生成PRD | 拆分为垂直切片 |
| 阶段4 | 软件测试（垂直切片TDD） | 开发时自动触发 |
| 阶段5 | 软件测试 + Bug修复 | 极限覆盖；发现bug时修复 |
| 阶段6 | 三轴审查 | 测试通过后自动触发 |
| 阶段7 | — | 手动交付 |

## 按需强制调用

| 场景 | 必须调用的skill | 触发条件 |
| --- | --- | --- |
| 涉及UI/前端实现 | 前端设计 | 任何网页/组件/UI/UX需求 |
| 前端实现后 | 视觉验证 | 前端设计交付后必须截图审查 |
| 需要生成Word文档 | 文档写作 | 交付物为docx时 |
| 需要生成Excel | 文档写作 | 交付物为电子表格时 |
| 涉及架构重构 | 架构深化 | 跨模块重构或浅模块深化时 |
| 需要原型验证 | 原型验证 | 复杂状态机或UI方案选择时 |

## 与其他skill的关系

| skill | 关系 |
| --- | --- |
| 循环工程 | 默认入口；本skill作为其Headless Worker被调用 |
| Bug修复 | 阶段5发现bug时调用 |
| 软件测试 | 阶段4、5必须调用 |
| 三轴审查 | 阶段6必须调用 |
| 方案审查 | 阶段1必须调用 |
| 生成PRD | 阶段2必须调用 |
| 生成PRD | 阶段3必须调用 |
| 方案审查 | 阶段2复杂方案前置审查时调用 |
| 前端设计 | 涉及前端/UI时调用 |
| 视觉验证 | 前端设计后必须调用 |
| 文档写作 | 交付docx时调用 |
| 文档写作 | 交付电子表格时调用 |

**核心区别**：

- **循环工程**是"自动驾驶"——设定目标后自主循环到完成，仅在熔断/阻塞/完成时停下。适合中大型/多功能点任务。
- **代码需求实现器**是"手动挡"——每个阶段停下等你确认。适合单功能点/小范围任务。

## 快速启动

用户只需说：

> "实现需求：我想给战士职业加一个冲锋技能"

或：

> "新功能：用户登录后需要显示欢迎弹窗"

或：

> "需求实现：添加导出PDF报告的功能"

或：

> "代码实现：给XRM的奥能法师加一个新法术——冰霜新星"

本技能将自动：

1. 识别当前项目和技术栈
2. 追问需求细节（或跳过，如果需求已足够明确）
3. 生成PRD文档（或跳过，如果是小型需求）
4. 拆解为垂直切片Issue（或跳过，如果是小型需求）
5. TDD开发（红→绿→重构循环）
6. 极限测试覆盖
7. 三轴审查
8. 交付确认
