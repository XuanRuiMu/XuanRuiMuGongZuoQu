# PRD拆Issue

使用垂直切片（示踪弹）将计划拆分为可独立抓取的Issue。

Issue跟踪器和分流标签词汇应该已提供给你——如果没有，运行 `/setup-matt-pocock-skills`。

## 流程

### 1. 收集上下文

使用对话上下文中已有的内容。如果用户传递了Issue引用（Issue编号、URL或路径）作为参数，从Issue跟踪器获取并阅读其完整正文和评论。

### 2. 探索代码库（可选）

如果你还没有探索过代码库，先探索以了解代码的当前状态。Issue标题和描述应使用项目的领域术语表词汇，并尊重你触及区域的ADR。

### 3. 起草垂直切片

将计划拆分为**示踪弹**Issue。每个Issue是贯穿所有集成层端到端的薄垂直切片，不是某一层的水平切片。

切片可以是'HITL'或'AFK'。HITL切片需要人工交互，如架构决策或设计评审。AFK切片可以在无需人工交互的情况下实现和合并。尽可能优先选择AFK而非HITL。

<vertical-slice-rules>
- 每个切片交付一条窄但完整的路径，贯穿每一层（模式、API、UI、测试）
- 完成的切片可独立演示或验证
- 宁可多个薄切片，不要少数厚切片
</vertical-slice-rules>

### 4. 询问用户

将提议的拆分呈现为编号列表。对每个切片，展示：

- **标题**：简短描述性名称
- **类型**：HITL / AFK
- **被阻塞于**：哪些其他切片（如果有）必须先完成
- **覆盖的用户故事**：此切片解决哪些用户故事（如果源材料有的话）

询问用户：

- 粒度感觉合适吗？（太粗 / 太细）
- 依赖关系正确吗？
- 有切片应该合并或进一步拆分吗？
- HITL和AFK的标记正确吗？

迭代直到用户批准拆分方案。

### 5. 发布Issue到Issue跟踪器

对每个批准的切片，发布新Issue到Issue跟踪器。使用下方Issue正文模板。这些Issue被视为可供AFK智能体使用，因此除非另有指示，以正确的分流标签发布。

按依赖顺序发布Issue（阻塞者先发），这样你可以在"被阻塞于"字段中引用真实的Issue标识符。

<issue-template>
## 父Issue

Issue跟踪器上父Issue的引用（如果来源是已有Issue，否则省略此节）。

## 要构建的内容

此垂直切片的简洁描述。描述端到端行为，而非逐层实现。

避免具体文件路径或代码片段——它们很快过时。例外：如果原型产生了比散文更精确地编码决策的代码片段（状态机、reducer、模式、类型形状），在此内联并简要注明它来自原型。只保留决策密集的部分——不是可运行的演示，只是重要的部分。

## 验收标准

- [ ] 标准1
- [ ] 标准2
- [ ] 标准3

## 被阻塞于

- 阻塞工单的引用（如果有）

或"无 - 可立即开始"如果没有阻塞。

</issue-template>

不要关闭或修改任何父Issue。
