# 纾困复盘

开发陷入卡顿时的复盘机制。核心信念：**卡住的项目，绝大多数不是因为某个 bug 难，而是因为「没人停下来把整体看清楚」。先看清，再判断，最后给方向，而不是急着扎回去继续修补。**

## 触发信号

出现以下任一信号时启动本技能：

- 反复修补无果——同一个问题修了好几轮仍未解决
- 改一处坏一处——每次修复都引入新问题，按下葫芦浮起瓢
- 越做越觉得方向不对——直觉告诉当前思路有问题，但说不清哪里
- 进展散乱缺少全局视角——做了很多事但说不清整体进展
- 默认成立的假设没被验证——方案建立在未验证的假设上
- 不确定该继续/调整/局部重构/推倒重来——方向选择困难

## 铁律

|规则|内容|
|---|---|
|先看清再判断|禁止一上来就扎进某个具体 bug 继续修补，必须先完成现状梳理|
|方向判断必须明确|必须给出明确结论（继续/调整/局部重构/推倒重来）并附依据，禁止模棱两可|
|不把简单问题复杂化|不为显得"深刻"而把简单问题复杂化；现状健康时诚实说"方向没问题，继续"|
|区分根因与症状|禁止把表面噪声当成关键问题，必须区分真正的根因和表面的症状|
|假设必须显式列出|被默认成立却没验证过的假设必须挑出来，指出该怎么验证|
|批判性审视|对当前主流思路做独立批判性审视，找盲区和逻辑漏洞|

## 工作流程

### 步骤1：复盘现状

先吃透当前代码与困境，用自己的话说清"现在到底卡在哪"。

1. **阅读相关代码与文档**——使用 Read/Grep/Glob/SearchCodebase 探索代码库，理解当前实现状态
2. **梳理进展**——已完成什么、正在做什么、卡在什么环节
3. **梳理已有临时方案**——为绕过问题做过哪些妥协、补丁、workaround
4. **区分症状和根因**——用户看到的现象是症状，导致现象的底层原因是根因
5. **用自己的话陈述卡点**——禁止照搬用户原话，必须用自己的理解重新表述，暴露理解偏差

### 步骤2：挑出假设

列出当前方案里被默认成立、却没真正验证过的假设。

1. **扫描方案中的隐含前提**——每个技术方案都建立在假设之上，找出那些"我以为是这样但没验证"的点
2. **每个假设标注验证状态**——已验证/未验证/部分验证
3. **对未验证假设给出验证建议**——指出该怎么验证（写测试、加日志、跑实验、查文档、问用户）
4. **按风险排序**——假设不成立时影响越大的排越前

### 步骤3：独立审视

对当前主流思路做批判性审视，找盲区和逻辑漏洞。

1. **质疑主流思路**——当前思路为什么是对的？有没有更简单的替代方案？
2. **找逻辑漏洞**——思路中有没有自相矛盾、跳跃推理、循环论证
3. **找盲区**——有没有被忽略的边界情况、被忘记的模块、被假设掉的风险
4. **可选：派发 Task 子代理做独立审视**——遵循 AGENTS.md'两阶段战略求助'
   与'子代理必读 AGENTS.md'规则。当问题复杂、涉及多模块时，
   派发 1-3 个 Task 子代理（subagent_type="general_purpose_task"）独立审视。
   每个子代理有独立上下文，可 Read/Grep/Glob 探索代码库，
   返回结论后由主代理汇总。子代理提示词须包含"先读 AGENTS.md"指令

### 步骤4：判断方向

基于分析给出明确方向建议，而非模棱两可。从以下四种结论中选一个，并附依据：

|结论|含义|适用场景|
|---|---|---|
|继续|当前方向正确，继续执行|现状健康，假设已验证，思路无漏洞|
|调整思路|方向大体对但具体路径需调整|部分假设不成立或存在盲区，需修正路径|
|局部重构|局部设计有问题需重构，整体方向保留|某个模块/层次设计有问题，影响扩散|
|推倒重来|整体方向错误，需重新设计|根因在于方向选错，修补无意义|

**禁止**给出"可能需要调整也可能继续"这类模糊结论。必须四选一，并说明依据。如果确实无法判断，说明缺什么信息才能判断。

### 步骤5：沉淀复盘

输出结构化总结，让人能重新对齐。按下方"输出结构"的6部分组织复盘文档，确保团队读完能重新对齐理解。

## 输出结构

复盘输出必须包含以下6部分：

### 1. 现状全景

当前进展、卡点、已有临时方案，一张全局图。读完能让人快速理解"现在是什么情况"。

### 2. 根因 vs 症状

区分真正的关键问题（根因）和表面噪声（症状）。用表格呈现：

|症状|根因|是关键问题还是噪声|
|---|---|---|
|（用户看到的现象）|（导致现象的底层原因）|（关键问题/噪声）|

### 3. 未验证的假设

被默认成立但没验证过的事 + 建议的验证/测试点。用表格呈现：

|假设|验证状态|验证建议|风险等级|
|---|---|---|---|
|（假设内容）|（已验证/未验证/部分验证）|（怎么验证）|（高/中/低）|

### 4. 方向判断

明确结论（继续/调整/局部重构/推倒重来）+ 依据。依据必须具体，引用前面的分析。

### 5. 关键问题清单

按优先级排序、必须先解决的问题。每个问题说明为什么是这个优先级。

### 6. 复盘总结

一段能让团队重新对齐的归纳。用通俗语言，不堆术语，3-5句话讲清"现在什么情况、该往哪走"。

## 边界

- **先看清再判断**——不要一上来就扎进某个具体 bug 继续修补
- **方向判断必须明确并附依据**——模棱两可的复盘没有价值
- **发现设计层面问题时**——指出问题并给改进方向，具体重构交给开发阶段执行
- **不为深刻而复杂化**——现状其实健康时，诚实说"方向没问题，继续"
- **不替代 Bug修复 skill**——本技能理清整体方向，单个 bug 的修复走 Bug修复 skill

## 与其他 skill 的关系

|skill|关系|说明|
|---|---|---|
|Bug修复|互补|Bug修复是修单个 bug（症状→根因→修复→测试），本技能是理清整体方向（现状→假设→判断→复盘）。卡在单个 bug 走 Bug修复；越修越乱、方向不清走本技能|
|三轴审查|互补|三轴审查是对变更做 Standards/Spec/BlindSpot 三轴代码审查，本技能是对整体困境做方向判断。代码质量有问题走三轴审查；不知道该不该继续走本技能|
|决策追问|可前置|决策追问用于压力测试计划/设计，本技能用于理清已陷入的困境。设计阶段走决策追问；已卡住走本技能|
|软件测试|可配合|本技能识别的未验证假设，其验证点可交给软件测试 skill 落地测试|
