# 深化

如何安全地深化一组浅模块，考虑其依赖项。假设 [LANGUAGE.md](LANGUAGE.md)
中的词汇——**模块**、**接口**、**接缝**、**适配器**。

## 依赖类别

评估深化候选时，对其依赖项进行分类。类别决定深化模块如何跨接缝测试。

### 1. 进程内

纯计算、内存状态、无I/O。总是可深化——合并模块并通过新接口直接测试。不需要适配器。

### 2. 本地可替代

具有本地测试替身的依赖项（Postgres用PGLite，内存文件系统）。如果替身存在则可深化。深化模块在测试套件中用替身运行。接缝是内部的；模块外部接口没有端口。

### 3. 远程但自有（端口与适配器）

跨网络边界的自有服务（微服务、内部API）。在接缝处定义**端口**（接口）。深模块拥有逻辑；传输作为**适配器**注入。测试使用内存适配器。生产使用HTTP/gRPC/队列适配器。

推荐表述：_"在接缝处定义端口，为生产实现HTTP适配器，为测试实现内存适配器，这样逻辑集中在一个深模块中，即使它跨网络部署。"_

### 4. 真正外部（模拟）

你无法控制的第三方服务（Stripe、Twilio等）。深化模块将外部依赖作为注入端口接收；测试提供模拟适配器。

## 接缝纪律

- **一个适配器意味着假设性接缝。两个适配器意味着真实接缝。**除非至少有两个适配器被证明合理（通常是生产+测试），否则不要引入端口。单适配器接缝只是间接层。
- **内部接缝 vs 外部接缝。**深模块可以有内部接缝（对其实现私有，由其自己的测试使用）以及接口处的外部接缝。不要仅仅因为测试使用内部接缝就通过接口暴露它们。

## 测试策略：替换，而非分层

- 一旦深化模块接口处的测试存在，浅模块上的旧单元测试就变成废料——删除它们。
- 在深化模块的接口处编写新测试。**接口就是测试面**。
- 测试通过接口断言可观察的结果，而非内部状态。
- 测试应在内部重构后存活——它们描述行为，而非实现。如果测试在实现变更时必须变更，它测试的是越过接口的东西。
