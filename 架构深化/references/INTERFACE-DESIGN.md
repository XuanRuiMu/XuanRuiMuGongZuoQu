# 接口设计

当用户想为选定的深化候选探索替代接口时，使用此并行子智能体模式。基于"设计两次"（Ousterhout）——你的第一个想法不太可能是最好的。

使用 [LANGUAGE.md](LANGUAGE.md) 中的词汇——**模块**、**接口**、**接缝**、**适配器**、**杠杆**。

## 流程

### 1. 框定问题空间

在生成子智能体之前，为选定的候选写一个面向用户的问题空间解释：

- 任何新接口需要满足的约束
- 它依赖的依赖项，以及它们属于哪个类别（见 [DEEPENING.md](DEEPENING.md)）
- 粗略的说明性代码草图以使约束具体化——不是提案，只是让约束具体化的方式

展示给用户，然后立即进入步骤2。用户在子智能体并行工作时阅读和思考。

### 2. 生成子智能体

使用Agent工具并行生成3+子智能体。每个必须为深化模块产生**根本不同**的接口。

用独立的技术简报提示每个子智能体（文件路径、耦合细节、来自 [DEEPENING.md](DEEPENING.md)
的依赖类别、接缝后面是什么）。简报独立于步骤1中面向用户的问题空间解释。给每个智能体不同的设计约束：

- 智能体1："最小化接口——目标是最多1-3个入口点。最大化每个入口点的杠杆。"
- 智能体2："最大化灵活性——支持许多用例和扩展。"
- 智能体3："为最常见调用者优化——让默认情况变得微不足道。"
- 智能体4（如适用）："围绕端口与适配器设计跨接缝依赖。"

在简报中同时包含 [LANGUAGE.md](LANGUAGE.md) 词汇和CONTEXT.md词汇，使每个子智能体的命名与架构语言和项目领域语言一致。

每个子智能体输出：

1. 接口（类型、方法、参数——加上不变量、顺序、错误模式）
2. 展示调用者如何使用它的使用示例
3. 实现在接缝后面隐藏了什么
4. 依赖策略和适配器（见 [DEEPENING.md](DEEPENING.md)）
5. 权衡——杠杆高的地方，杠杆薄的地方

### 3. 呈现和比较

按顺序呈现设计，让用户逐个吸收，然后用散文比较它们。按**深度**（接口处的杠杆）、**局部性**（变更集中在哪里）和**接缝位置**进行对比。

比较后，给出你自己的推荐：你认为哪个设计最强以及为什么。如果不同设计的元素组合效果好，提出混合方案。要有主见——用户想要一个强烈的判断，不是菜单。
