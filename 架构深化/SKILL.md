# 架构深化

揭示架构摩擦并提出**深化机会**——将浅模块转变为深模块的重构。目标是可测试性和AI可导航性。

## 术语表

在每个建议中精确使用这些术语。一致的语言是重点——不要漂移到"组件"、"服务"、"API"或"边界"。完整定义见
[LANGUAGE.md](references/LANGUAGE.md)。

- **模块** — 任何具有接口和实现的东西（函数、类、包、切片）。
- **接口** — 调用者使用模块必须知道的一切：类型、不变量、错误模式、顺序、配置。不仅仅是类型签名。
- **实现** — 内部的代码。
- **深度** — 接口处的杠杆：大量行为隐藏在小接口之后。**深** = 高杠杆。**浅** = 接口几乎和实现一样复杂。
- **接缝** — 接口所在的位置；可以在不原地编辑的情况下改变行为的地方。（使用这个词，不用"边界"。）
- **适配器** — 在接缝处满足接口的具体实现。
- **杠杆** — 调用者从深度中获得的东西。
- **局部性** — 维护者从深度中获得的东西：变更、Bug、知识集中在一个地方。

关键原则（完整列表见 [LANGUAGE.md](references/LANGUAGE.md)）：

- **删除测试**：想象删除模块。如果复杂度消失了，它是透传的。如果复杂度在N个调用者中重新出现，它是有价值的。
- **接口就是测试面。**
- **一个适配器 = 假设性接缝。两个适配器 = 真实接缝。**

此技能受项目的领域模型**指导**。领域语言为好的接缝命名；ADR记录技能不应重新争论的决策。

## 流程

### 1. 探索

首先阅读项目的领域术语表和你触及区域的任何ADR。

然后使用Agent工具以 `subagent_type=Explore` 遍历代码库。不要遵循僵化的启发式——有机地探索并注意你体验到摩擦的地方：

- 理解一个概念需要在许多小模块之间跳转的地方？
- 模块**浅**的地方——接口几乎和实现一样复杂？
- 纯函数仅为了可测试性被提取出来，但真正的Bug隐藏在它们的调用方式中（没有**局部性**）的地方？
- 紧耦合模块跨接缝泄漏的地方？
- 代码库中哪些部分未测试，或难以通过当前接口测试？

对你怀疑是浅的任何东西应用**删除测试**：删除它会集中复杂度，还是只是移动它？"是的，集中了"就是你要的信号。

### 2. 呈现候选

呈现深化机会的编号列表。对每个候选：

- **文件** — 涉及哪些文件/模块
- **问题** — 当前架构为什么造成摩擦
- **解决方案** — 会变更什么的纯中文描述
- **收益** — 用局部性和杠杆解释，以及测试会如何改善

**对领域使用CONTEXT.md词汇，对架构使用 [LANGUAGE.md](references/LANGUAGE.md) 词汇。**如果 `CONTEXT.md`
定义了"订单"，就说"订单接收模块"——而不是"FooBarHandler"，也不是"订单服务"。

**ADR冲突**：如果候选与现有ADR矛盾，只有当摩擦足够真实值得重新审视ADR时才揭示。清晰标记（例如
_"与ADR-0007矛盾——但值得重新开启，因为…"_）。不要列出ADR禁止的每个理论性重构。

不要提出接口设计。问用户："你想探索哪些？"

### 3. 烤问循环

一旦用户选择候选，进入烤问对话。和他们一起走设计树——约束、依赖、深化模块的形状、接缝后面是什么、哪些测试存活。

决策结晶时内联产生副作用：

- **用CONTEXT.md中没有的概念命名深化模块？**将术语添加到 `CONTEXT.md`——与 `/grill-with-docs` 相同的纪律（见
  [CONTEXT-FORMAT.md](../决策追问/references/CONTEXT-FORMAT.md)）。如果文件不存在则延迟创建。
- **在对话中锐化模糊术语？**立即更新 `CONTEXT.md`。
- **用户以承重理由拒绝候选？**提供ADR，措辞为：_"要我记录为ADR，以便未来的架构审查不会重新建议它吗？"_只有当理由确实会被未来探索者需要以避免重新建议相同内容时才提供——跳过短暂理由（"现在不值得"）和不言自明的理由。见
  [ADR-FORMAT.md](../决策追问/references/ADR-FORMAT.md)。
- **想探索深化模块的替代接口？**见 [INTERFACE-DESIGN.md](references/INTERFACE-DESIGN.md)。
