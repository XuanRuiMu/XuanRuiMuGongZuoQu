# 写作策略

> 基于两篇"原文极具参考意义"范文（AIGC率0.64%和6.33%）的风格提取 +
> 38篇论文AIGC检测数据的三轮定量反推验证。本文档是**仁爱学院计算机论文编写skill的唯一写作参考文件**，同时包含文笔文风规范和AIGC防御规则。

## 一、两篇范文的风格定位

### 范文A（0.64%）：硬件论文代表

- 题目：基于嵌入式的光伏板多污渍自适应清洗装置
- AIGC率：0.64%（全文仅1个AI段，约721字）
- 风格关键词：**传感器拟人化 + 决策叙事 + 流水长句 + 技术细节密集**

### 范文B（6.33%）：软件论文代表

- 题目：基于微信小程序的垃圾分类回收系统
- AIGC率：6.33%（20个AI段/21个正常段，AI段短且分散）
- 风格关键词：**功能叙述 + 具体数据表格 + 数据库细节 + 个人总结**

## 二、范文A（0.64%硬件论文）的风格精确分析

### 2.1 主语模式（最核心的风格特征）

**传感器/模块拟人化是定义性风格**。全文技术名词作主语密度高达72.6次/万字。

````text
DS18B20持续读取光照强度并将结果发送给单片机
BH1750利用I²C总线与STM32单片机通信
雨滴传感器模块持续监测周围环境是否有雨水存在
步进电机旋转带动清洁机构对光伏板表面进行物理擦洗
ESP8266模块通过串口与STM32单片机连接
```text
**规则**：让每个硬件组件"活"起来——传感器自己"读取"、自己"发送"、自己"监测"。

### 2.2 句子节奏

**超长流水句与短句交替**。无标点长句可达70字以上，然后突然插入短句。

长句示例（约70字无标点）：
> "系统用STM32F103C8T6单片机为主控芯片进行各种传感器的数据读取以及相应的逻辑判定在不同的环境下采取不同方式进行清扫利用多个传感器相互配合使得清扫机器人可以应对各种恶劣环境增加其实用性与稳定性。"

短句打断（突然缩短）：
> "从功耗角度考虑"、"从电路结构来看"、"由以上工作得出"

**规则**：每200-300字插入一个5-15字的突然短句，打破节奏。

### 2.3 段落结构

**整章一段到底**。全文正常段仅2个（摘要+正文），正文从第一章到结语几乎是一个超长连续段落。不分段是这篇论文的核心风格特征。

### 2.4 技术细节密度

每个技术描述必须包含至少2-3个具体细节：

- 具体型号：STM32F103C8T6、DS18B20、BH1750、ESP8266
- 具体协议：I²C、单总线、GPIO、串口
- 具体电路：上拉电阻、去耦电容、滤波电容、继电器
- 具体操作：旋转、带动、刷洗、闭合、断开

### 2.5 动词风格

操作级具体动词远多于架构级抽象动词：

- 高密度（操作级）：读取、发送、比较、启动、驱动、旋转、带动、刷洗、喷水、闭合、断开、上传、显示
- 低密度（架构级）：进行了（仅1.3/万字）、实现了（仅1.3/万字）、完成了（仅1.3/万字）

### 2.6 避免的模式

全文完全不含：首先...其次...最后...、综上所述、值得注意的是、笔者、具有重要的理论和实践意义

## 三、范文B（6.33%软件论文）的风格精确分析

### 3.1 主语模式

以功能角色为主语，密度远低于范文A：

- 管理员/用户/学生：频繁使用
- 本系统/该系统：中等密度
- 本文/本研究：较低
- 我：集中在总结/致谢章

### 3.2 句子节奏

比范文A更均匀但仍然有明显的不规则性。存在大量"动作流"句子——用逗号串联多个连续动作。

### 3.3 数据库/表格作为AIGC盾牌

范文B的核心策略：大量具体数据库表结构（12张表的完整字段定义）、具体测试数据（用户名"Li"、密码"1"等）散布在正文中，这些高信息密度内容不会被AIGC检测标记。

### 3.4 AI段与正常段混合策略

范文B的20个AI段分散在正文中，每个AI段较短（平均246字），正常段环绕在AI段周围。这说明范文B采用了"人工写框架+AI辅助写部分段落+人工改写AI输出"的混合模式。

### 3.5 总结/致谢的个人化

总结章使用第一人称"我"，有具体指导教师姓名（"姜博畅老师"）、有具体感悟（"之前学的专业知识还不够扎实也不够全面"）。

## 四、两个范文的共同风格（必须遵守）

### 4.1 决策叙事结构

两个范文都不说"系统分为XX模块"，都说"需要XX→比较XX方案→选择XX因为..."。

**硬件论文的决策叙事示例**：
> "单片机要完成多种传感器的数据读取、环境状态检测、驱动模块操作、显示信息还有无线通讯等功能，所以对其运算能力、外设资源以及可靠性都有较高的要求。从系统的功能要求、开发难度、成本控制以及功耗等方面进行比较分析后，在本设计中采用STM32F103C8T6单片机作为系统的主控芯片"

**软件论文的决策叙事示例**：
> "仅需手机，无需再购买其他硬件→仅用于毕业设计→考虑的问题也相对简单→与商业软件存在差距→因此在经济上是可行的"

### 4.2 技术细节必须下沉到操作级

| 抽象级（AI风格，禁止） | 操作级（范文风格，必须） |
| ---------------------- | ------------------------ |
| 系统采用传感器监测环境参数 | DS18B20通过单总线将温度数据发送给STM32 |
| 系统实现了数据存储功能 | 数据通过MyBatis写入MySQL的users表 |
| 进行了详细的需求分析 | 分析了12张数据表之间的关系和业务流程 |
| 使用成熟的开发框架 | 选用Spring Boot 2.2.2，因为它的自动配置省去了约200行XML |

### 4.3 句子不允许均匀

AI文本特征：每句15-25字，节奏均匀。
范文特征：长句可达70-100字，短句5-10字，交替出现。

**规则**：每段至少有一个超过50字的长句和一个不到15字的短句。

### 4.4 AI危险信号源（两个范文都完全避免）

以下模式在两个范文中密度均极低或为零，必须避免。

**注意**：以下结论基于ML模型的全特征分析，已控制文本长度等混淆变量。

| 危险模式 | 验证强度 | 规则 |
| --------- | --------- | ------ |
| 通用数据模板（"不超过/支持/小于"+数字） | 243.6x + ML权重+0.292 | **最强禁止** — 完全不用 |
| "本研究"作主语 | 31.0x + ML权重+0.451 | **完全禁止** — 改用"本系统"或具体名词 |
| "采用了" | 17.3x + ML权重+0.276 | **禁止** — 用"选用""使用"替代 |
| "实现了" | 3.9x + ML权重+0.132 | **限制** — 每千字 ≤ 2次 |
| "完成了"+"在此基础上" | 8.6x + ML权重+0.066 | **限制** — 每千字 ≤ 2次 |
| "因此" | 3.8x + ML权重+0.069 | **限制** — 每千字 ≤ 3次 |
| "综上所述/值得注意的是" | 范文密度0 | **完全不用** |
| "具有重要的理论和实践意义" | 范文密度0 | **完全不用** |
| "笔者" | 范文密度0 | **完全不用** |

**ML修正说明**：

- ⚠️ "进行了"：单变量分析中AI段4.2x更多，但**ML控制其他特征后权重为-0.407，实际是人类标记**。因此"进行了"本身不危险，危险的是伴随它出现的其他AI特征。
- ✅ "可以/能够/允许"：ML权重-0.413 → **这些情态动词是人类标记**，可以正常使用。
- ✅ "首先其次"：验证为正常段特征（0.3x vs AI段）→ **不需要禁止**。危险的是大量"首先其次"配合功能枚举的结构，而非词汇本身。

## 五、各章节写作精确模板

### 5.1 中文摘要模板

**结构**：三段，250-350字（纯中文字符，不含标点）。

**第一段（背景与目的）**：简单概括本文要解决什么问题、目的是什么。1-2句话。
**第二段（需求与技术）**：概括采用了哪些关键技术、什么架构、什么技术栈。不列举，融入叙述。
**第三段（价值升华）**：用个人判断收尾，说明本方案的价值所在。不用"测试验证了"。

**示例三段结构**：

```text
[第一段 - 背景与目的]沙盒游戏平台缺乏系统性的角色成长机制，玩家强度完全由装备堆叠决定。本课题选择了属性管线、战斗伤害管线和技能释放框架三个维度切入——这三个子系统构成战斗体验的闭环。

[第二段 - 需求与技术]属性系统采用三层管线——基础值由职业等级决定，修饰器以覆盖、加法、乘法三种方式修改，次级属性经过十段递减收益函数衰减。技能注册以按键类型和职业为联合键，公CD用急速分段直减模型。任务系统以五状态机驱动NPC交互。

[第三段 - 价值升华]修饰器标签匹配是架构中最关键的抽象——同一接口同时服务全局属性加成和单技能冷却缩减，避免了碎片化接口设计。方案并不新颖，但每个环节经实际游戏场景反复验证，可靠性与可扩展性之间达成了可接受的平衡。
```text
### 5.2 第一章绪论模板

**1.1 课题背景**：

- 从具体问题切入，不说"随着XX的发展"
- 给出具体数据说明问题严重性
- 用"所以"或"因此"引出本课题的必要性

**1.2 研究现状**：

- 不说"XX指出""YY提出"——这种文献综述式是AI最爱
- 说"查阅文献发现，现有研究普遍强调XX，但均未涉及YY问题"
- 每引用一人附带一句评价或对比

**1.3 研究内容**：

- 不编号列举((1)(2)(3)马上触发AI)
- 段落叙述，一段一个方向
- 融入决策理由

### 5.3 第二章相关技术（最小化）

**这是AIGC风险最高的章节！** 技术介绍=教科书=AI训练密集区。

- 每个技术点200-300字即可
- 只写"技术名+本项目为什么选它+一个最关键的技术特性"
- 不要写成教科书式介绍
- 范文B的写法："JSP技术被该系统用来借助网络表单收集用户数据以便访问数据库"——一句话带过JSP是什么，重点在怎么用

### 5.4 第三章需求分析

- 每个需求先从遇到的问题说起
- 范文中"手工清洗简单方便但是工作量大""传统方式效率低下而且容易出错"
- 禁止编号需求((1)(2)(3))

### 5.5 第四章系统设计

- 每个设计决策都是一个小型决策叙事
- 范文A：每个传感器选型都是"需求→选项→比较→选择→理由"五段式
- 范文B：每个数据库表都有"目的→字段→关系"三段式

### 5.6 第五章系统实现

- 代码前后穿插实现决策而非"如上代码所示"
- 范文A风格：代码+硬件描述+电路连接细节
- 范文B风格：代码+界面截图描述+交互流程

### 5.7 第六章系统测试

- 范文风格：问题发现式
- "从实验来看""由实验可知"而非"验证了XX符合预期"
- 测试数据用具体值而非整数

### 5.8 总结

- 范文B风格：个人反思式，"在所有设计决策中""事后回顾"
- 禁止展望、禁止遗憾不足表述

## 六、AIGC防御速查表

### 写前禁止（6条）

1. 禁止定义+展开（"XX是XX的XX"）
2. 禁止功能枚举（"系统分为ABC模块"）
3. 禁止"本研究"作主语（31x AI信号 + ML权重+0.451）
4. 禁止"通用数据模板"（"不超过/支持/小于"+数字 → 243x AI信号！）
5. 禁止(1)(2)(3)括号编号列举
6. 禁止"采用了"（17x AI信号 + ML权重+0.276）和"综上所述/值得注意的是"

### 写中控制（5条）

1. 每段有≥1个超过50字长句和≥1个不到15字短句
2. 每200字插入突然的技术细节深入
3. 技术名词密度 ≥ 50/万字（硬件）或具体数据/表格密度高（软件）
4. 每个设计决策附带方案对比过程
5. 句子长度标准差尽量大——宁可不通顺，不可太均匀

### 写后验证（6条）

1. "实现了+完成了+采用了"每千字 ≤ 2次（⚠️ "进行了"不在此限制内，ML验证其为人类标记）
2. "因此"每千字 ≤ 3次
3. "本研究"出现次数 = 0
4. 功能枚举段落 = 0
5. 编号列举(1)(2)(3) = 0
6. 句子长度随机抽样5句，最长/最短 ≥ 3:1

## 七、风格模仿实操指南

### 如果要写硬件论文（模仿范文A 0.64%）

1. **开门见山传感器**：第一段就列出DS18B20/BH1750/ESP8266等具体型号
2. **让传感器说话**：每个技术名词都是句子的主语
3. **写电路细节**：上拉电阻、去耦电容、滤波电容、3.3V工作电压
4. **超长段不分段**：一章就是一段，一气呵成
5. **动词要"硬"**：旋转、带动、刷洗、闭合、断开——不要"实现""完成"
6. **数据要裸**：直接给数字（"低于一定数值""大于一定阈值"），不要"不超过""支持"

### 如果要写软件论文（模仿范文B 6.33%）

1. **数据库表是AIGC盾牌**：至少8-12张表的完整字段定义散布正文
2. **测试数据要真**：测试用例用"用户名Li密码1"这种具体值
3. **技术介绍最简化**：JSP/Spring Boot/MySQL各200字带过
4. **功能描述用角色主语**：管理员可以XX，用户能够YY
5. **总结用"我"**：个人反思+具体教师名+具体感悟
6. **合理使用AI辅助**：自己写框架→AI辅助写部分→人工改写AI输出
````
