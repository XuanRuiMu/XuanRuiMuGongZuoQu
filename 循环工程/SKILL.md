# 循环工程

Loop Engineering 的 TRAE CN 模拟实现。核心思想： **项目的当前状态住在外部状态文件里，不在任何 LLM 的对话窗口里。**
PROGRESS.md 只保存"现在需要知道什么"，不保存历史档案。

## 铁律

| 规则                      | 内容                                                                                                                                     |
| ------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| 禁止主代理亲自干活        | 主代理（Orchestrator）只负责读PROGRESS.md→判断状态→派发Task子代理→接收简短摘要→更新PROGRESS.md。禁止在主对话中直接写代码/测试/审查       |
| 禁止子代理返回完整过程    | 子代理（Headless Worker）只返回简短结构化摘要（做了什么/验证结果/下一步建议），禁止返回完整代码或日志，避免污染主代理上下文              |
| 必须先读PROGRESS.md       | 每轮循环开始、每个子代理启动时，必须先读PROGRESS.md了解当前状态，禁止凭对话记忆判断进度                                                  |
| PROGRESS.md只写进度       | PROGRESS.md只保留推进任务所必需的状态信息；禁止写入详细设计、完整代码、循环过程、修改文件清单、已失效决策等可由git或最终交付物承载的内容 |
| 必须压缩已完成项          | 功能点一旦完成，仅保留一行状态摘要；删除该功能点的详细验收标准、涉及文件、历史记录、备注等冗余信息                                       |
| 必须清理本项目过程性文件  | 全部功能点完成后，用户明确表示"没有问题"，必须删除**本工作区/本项目内**由本技能创建或修改的PROGRESS.md及其他过程性文件（代码和文件无冗余铁律）；禁止删除其他项目、工作区根目录或非本技能创建的过程性文件；删除前必须告知用户此举将使本次任务的断点续跑信息丢失 |
| 必须每轮更新并压缩        | 每完成一个功能点、每轮循环结束，必须立即更新PROGRESS.md（状态、计数器、待处理项），并同步压缩已完成行                                    |
| 必须可验证停止条件        | 停止条件必须是可机器验证的（测试通过/lint零报错/文件处理完），禁止"优化一下"这种模糊目标                                                 |
| 必须熔断                  | 同一问题修复超5次未解决→跳过记录；总循环次数达上限→停下汇报；遇到无法自行解决的阻塞→AskUserQuestion                                      |
| 禁止过度烘焙              | PROGRESS.md中必须明确列出"做什么"和"不做什么"，子代理只能做列出的功能点，禁止自行加戏                                                    |
| 必须用AskUserQuestion交付 | 仅在三种情况停下用AskUserQuestion：触发熔断、遇到阻塞、目标达成                                                                          |

## 核心架构：Orchestrator + Headless

```text
┌─────────────────────────────────┐
│  Orchestrator（主对话，保持精简） │
│  职责：                          │
│  - 读取PROGRESS.md判断当前状态    │
│  - 判断是否达成停止条件           │
│  - 判断是否触发熔断               │
│  - 派发Task子代理（粗粒度：一个功能点一个）│
│  - 接收子代理简短摘要             │
│  - 更新并压缩PROGRESS.md          │
│  - 循环回到第一步                 │
│  禁止：亲自写代码/测试/审查       │
└──────────────┬──────────────────┘
               │ 每次派发：
               ▼
┌─────────────────────────────────┐
│  Headless Worker（Task子代理）    │
│  全新上下文窗口，干完即终止       │
│  职责：                          │
│  - 先读PROGRESS.md了解全局       │
│  - 实现该功能点                   │
│  - 调用软件测试skill验证          │
│  - 调用三轴审查skill审查          │
│  - 验证失败则修复（内部小循环）    │
│  - 更新PROGRESS.md该功能点状态    │
│  - 返回简短结构化摘要             │
│  禁止：返回完整代码/日志          │
│  禁止：在PROGRESS.md写过程/历史   │
└──────────────┬──────────────────┘
               │
               ▼
        PROGRESS.md（项目根目录）
        项目的当前状态快照
```

**为什么这样设计**：TRAE
CN 的 Task 子代理每次调用都是全新上下文窗口，干完即返回结果终止。这完美对应 Orchestrator+Headless 模式中的 Headless
Worker，彻底绕开了"所有迭代共享一个上下文窗口导致Context
Wall"的问题。主对话保持精简，只存编排逻辑，不存具体工作过程。PROGRESS.md 也只存当前状态，不存历史。

## 阶段跳过规则

| 情况                         | 可跳过的阶段      | 条件                                  |
| ---------------------------- | ----------------- | ------------------------------------- |
| 用户已提供明确目标和验收标准 | 阶段1（目标定义） | 用户描述已包含完整的功能列表+验收标准 |
| 单个功能点的简单修复         | 阶段2（任务拆解） | 只有1个功能点，无需拆解               |
| 用户明确说"紧急"或"hotfix"   | 阶段1-2           | 直接进入阶段3循环                     |

**跳过前必须用AskUserQuestion向用户确认**。

## 完整工作流

```text
阶段1：目标定义 ──→ 阶段2：任务拆解 ──→ 阶段3：自主循环 ──→ 阶段4：交付确认
    ↓ 明确可验证目标      ↓ 拆成功能点列表      ↓ Orchestrator+Headless    ↓ 先跑Self-Harness元循环
    ↓ 定义停止条件        ↓ 写入PROGRESS.md     ↓ 熔断保护                ↓ 再AskUserQuestion
    ↓ 设定熔断上限        ↓ 用户确认            ↓ 仅阻塞/熔断/完成时停     ↓ 最终验证与询问下一步
```

---

### 阶段1：目标定义

**目标**：将用户需求转化为可验证的目标、停止条件和熔断上限

1. **理解原始需求**：仔细阅读用户描述，提取核心意图

2. **探索代码库**（如涉及代码任务）：
   - 根据需求涉及的功能模块，搜索并阅读对应源代码
   - 理解现有架构和约束

3. **定义可验证的停止条件**（必须可机器验证）：
   - ✅ 好的停止条件：所有测试通过且lint零报错、所有功能点状态为已完成、构建成功且无报错
   - ❌ 坏的停止条件：代码优化一下、把功能做好、看起来没问题了

4. **设定熔断上限**：
   - 总循环次数上限：根据功能点数量设定，通常 = 功能点数 × 3（每个功能点最多3轮：实现+修复+审查）
   - 单问题修复上限：同一问题修复超5次未解决→跳过记录到PROGRESS.md
   - Token预算上限（可选）：用户可设定token预算，超限则停

5. **明确范围边界**（防过度烘焙）：
   - 列出"做什么"：明确要完成的功能点
   - 列出"不做什么"：明确禁止自行扩展的功能
   - 列出"禁止触碰"：不能修改的文件/模块

6. **用AskUserQuestion向用户确认目标定义**

---

### 阶段2：任务拆解

**目标**：将目标拆解为粗粒度功能点列表，写入精简的PROGRESS.md

1. **拆解原则**（粗粒度）：
   - 一个功能点 = 一个Task子代理的任务
   - 每个功能点应是一个完整的可验证单元（如"实现用户登录接口"而非"写登录方法"）
   - 标注依赖关系（功能点B依赖功能点A完成）
   - 按依赖顺序排列

2. **为每个功能点定义**：
   - 功能点ID（如FP-01）
   - 描述（做什么）
   - 验收标准（怎么算完成，可机器验证）
   - 依赖（依赖哪些其他功能点）
   - 状态（待开始/进行中/已完成/已阻塞/已跳过）

3. **创建PROGRESS.md**（项目根目录）：
   - 使用 references/PROGRESS模板.md 的精简格式
   - 只包含：元信息、目标、停止条件、熔断上限、范围边界、待处理功能点列表
   - **禁止**包含：详细设计说明、完整代码片段、修改文件清单、循环历史摘要、已完成项的详细描述

4. **用AskUserQuestion向用户确认PROGRESS.md**

---

### 阶段3：自主循环（核心）

**目标**：Orchestrator自主循环，派发Headless Worker执行，直到停止条件达成或熔断

#### 主代理（Orchestrator）循环逻辑

```text
while true:
    1. 读取PROGRESS.md
    2. 检查停止条件：
       - 所有功能点状态=已完成？→ 进入阶段4交付
       - 否则继续
    3. 检查熔断条件：
       - 总循环次数 ≥ 上限？→ 停下用AskUserQuestion汇报
       - 有功能点已阻塞且无法绕过？→ 停下用AskUserQuestion汇报
       - 否则继续
    4. 选择下一个待执行功能点：
       - 优先选择"待开始"且依赖已满足的功能点
       - 若有"进行中"的功能点，继续推进它
    5. 派发Task子代理（Headless Worker）：
       - subagent_type = "general_purpose_task"
       - 传入：功能点ID、PROGRESS.md路径、项目配置、验收标准
       - 子代理提示词使用 references/子代理提示词模板.md
    6. 接收子代理返回的简短摘要（禁止接收完整代码/日志）
    7. 更新并压缩PROGRESS.md：
       - 更新该功能点状态
       - 若状态变为"已完成"，将其压缩为一行摘要，移入"已完成（仅保留一行摘要）"小节，并删除"待处理功能点"中该功能点的详细行
       - 仅追加影响未决事项的当前决策
       - 循环计数器+1
    8. 回到步骤1
```

#### 子代理（Headless Worker）内部流程

每个Task子代理收到任务后，在自己的全新上下文中执行：

1. **读取PROGRESS.md**：了解全局目标、当前功能点、待处理的功能点、当前决策
2. **实现功能点**：
   - 如涉及代码开发，调用"代码需求实现器"skill的TDD流程
   - 如涉及Bug修复，调用"Bug修复"skill的诊断流程
   - 如涉及重构，遵循现有架构风格
3. **验证**：
   - 调用"软件测试"skill运行测试
   - 运行项目构建命令确认无编译错误
   - 运行lint确认无报错
4. **审查**（强制，不得跳过）：
   - **必须调用"三轴审查"skill**（Standards轴+Spec轴+BlindSpot轴），即使只修改配置文件（YAML/JSON/properties）也必须调用
   - 禁止以"仅修改配置无代码逻辑变更""FP-XX已完成三轴审查且本FP无逻辑变更"等理由跳过三轴审查
   - 修复审查发现的问题
5. **内部小循环**（最多5次）：
   - 验证失败→修复→重新验证
   - 同一问题修复超5次→标记阻塞，返回摘要
6. **更新PROGRESS.md**（只写状态，不写过程）：
   - 更新该功能点状态（已完成/已阻塞）
   - 若已完成，按模板将其压缩为一行摘要
   - **禁止**记录修改文件清单、循环历史、详细决策日志
7. **返回简短结构化摘要**（禁止返回完整代码）：

   ```text
   功能点：FP-01 用户登录接口
   状态：已完成
   修改文件：src/auth/login.ts, tests/auth/login.test.ts
   验证结果：测试12/12通过，lint零报错，构建成功
   审查结果：Standards通过，Spec通过
   关键决策：选择JWT而非session（原因：无状态更易扩展）
   遗留问题：无
   下一步建议：可开始FP-02
   ```

#### 熔断机制详解

| 熔断类型              | 触发条件                           | 处理方式                                                                           |
| --------------------- | ---------------------------------- | ---------------------------------------------------------------------------------- |
| 单问题修复熔断        | 同一问题修复超5次未解决            | 子代理标记该功能点为"已阻塞"，记录问题详情，返回摘要，主代理跳过该功能点继续下一个 |
| 总循环次数熔断        | 总循环次数≥上限                    | 主代理停下，用AskUserQuestion汇报当前进度、已完成/未完成功能点、消耗的循环次数     |
| 阻塞熔断              | 有功能点已阻塞且后续功能点都依赖它 | 主代理停下，用AskUserQuestion汇报阻塞原因，询问用户决策                            |
| Token预算熔断（可选） | Token消耗超预算                    | 主代理停下，用AskUserQuestion汇报                                                  |

#### 熔断触发纾困复盘

当触发以下熔断时，主代理在用 AskUserQuestion 汇报前，应先调用"纾困复盘"skill 理清整体方向：

- **单问题修复熔断**——同一问题修复超5次未解决，可能是方向性问题而非具体修复方法问题
- **阻塞熔断**——功能点阻塞且后续依赖它，可能是任务拆解或依赖关系设计有误

触发时：主代理先调用"纾困复盘"skill 进行整体方向复盘，复盘得出方向判断（继续/调整/局部重构/推倒重来）后，再用 AskUserQuestion 向用户汇报复盘结论和建议方向。

注意：总循环次数熔断和Token预算熔断**不触发**纾困复盘，因为这两者是资源耗尽而非方向问题，直接用 AskUserQuestion 汇报即可。

#### 并行优化（可选）

当多个功能点之间无依赖关系时，主代理可并行派发多个Task子代理（最多5个）：

```text
主代理识别无依赖的功能点FP-03、FP-04、FP-05
→ 单条消息中发起3个Task工具调用（并行）
→ 等待3个子代理全部返回
→ 分别更新并压缩PROGRESS.md中3个功能点的状态
→ 继续下一轮循环
```

**并行条件**：功能点之间无文件冲突（不修改同一文件）、无逻辑依赖。

### 阶段4：交付确认

**目标**：向用户正式交付循环结果

1. **最终验证**：
   - 读取PROGRESS.md，确认所有功能点状态
   - 运行全量测试
   - 运行完整构建
   - 运行lint

2. **触发 Self-Harness 元循环**（见下节）：
   - 在 AskUserQuestion 最终交付前，先跑一轮 Self-Harness 元循环
   - 若元循环提出"待确认"或"禁止自动"类改进，必须用 AskUserQuestion 向用户汇报，不得自动执行
   - 元循环完成后，将其结果追加到 PROGRESS.md 的"元循环摘要"小节（仅一行摘要）

3. **用AskUserQuestion向用户交付**，包含：
   - 完成了什么目标
   - 功能点完成情况（已完成/已跳过/已阻塞 各多少个）
   - 总循环次数和Token消耗（如可统计）
   - 修改的文件列表（从git diff/status获取，不在PROGRESS.md中维护）
   - 测试结果（通过数/失败数）
   - 审查结果
   - Self-Harness 元循环发现的待确认项（如有）
   - 关键决策摘要
   - 遗留问题（已跳过/已阻塞的功能点详情）
   - PROGRESS.md路径（供用户查看当前状态）

4. **清理本项目过程性文件并询问下一步计划**：
   - **前置检查（强制）**：运行 `git status` 和 `git diff --stat` 检查本工作区是否有未提交修改；如有未提交修改，必须先用 AskUserQuestion 提醒用户上传（违反项目级 AGENTS.md [P0] "所有代码修改必须上传"规则），待用户确认上传完成或明确授权主代理上传后再进行清理；禁止在存在未提交修改的情况下清理过程性文件或宣告任务完成
   - **范围判定（强制）**：清理前必须确认待删除文件位于**当前工作区/本项目内**，且**由本技能创建或修改**。禁止删除以下文件：
     - 其他项目或子工作区内的PROGRESS.md或过程性文件
     - 工作区根目录下不属于本项目的PROGRESS.md或过程性文件
     - 项目原有文件、用户手动创建的文件、其他skill创建的文件
   - 询问用户是否还有问题：
     - 若用户明确表示"没有问题/不需要保留进度"：
       - 删除**本工作区/本项目内**由本技能创建的PROGRESS.md
       - 删除**本工作区/本项目内**由本技能生成的其他过程性文件（如临时计划、中间摘要、过程日志、EVIDENCE.md等）
       - 删除前必须告知用户：PROGRESS.md删除后，本次任务的断点续跑信息将丢失
     - 若用户需要保留进度（如处理阻塞项、跨会话续跑），保留PROGRESS.md
   - 询问是否需要处理已跳过/已阻塞的功能点
   - 询问是否需要提交代码

## Self-Harness 元循环

### 目标

在每次用户任务完成后、阶段4最终交付前，主动审查本次任务过程中暴露的弱点，提出针对循环工程自身 harness（约束与支撑结构）的改进提案，并按修改分级规则决定处理方式。Self-Harness 不是自我无限迭代，而是受控的、有停止条件的单轮/多轮元循环。

### 核心思想

- **Harness**：约束 AI 行为、防止失败复现的结构（规则、模板、检查清单、预算、证据）。
- **Weakness**：本次任务中暴露的、可复用的失败模式，而非一次性偶发错误。
- **Self-Harness**：用 loop 工程自身的流程来改进 loop 工程自身的 harness。

### 触发时机

| 时机 | 说明 |
| --- | --- |
| 每个用户任务完成后 | 阶段3停止条件达成（所有功能点已完成/跳过/阻塞，或触发熔断） |
| 阶段4交付前 | 在 AskUserQuestion 最终交付用户之前，必须先完成元循环 |
| 可选：纾困复盘后 | 若触发纾困复盘并判定方向为"调整"，可在复盘后追加一轮 Self-Harness |

### 输入来源

| 来源 | 文件路径 | 用途 |
| --- | --- | --- |
| 当前任务轨迹/摘要 | PROGRESS.md | 了解本次任务的功能点、循环次数、阻塞项、关键决策 |
| 证据记录 | `HARNESS.md` 同目录下的 `EVIDENCE.md` | 查看历史失败证据，避免重复挖掘 |
| Harness 组件清单 | `HARNESS.md` | 查看现有 harness 组件及其修改级别 |
| 预算与熔断 | `BUDGET.md` | 检查预算是否允许元循环继续运行 |
| 回归任务集 | `references/harness-test-suite/` | 验证 harness 改进提案是否破坏现有行为 |

### 流程：Weakness Mining → Harness Proposal → Proposal Validation

```text
┌──────────────────────────────────────────────────────────────┐
│  Self-Harness 元循环                                          │
│  1. Weakness Mining                                            │
│     - 读取 PROGRESS.md / EVIDENCE.md / HARNESS.md / BUDGET.md │
│     - 识别本次任务中暴露的可复用失败模式                       │
│     - 按错误分类体系标记失败类型（鉴权/网络/数据/逻辑/依赖/未知）│
│     - 输出：弱点列表（带证据链接）                             │
│  2. Harness Proposal                                           │
│     - 针对每个弱点，提出 harness 层预防措施                    │
│     - 给出修改分级：自动 / 待确认 / 禁止自动                   │
│     - 输出：提案列表（含目标组件、修改分级、预期效果）         │
│  3. Proposal Validation                                        │
│     - 自动级提案：直接应用并运行回归任务集验证                 │
│     - 待确认/禁止自动提案：汇总为 AskUserQuestion 待确认项     │
│     - 输出：验证结果 + 待确认项清单                            │
└──────────────────────────────────────────────────────────────┘
```

#### 1. Weakness Mining（弱点挖掘）

主代理（Orchestrator）读取当前任务的 PROGRESS.md 和历史 EVIDENCE.md，识别以下类型的可复用失败模式：

| 关注点 | 示例 |
| --- | --- |
| 子代理是否返回完整代码/日志而非摘要 | 违反 SKILL.md 铁律"禁止子代理返回完整过程" |
| 子代理是否未先读 PROGRESS.md | 违反 SKILL.md 铁律"必须先读PROGRESS.md" |
| PROGRESS.md 是否膨胀 | 写入历史、过程、文件清单等冗余内容 |
| 熔断条件是否被绕过 | 同一问题修复次数未正确计数 |
| 范围边界是否被突破 | 子代理修改了"禁止触碰"列表中的文件 |
| 预算/熔断是否触发异常 | 总循环次数未正确递增、阻塞未正确标记 |
| 三轴审查是否被跳过 | 子代理未调用三轴审查skill |

挖掘原则：
- 只记录**可复用失败模式**，不记录一次性偶发错误。
- 每个弱点必须有证据：PROGRESS.md 中的阻塞记录、子代理摘要中的失败描述、或 `.agents/evidence/traces/` 中的轨迹片段。
- 禁止编造证据；找不到来源的弱点标记为"待补充"并跳过。

#### 2. Harness Proposal（Harness 改进提案）

针对每个确认的弱点，提出一个或多个 harness 层改进提案。提案必须落在具体组件上，如：

| 目标组件 | 示例提案 |
| --- | --- |
| SKILL.md | 在阶段3子代理流程中增加"必须先读 HARNESS.md"检查 |
| references/子代理提示词模板.md | 增加返回字段"是否遵守了禁止返回完整过程铁律" |
| references/PROGRESS模板.md | 增加"禁止写入内容"检查清单 |
| HARNESS.md | 将某组件的修改级别从"自动"提升为"待确认" |
| BUDGET.md | 增加 wall-clock 时间熔断字段 |

每个提案必须明确：
- 针对弱点
- 目标组件
- 修改分级（见下节）
- 预期效果
- 验证方式（回归任务集/检查清单/用户确认）

#### 3. Proposal Validation（提案验证）

| 修改分级 | 处理方式 |
| --- | --- |
| 自动 | 直接应用提案，运行 `references/harness-test-suite/run_all.py` 回归任务集；若通过，更新 EVIDENCE.md 摘要 |
| 待确认 | 不自动应用；汇总为 AskUserQuestion 待确认项，等待用户决策 |
| 禁止自动 | 不自动应用；汇总为 AskUserQuestion 待确认项，并明确告知用户需手动执行或明确授权后方可执行 |

验证通过标准：
- 回归任务集 3/3 通过（或当前定义的固定回归任务集全部通过）。
- 若提案涉及 SKILL.md / AGENTS.md 核心流程，必须经用户明确同意后方可视为验证通过（禁止自动改写）。

### 三类修改分级

本技能对所有 harness 组件的修改实行分级管控。

| 级别 | 定义 | 触发条件 | 处理流程 |
| --- | --- | --- | --- |
| 自动 | 不违反 PROGRESS.md 铁律、不改动核心工作流、不涉及用户确认文件的修改 | 仅调整格式、新增纯说明性 references、补充示例、修复笔误、更新 EVIDENCE.md 摘要 | 子代理可直接修改，无需额外确认；修改后必须运行回归任务集验证 |
| 待确认 | 可能影响核心工作流、返回结构或用户认知，但风险可控 | 修改提示词返回结构、调整 PROGRESS 字段、新增/删除熔断规则、扩展 budget 字段、修改 references/Orchestrator-Headless模式.md | 修改前必须用 AskUserQuestion 向用户说明变更内容、原因、影响范围，取得同意后方可执行 |
| 禁止自动 | 涉及项目级规则或 skill 核心契约，必须由用户明确授权 | 修改 SKILL.md 核心流程、修改 AGENTS.md、删除铁律、变更停止条件/熔断上限默认值 | 任何子代理不得自动执行；仅可在用户通过 AskUserQuestion 明确回复同意后，由主代理或用户指定的代理执行 |

### 停止条件

Self-Harness 元循环在以下任一条件满足时停止：

| 条件 | 说明 |
| --- | --- |
| 预算熔断 | 元循环自身消耗达到 BUDGET.md 中定义的 `total_turn_limit` 或 `wall_clock_limit_minutes` 上限；元循环不会无限运行 |
| 无新证据 | Weakness Mining 未发现新的可复用失败模式，或新弱点已被现有 harness 覆盖 |
| 回归任务集通过 | 本轮提出的自动级提案已全部应用并通过 `references/harness-test-suite/run_all.py`；待确认/禁止自动项已汇总完毕 |

### 与 HARNESS.md / EVIDENCE.md / BUDGET.md 的交互

| 文件 | 交互方式 |
| --- | --- |
| `HARNESS.md` | 作为 harness 组件清单和修改分级的权威来源；元循环开始前读取，确认各组件当前修改级别；若提案需调整某组件级别，按分级规则处理 |
| `EVIDENCE.md` | 作为历史失败证据的索引；Weakness Mining 时读取以避免重复挖掘；Proposal Validation 后将本轮新证据按一行摘要格式追加；超过20条时按文件内规则压缩 |
| `BUDGET.md` | 作为元循环自身的预算约束；元循环开始前检查剩余预算；元循环结束后更新实际消耗（若字段可获取）；若预算不足，立即停止元循环并将原因写入 PROGRESS.md 摘要 |

### 元循环与 AGENTS.md P0 规则的兼容性

| AGENTS.md P0 规则 | Self-Harness 中的落实 |
| --- | --- |
| 禁止假设 | 元循环不假设弱点存在；必须从 PROGRESS.md / EVIDENCE.md / 子代理摘要等授权来源中读取证据 |
| 禁止幻觉 | 所有弱点、提案、验证结果必须可追溯；禁止编造证据或虚构文件内容 |
| 禁止跳步 | 自动级提案也必须运行回归任务集验证；禁止跳过验证直接交付 |
| 禁止交付未验证产物 | 元循环产生的任何修改都必须验证；待确认/禁止自动项未获用户同意前不得交付 |
| 必须使用 AskUserQuestion 交付 | 元循环产生的待确认/禁止自动项必须用 AskUserQuestion 向用户汇报；元循环摘要也需在阶段4交付中呈现 |
| 不自动改写 AGENTS.md | SKILL.md 中明确：AGENTS.md 属于禁止自动级别，任何子代理不得自动修改 |
| 关键决策必须 AskUserQuestion | 修改分级为"待确认"或"禁止自动"的提案均属于关键决策，必须 AskUserQuestion |

### 元循环输出格式

Self-Harness 元循环完成后，主代理应在 PROGRESS.md 中写入一行摘要：

```text
Self-Harness：本轮发现 X 个弱点，Y 个自动级提案已应用并通过回归，Z 个待确认/禁止自动项待用户决策。
```

并在阶段4 AskUserQuestion 交付中单独列出待确认项清单。

## 快速启动

用户只需说：

> "循环工程：把这个项目的所有测试失败修复，直到全部通过"

或：

> "loop：给XRM插件添加5个新法术，自主循环直到全部实现并通过测试"

或：

> "自主循环：把和我恋爱吧项目的所有中文硬编码提取到翻译文件，别停下一直做"

或：

> "自动开发：重构认证模块，按依赖顺序逐个功能点推进，熔断上限30轮"

本技能将自动：

1. 定义可验证目标和停止条件
2. 拆解为功能点列表，写入精简的PROGRESS.md
3. 自主循环：主代理编排 + Task子代理执行（全新上下文）
4. 熔断保护：单问题5次/总循环上限/阻塞即停
5. 每轮压缩已完成项，保持PROGRESS.md精简
6. 仅在阻塞/熔断/完成时用AskUserQuestion停下
7. 阶段4交付前触发 Self-Harness 元循环，发现 harness 改进项并按分级处理
8. 交付确认；用户确认无问题后删除**本工作区/本项目内**由本技能创建或修改的PROGRESS.md等过程性文件，需保留进度时则保留当前状态（非历史档案）

## 与其他技能的关系

| 技能           | 关系                                 |
| -------------- | ------------------------------------ |
| 代码需求实现器 | 循环工程的子代理在开发新功能时调用它 |
| Bug修复        | 循环工程的子代理在修复bug时调用它    |
| 软件测试       | 循环工程的子代理在验证时调用它       |
| 三轴审查       | 循环工程的子代理在审查时调用它       |
