# Orchestrator + Headless 模式

这是循环工程技能的核心架构原理。理解这个模式，才能正确执行循环工程。

---

## 一、问题：Context Wall（上下文墙）

### 什么是 Context Wall

所有 LLM 都有上下文窗口大小限制。当一个 Agent 在同一个对话窗口里持续工作时，对话历史会不断增长：

```text
轮次1：用户指令 + AI回复 + 工具调用结果
轮次2：以上全部 + 新的AI回复 + 新的工具调用结果
轮次3：以上全部 + ...
...
轮次N：上下文窗口被填满
```

### Context Wall 导致的三种劣化

1. **指令遗忘（Context Drift）**：前半段设定的规则和决策，到后半段被稀释，AI 不再遵守
2. **回归生成（Regression）**：把之前已经修好的东西又改坏，因为忘了之前修过
3. **幻觉加速（Hallucination）**：自信满满地编造不存在的状态，比如幻觉"目标已达成"

### 为什么 Claude Code 原生的 /goal 也有这个问题

Claude Code 的 `/goal` 命令所有迭代共享同一个上下文窗口。随着对话增长，模型有效
准确度下降，最坏情况下评估器会幻觉"目标已达成"而实际没有。这是 `/goal` 无法从内
部解决的问题——做"是否完成"判断的系统，正是注意力被侵蚀的系统。

---

## 二、解法：Orchestrator + Headless 模式

### 核心思想

**项目的记忆住在外部状态文件里，不在任何 LLM 的对话窗口里。**

把系统拆成两个角色：

```text
Orchestrator（编排者）
  - 保持精简，低上下文占用
  - 只负责：读状态文件 → 判断下一步 → 派发任务 → 接收简短结果 → 更新状态文件
  - 不亲自干活

Headless Worker（无头工人）
  - 每次都是全新上下文窗口
  - 干真正的活：实现、测试、审查、修复
  - 干完即终止，只返回简短摘要
  - 项目的记忆不存在这里
```

### 为什么有效

1. **主代理永不膨胀**：主代理只存编排逻辑和简短摘要，不存具体工作过程，上下文始终精简
2. **子代理永远新鲜**：每个 Task 子代理都是全新上下文窗口，不受之前轮次的影响，没有 Context Rot
3. **状态永久持久**：进度存在 PROGRESS.md 里，即使会话崩溃也能从断点恢复
4. **判断独立可靠**：每个子代理在自己的干净上下文中验证，不会被之前轮次的错误判断污染

## 三、TRAE CN 如何完美还原这个模式

TRAE CN 的工具链恰好为这个模式量身定制：

| Orchestrator+Headless 要素 | TRAE CN 对应能力 | 说明 |
| --- | --- | --- |
| Orchestrator | SOLO 模式主对话 | 保持精简，只编排调度 |
| Headless Worker | **Task 子代理** | 每次调用都是全新上下文窗口，干完即返回终止 |
| 状态文件 | PROGRESS.md（项目根目录） | 项目的唯一记忆来源 |
| 决策检查点 | `.claude/checkpoints/` | 重要决策持久化 |
| 跨会话记忆 | MCP Memory（可选） | 跨会话知识图谱 |
| 人工检查点 | AskUserQuestion | 仅阻塞/熔断/完成时触发 |
| 技能复用 | Skills 系统 | 子代理可调用代码需求实现器、软件测试、三轴审查等 |
| 工具连接 | MCP | GitHub、Playwright、Memory 等 |

### 关键洞察

**TRAE CN 的 Task 工具就是天然的 Headless Worker。**

每次调用 Task 工具启动子代理：

- ✅ 全新上下文窗口（不受主对话历史影响）
- ✅ 独立执行（可以调用所有工具）
- ✅ 干完即返回结果终止（不持续占用资源）
- ✅ 只返回最终摘要给主代理（不污染主对话）

这完美对应 Orchestrator+Headless 模式中的 Headless Worker，彻底绕开了 Context Wall。

## 四、与原生 Loop Engineering 的对比

| 维度 | Claude Code /goal（原生Loop） | TRAE CN 循环工程（模拟Loop） |
| --- | --- | --- |
| 上下文管理 | 所有迭代共享一个窗口，会撞墙 | 主代理精简+子代理全新窗口，**彻底绕开Context Wall** |
| 状态持久化 | 对话压缩后可能丢失 | PROGRESS.md永久记录，断点续跑 |
| 目标判断 | 同一个模型判断（可能幻觉） | 子代理在干净上下文中验证（更可靠） |
| 自治程度 | 设定目标后自动跑到完成 | 同样自动跑到完成，带熔断保护 |
| 失败恢复 | 从零开始 | 从最近PROGRESS.md恢复 |
| 过度烘焙防护 | 需手动设循环上限 | 内置熔断机制（单问题5次/总循环上限） |
| 并行能力 | 需 Git WorkTree | Task子代理天然并行（最多5个） |

**结论**：TRAE CN 的模拟方案在上下文管理上甚至**优于**原生 /goal，因为它从架构层面避免了 Context Wall。

## 五、注意事项

### Token 消耗

每个 Task 子代理都是一次完整的 AI 调用，有自己的上下文构建成本。粗粒度调度（一个功能点一个子代理）比细粒度更省 token。

### 调试难度

当循环跑偏时，定位问题比单轮对话难。PROGRESS.md 的"循环历史摘要"和"决策日志"是调试的主要依据——每轮记录不超过3行，便于追溯。

### 子代理间一致性

多个子代理可能做出冲突的决策。通过 PROGRESS.md 的"决策日志"协调——每个子代理启动时先读决策日志，遵循已有决策，不做冲突决策。

### 不要让主代理干活

最常见的违规：主代理忍不住自己写代码/测试。这会迅速膨胀主对话上下文，重新撞上 Context Wall。**主代理只编排，不干活。**
