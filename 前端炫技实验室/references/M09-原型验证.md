# M09 原型验证

## 何时调用本模块

当用户需要在正式投入开发前**用可运行代码回答一个具体问题**时调用本模块。典型触发词包括：原型验证、做个原型、让我试试、试几个设计、prototype、验证数据模型/状态机/交互流程。

本模块产出**一次性代码**，目标是用最小成本验证假设，而非交付生产实现。

## 两条分支

根据要回答的问题选择分支。两条分支的产物完全不同，选错会浪费整个原型。

### 分支 A：逻辑原型（终端交互应用）

适用问题：状态机是否自洽、数据流转是否正确、业务规则在边界情况下是否成立。

产物：一个微型终端交互应用，把状态机推入难以在纸上推理的用例。

- 使用项目现有脚本运行器（`pnpm <name>`、`python <path>`、`bun <path>` 等）。
- 每次操作后打印完整相关状态。
- 状态默认保存在内存中。

### 分支 B：UI 原型（多方案变体切换）

适用问题：界面布局/视觉方向/交互节奏哪个更合理、用户是否能理解当前设计。

产物：在单一路由上生成多个截然不同的 UI 变体，通过 URL 搜索参数和浮动底栏切换。

- 不引入新的顶层路由结构，遵循项目已有路由约定。
- 每次变体切换后渲染完整相关状态或当前变体标识。
- 只验证方向，不验证实现细节。

## 如何选择分支

1. 先识别问题：它关乎**逻辑正确性**还是**外观与体验**？
2. 如果用户不在场且问题模糊，按上下文默认选择：
   - 后端模块、状态机、数据模型 → 分支 A
   - 页面、组件、交互方案 → 分支 B
3. 在原型顶部声明所做的分支假设，方便后续复核。

## 一次性代码规则

- **明确标记为一次性**：命名和位置要让随意浏览的人看出这是原型，而非生产代码。放在被验证模块附近即可。
- **在原型顶部显式列出所有假设**：包括分支选择理由、硬编码的业务规则（如失败几次锁定）、测试凭据来源、范围边界。禁止隐含假设。
- **跳过打磨**：不写测试、不做超出可运行的错误处理、不做抽象。
- **默认无持久化**：状态存于内存。持久化本身是需要验证的东西，不是原型依赖的前提。若问题涉及数据库，使用临时数据库或带明确"原型——请清除"标记的本地文件。
- **暴露状态**：每次操作或变体切换后，必须让用户看到变化后的完整相关状态。
- **一条命令运行**：用户必须能不假思索地启动它。

### 分支 A 示例模板

```js
// prototypes/login-state-machine/index.js
// 分支：A（逻辑原型）
// 假设：连续失败 3 次触发锁定；正确凭据为 demo / demo123；锁定后需重启程序解锁。

const readline = require('readline')

let state = 'IDLE'
let failCount = 0
const MAX_FAIL = 3
const VALID = { username: 'demo', password: 'demo123' }

function transition(input) {
  if (state === 'LOCKED') return { state, failCount, locked: true }
  const ok = input.username === VALID.username && input.password === VALID.password
  if (ok) {
    state = 'SUCCESS'
    failCount = 0
  } else {
    failCount += 1
    state = failCount >= MAX_FAIL ? 'LOCKED' : 'FAILURE'
  }
  return { state, failCount, locked: state === 'LOCKED' }
}

// 交互式终端循环，每次操作后打印 transition 结果
```

## 验证结束后的处理

原型中唯一值得保留的是**答案**。验证完成后：

1. 将结论记录到持久位置（提交消息、ADR、issue 或原型旁边的 `NOTES.md`），并写清楚它回答的问题。
2. 要么删除原型，要么将验证过的决策折叠进真实代码。
3. 不允许原型在仓库中腐烂。

### 从原型到生产代码的迁移提示

- 分支 A（逻辑原型）验证通过后：将状态机迁移到正式状态管理（XState、Reducer Hook、Zustand 等），补充边界测试，移除硬编码凭据与阈值。
- 分支 B（UI 原型）验证通过后：与 M02 设计系统、M03 创意实现或 M07 组件与框架衔接，将选定的 UI 变体落地为生产组件。
- 若用户后续还需要验证交互体验，明确告知当前仅完成逻辑验证，下一步可切到分支 B 或 M03/M07。
