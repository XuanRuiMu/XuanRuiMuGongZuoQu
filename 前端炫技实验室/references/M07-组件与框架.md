# M07 组件与框架

> 定位：复杂多组件前端应用的框架工程指南。覆盖：栈选择、React 19 新范式、Next.js App
> Router、shadcn/ui 组件体系、复杂应用架构、状态与路由、TypeScript 与性能最佳实践。边界：不包含设计 token（M02）、布局/构图基础（M03）、动画规则（M04）、3D/WebGL（M05）、生成艺术（M06）、AI
> SDK（M08）、验证流程（M11/M12）。

---

## 1. 何时调用本模块

在以下场景进入 M07：

- 用户明确要求 React、Next.js、Vue、Svelte 或 shadcn/ui 实现。
- 任务涉及多组件交互、客户端路由、服务端渲染、数据获取、状态管理。
- 需要构建 Dashboard、后台系统、复杂表单、多步骤流程、数据密集型界面。
- 已确定需要现成 accessible 组件库（Dialog、Sheet、DataTable、Chart 等）。

**与 M03 的选择**：Dashboard、后台系统、数据可视化、复杂交互页面即使具备创意展示属性，也优先进入 M07；只有纯展示型落地页、营销页、单页作品才使用 M03 的 `html-tailwind` 单文件方案。

若只是设计系统决策，返回 M02；若涉及动画或 3D，分别转交 M04/M05。

## 2. 栈选择指南

按复杂度与需求递增选择，默认优先最小够用栈。

| 栈               | 适用场景                                                     | 默认条件                                            |
| ---------------- | ------------------------------------------------------------ | --------------------------------------------------- |
| `html-tailwind`  | 单页、静态展示、无复杂状态                                   | 用户未指定栈；页面少于 3 个；无路由需求             |
| `react`          | 组件化、局部状态、表单交互、可复用 UI                        | 用户明确说 React；中型单页应用                      |
| `nextjs`         | 需要路由、SSR/SSG、API Route、Server Actions 的完整 Web 应用 | 用户说 Next.js / 全栈网站 / 多页面应用              |
| `vue` / `svelte` | 团队或用户已指定对应生态                                     | 按用户要求                                          |
| `shadcn/ui`      | 复杂后台、Dashboard、需要现成 accessible 组件库              | 与 Next.js/React 配合使用，仅在多组件复杂架构时启用 |

### 栈选择原则

1. **最小够用**：能用 HTML/Tailwind 完成的，不用 React；能用 React 完成的，不用 Next.js。
2. **框架一致**：同一会话内不混用多个框架；优先使用项目已有技术栈。
3. **默认 React 19 + Next.js 15**：当用户要求现代 React 全栈方案时，默认采用 React 19 与 Next.js App Router。
4. **shadcn/ui 作为实现加速器**：用于复杂 UI 而非设计系统本身，仍需遵循 M02 的 token 与审美规则。

## 3. React 19 specifics

React 19 引入了新的服务端优先范式，使用时应优先采用这些原生能力，而不是回到旧模式。

### 3.1 React Server Components（RSC）

- **默认组件是 Server Component**：在 Next.js App
  Router 中，组件默认在服务端运行，可直接访问数据库、文件系统、环境变量。
- **只在需要客户端交互时使用 `'use client'`**：事件处理、浏览器 API、React
  Context、useState/useEffect 才需要客户端指令。
- **不在 RSC 中使用浏览器 API**：`window`、`document`、`localStorage` 必须放到 Client Component 中。
- **props 必须可序列化**：Server Component 向 Client Component 传递的数据必须是 JSON 可序列化的。

### 3.2 Server Actions

- **优先用 Server Actions 处理变更**：表单提交、数据修改、删除操作优先用 Server Actions，而不是创建 API Route。
- **与表单原生集成**：

  ```tsx
  async function createPost(formData: FormData) {
    "use server";
    const title = formData.get("title");
    // 直接操作数据库
  }
  ```

- **安全边界**：在 Server Action 中校验所有输入；不要信任来自客户端的数据；敏感逻辑必须放在服务端。
- **渐进增强**：Server Actions 天然支持无 JavaScript 的表单提交。

### 3.3 `use` Hook

- **在 Client Component 中读取 Promise**：`use` 用于在 render 阶段消费 Promise、Context，使数据获取更声明式。

  ```tsx
  const data = use(fetchData());
  ```

- **与 Suspense 配合**：`use` 抛出的 Promise 可被最近的 Suspense 边界捕获。
- **不替代所有数据获取**：服务端数据获取仍优先在 RSC 中直接 `await`。

### 3.4 React Compiler（React 19 自动记忆化）

- **默认信任编译器**：React Compiler 自动进行记忆化，优先移除不必要的 `useMemo` / `useCallback` / `memo`。
- **不要为预防而包裹**：除非有实测性能问题，否则不要手动加 `useMemo`。
- **编译器未覆盖的代码**：复杂对象稳定引用、自定义 hook 返回的回调等场景仍可能需要 `useCallback`。
- **启用方式**：在 Next.js 中通过 `reactCompiler: true` 开启。

## 4. Next.js App Router 模式

### 4.1 数据获取

- **RSC 中直接获取数据**：在 Server Component 中直接 `await` 数据库/ORM/API 调用。

  ```tsx
  export default async function Page() {
    const posts = await db.post.findMany();
    return <PostList posts={posts} />;
  }
  ```

- **避免客户端数据瀑布**：不要把本可在服务端获取的数据放到客户端 `useEffect` 中。
- **并行获取独立数据**：相互独立的请求使用 `Promise.all` 并行化。

  ```tsx
  const [users, posts] = await Promise.all([getUsers(), getPosts()]);
  ```

- **请求级去重**：使用 `React.cache()` 对同一请求在单次渲染中 dedupe。

### 4.2 缓存策略

| 层级         | 作用                         | 控制方式                                            |
| ------------ | ---------------------------- | --------------------------------------------------- |
| `fetch` 缓存 | HTTP 请求缓存                | `cache: 'force-cache'` / `'no-store'`               |
| 路由段缓存   | 静态生成 / 动态渲染          | `export const dynamic = 'auto'` / `'force-dynamic'` |
| 数据缓存     | React.cache / unstable_cache | 服务端数据复用                                      |
| 全路由缓存   | 已渲染页面缓存               | `revalidatePath` / `revalidateTag`                  |
| 客户端缓存   | SWR / TanStack Query         | 客户端状态与重新验证                                |

- **默认动态优先**：不确定缓存需求时先使用动态渲染，明确可静态化后再加缓存。
- **缓存失效**：对 Server Action 后的缓存失效使用 `revalidatePath` 或 `revalidateTag`。

### 4.3 并行路由与拦截路由

- **并行路由 (`@folder`)**：在同一布局中同时渲染多个独立页面区域，如 Dashboard 的侧边内容区与主内容区。
- **拦截路由 (`(.)`、`(..)` 等)**：在保持当前布局的同时打开模态框或详情页，实现无缝导航体验。
- **保持 URL 语义清晰**：并行路由不应造成用户无法理解的 URL 结构。

### 4.4 错误与加载边界

- **使用 `loading.tsx`**：为路由段提供 Suspense fallback。
- **使用 `error.tsx`**：为路由段提供错误边界，需是 Client Component。
- **渐进式降级**：错误边界只覆盖局部 UI，不要因小错误导致整个页面崩溃。

## 5. shadcn/ui 使用规则

shadcn/ui 是可复制组件集合，不是传统 UI 库。使用时应遵循其工程约定。

### 5.1 组件发现

- **优先搜索现有组件**：运行 `npx shadcn@latest search <keyword>` 查找官方与社区注册表组件。
- **读取项目上下文**：通过 `npx shadcn@latest info --json`
  获取 aliases、isRSC、tailwindVersion、base、iconLibrary、resolvedPaths 等配置。
- **预览变更**：添加/更新组件前使用 `--dry-run` 和 `--diff`，不手动拉取 GitHub raw 文件。

### 5.2 组合优先

- **不重新发明组件**：Settings = Tabs + Card + 表单控件；Dashboard = Sidebar + Card + Chart + Table。
- **优先使用内置 variant**：`variant="outline"`、`size="sm"` 等。
- **使用语义颜色**：`bg-primary`、`text-muted-foreground`，禁止裸值如 `bg-blue-500`。

### 5.3 样式规则

| 规则     | 正确做法                             | 避免                                                     |
| -------- | ------------------------------------ | -------------------------------------------------------- |
| 颜色     | `bg-primary text-primary-foreground` | `bg-blue-500 text-white`                                 |
| 间距     | `flex gap-4`                         | `space-x-4 space-y-4`                                    |
| 尺寸     | `size-8`                             | `w-8 h-8`                                                |
| 截断     | `truncate`                           | 三段式 `overflow-hidden whitespace-nowrap text-ellipsis` |
| 条件类   | `cn(condition && 'class')`           | 模板字符串三元                                           |
| 深色模式 | 语义 token                           | 手动 `dark:` 覆盖                                        |
| z-index  | 不手动设置                           | 任意 `z-50`                                              |

### 5.4 表单与可访问性

- 使用 `FieldGroup + Field` 模式组织表单。
- 验证状态使用 `data-invalid` + `aria-invalid`。
- 浮层组件（Dialog/Sheet/Drawer）必须包含 Title。
- 图标按钮必须加 `aria-label`。
- Item 必须放在 Group 内；Card 使用完整结构。

### 5.5 图标

- Button 内图标使用 `data-icon="inline-start|inline-end"`。
- 不加尺寸类，传递图标对象而非字符串。
- 统一使用项目配置的图标库（Lucide 等）。

## 6. 复杂多组件应用架构

当需求超出单文件能力时，采用以下多组件架构（源自 Web作品构建）。

### 6.1 启用条件

以下场景跳出单文件方案，启用复杂架构：

1. 多页面或客户端路由需求。
2. 复杂状态管理：跨组件状态、表单状态、异步数据缓存、购物车、仪表盘筛选。
3. 大量使用现成 UI 组件：Dialog、Sheet、Dropdown、Tabs、DataTable、Chart 等。
4. 需要构建工具链：TypeScript、热更新、打包优化、环境变量管理。
5. 团队协作或长期维护：组件化、测试、类型安全成为硬需求。

### 6.2 目录结构基线

```
项目根目录/
├── app/
│   ├── layout.tsx          # 根布局、主题、字体
│   ├── page.tsx            # 首页
│   ├── globals.css         # 全局样式与 CSS 变量
│   ├── (routes)/           # 路由组
│   │   └── dashboard/
│   │       ├── page.tsx
│   │       └── layout.tsx
│   ├── api/                # API Route（如需要）
│   ├── actions/            # Server Actions
│   ├── components/
│   │   ├── ui/             # shadcn/ui 组件
│   │   ├── forms/          # 表单组件
│   │   ├── layout/         # 布局组件
│   │   └── features/       # 功能模块组件
│   ├── hooks/              # 自定义 Hooks
│   ├── lib/                # 工具函数、数据库、API 客户端
│   └── types/              # 全局类型
├── design-system/          # 来自 M02 的 token 与规范（持久化目录）
│   ├── MASTER.md
│   └── pages/
└── tests/                  # 测试文件（如需）
```
### 6.3 组件分层

| 层级 | 职责 | 示例 |
| --- | --- | --- |
| UI 原语 | shadcn/ui、Button、Input、Card | 不处理业务逻辑 |
| 表单层 | 字段封装、验证、提交 | `PostForm`、`LoginForm` |
| 功能层 | 特定业务功能模块 | `UserTable`、`OrderChart` |
| 页面层 | 组合功能模块、处理数据获取 | `DashboardPage` |
| 布局层 | 页面骨架、导航、主题切换 | `RootLayout`、`DashboardLayout` |

### 6.4 代码组织原则

- **每个文件一个主要组件**：避免单个文件堆积多个组件。
- **就近放置测试与样式**：与组件相关的测试、hook、类型放在相邻位置或同目录。
- **公共工具放 `lib/`**：跨模块使用的工具函数、API 客户端、类型定义。
- **业务常量可配置**：状态映射、枚举、默认值抽到常量文件，禁止硬编码。

### 6.5 组件组合模式（Composition Patterns）

复杂可复用组件的 API 设计决定项目的可维护性。以下规则在框架工程层面必须遵守，详细模式与代码示例见 [M18 组件组合模式](M18-组件组合模式.md)。

#### 触发条件

满足以下任一条件时，进入 M18 做详细设计：

- 一个组件出现 3 个以上相互关联的 boolean props，状态组合指数级增长。
- 需要为同一底层状态设计多种可互换的 UI 形态（如只读/编辑/创建）。
- 正在构建项目级组件库，或在 shadcn/ui 之上做领域语义封装。

#### 避免 boolean prop 爆炸

每个 boolean prop 都会让组件潜在状态翻倍。优先用命名变体（Explicit Variants）或 Compound Components 替代 boolean flags。

```tsx
// 反例：boolean prop 爆炸
interface ModalProps {
  isOpen: boolean
  isAlert: boolean
  isFullScreen: boolean
  showCloseButton: boolean
  closeOnOverlayClick: boolean
}

// 正例：命名变体 + Compound Components
<Dialog.Root>
  <Dialog.Trigger>打开</Dialog.Trigger>
  <Dialog.Content>
    <Dialog.Title>确认删除</Dialog.Title>
    <Dialog.Close>取消</Dialog.Close>
  </Dialog.Content>
</Dialog.Root>
```

#### Context 接口拆分为 state / actions / meta

当同一套 UI 需要适配多种数据源或交互行为时，用 Context 约定三类接口，而不是为每种场景复制组件。

```tsx
interface DataTableContext<T> {
  state: { rows: T[]; selectedIds: Set<string>; sort: SortState }
  actions: { toggleSelect: (id: string) => void; setSort: (key: keyof T) => void }
  meta: { rowKey: keyof T & string; columns: ColumnDef<T>[] }
}
```

- `state`：只读数据与派生状态。
- `actions`：修改状态的方法。
- `meta`：不随交互变化的配置。

#### 与 M18 的边界

| 内容 | 归属 |
| ---- | ---- |
| React/Next.js 基础架构、路由、数据获取 | M07 |
| Compound Components、命名变体、Children over Render Props、反模式 | M18 |

M07 在架构设计中决定「何时需要组合模式」，M18 负责「如何具体实现」。

## 7. 状态管理与路由

### 7.1 状态管理选择

| 场景 | 推荐方案 |
| --- | --- |
| 局部 UI 状态 | `useState`、`useReducer` |
| 跨组件共享状态 | React Context + `useReducer` |
| 表单状态 | React Hook Form + Zod |
| 服务端/缓存状态 | Server Actions + revalidate / SWR / TanStack Query |
| 复杂全局状态 | Zustand、Jotai、Redux Toolkit（按团队熟悉度） |

### 7.2 状态管理原则

- **服务端状态优先放在服务端**：用 Server Actions + 缓存替代大量客户端状态。
- **避免过度全局化**：只有真正跨多路由、多组件的状态才放入全局 store。
- **表单状态与展示状态分离**：不要用全局状态管理临时表单值。
- **URL 作为状态来源**：筛选、分页、排序等状态优先同步到 URL query params。

### 7.3 路由

- **Next.js App Router 为默认路由方案**：使用文件系统路由，`page.tsx` 定义页面，`layout.tsx` 定义共享布局。
- **动态路由**：`[id]`、`[...slug]` 用于参数化路径；配合 `generateStaticParams` 生成静态路径。
- **导航使用 `Link`**：客户端导航优先使用 `next/link`，避免整页刷新。
- **程序化导航**：使用 `redirect`（服务端）或 `useRouter`（客户端）。

### 7.4 表单处理

- **优先使用 Server Actions 提交表单**：简单场景无需额外 API Route。
- **客户端校验使用 Zod**：与 Server Action 共享同一个 schema，保证前后端校验一致。
- **错误处理**：Server Action 返回结构化错误，表单组件根据错误显示对应字段提示。

## 8. TypeScript 与性能最佳实践

### 8.1 TypeScript 规范

- **严格模式启用**：`strict: true`，不绕过类型检查。
- **优先推断，显式标注边界**：函数参数、公共 API、返回类型显式标注；局部变量优先推断。
- **避免 `any`**：使用 `unknown` 或具体类型；必须转换时用类型守卫。
- **组件 props 类型**：使用接口或类型别名，props 变化时同步更新。

  ```tsx
  interface ButtonProps extends React.ButtonHTMLAttributes<HTMLButtonElement> {
    variant?: 'default' | 'outline' | 'ghost'
  }
  ```

- **服务端类型安全**：ORM 类型、API 响应类型、表单 schema 类型共享。

### 8.2 性能最佳实践

基于 Vercel React 最佳实践，按优先级执行。详细规则、代码示例与优先级说明见 [M17 React 性能模式](M17-React性能模式.md)。

#### 消除瀑布（CRITICAL）

- 独立操作使用 `Promise.all` 并行化。
- 不要把本可在服务端并行获取的数据串行化。

#### 包体积优化（CRITICAL）

- 直接导入组件，避免 barrel 文件重新导出整个库。
- 使用 `next/dynamic` 懒加载重组件或页面外内容。
- 审查大型依赖，优先使用树摇友好的库。

#### 服务端性能（HIGH）

- 使用 `React.cache()` 进行请求级去重。
- 合理设置缓存策略，避免重复计算与数据库查询。

#### 重渲染优化（MEDIUM）

- 信任 React Compiler，不预防性加 `useMemo`。
- 将昂贵工作提取到独立 memoized 组件。
- 条件渲染使用三元表达式，避免 `&&` 造成的意外渲染。

#### 渲染性能（MEDIUM）

- 大列表使用虚拟化（TanStack Virtual、react-window）。
- 图片使用 Next.js `<Image>` 组件，启用优化与懒加载。

#### JavaScript 性能（LOW-MEDIUM）

- 批量 DOM/样式修改：通过 class 或 `cssText` 一次性修改，减少重排。
- 事件处理函数稳定引用，避免子组件不必要重渲染。

### 8.3 代码质量红线

- 不硬编码颜色、路径、名称、变量；所有可配置项抽到常量或环境变量。
- 错误消息走翻译文件或统一错误体系，不暴露内部实现。
- 非法输入不致崩溃：服务端校验、错误边界、降级 UI。
- 用户输入严格校验：白名单优先于黑名单。

## 10. 执行入口

当本模块被激活时，按以下顺序执行：

1. **前置条件检查**
   - 若项目尚无 M02 设计系统，告知用户需要先进入 [M02 设计系统](M02-设计系统.md) 完成设计规范，**完成后再回到 M07**。禁止在同一轮内同时执行 M02 与 M07。
   - 检查工作区是否已有 Next.js + shadcn/ui 工程结构；若无，先执行初始化（如 `npx shadcn@latest init`），再进入组件实现。
2. **需求澄清**
