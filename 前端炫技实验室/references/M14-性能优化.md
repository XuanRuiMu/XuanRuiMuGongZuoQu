# M14 性能优化

> 定位：前端性能工程指南，聚焦页面加载、交互响应、视觉稳定性与运行时渲染效率。覆盖：Core Web
> Vitals、动画性能、图片与资源优化、包体积、渲染性能、测量与监控。边界：不包含 CSS 动画设计规则（M04）、React/Next.js 框架基础与组件模式（M07）、3D/WebGL 性能（M05）。

---

## 1. 何时激活本模块

当用户需求涉及以下任一方向时，在整体方案中激活 M14：

- 页面加载慢、LCP/INP/CLS 不达标，需要系统性性能优化。
- 需要优化图片、字体、脚本、CSS 等静态资源。
- 需要减少包体积、实现按需加载或提升 tree-shaking 效率。
- 需要优化 React 重渲染、长列表、复杂交互的响应速度。
- 需要建立性能测量流程（Lighthouse、Web Vitals、真实用户监控）。
- 动画出现卡顿、掉帧，需要从合成层、绘制、布局角度排查。

> 边界提示：
>
> - CSS 动画选型、微交互时序、scroll-driven animations → 见 M04
> - React 19 / Next.js / shadcn/ui 框架用法 → 见 M07
> - Three.js / WebGL / Shader 性能 → 见 M05
> - 视觉验证、截图回归 → 见 M11
> - React 并发特性、Suspense 边界、async waterfall、React.cache → 见 M17 React性能模式

M14 处理通用性能问题；当优化深入到 React 并发模式、Suspense 误用、async waterfall、React.cache 等框架特定模式时，在 M14 通用优化完成后建议用户切换至 M17 继续深挖。

## 2. Core Web Vitals 优化

### 2.1 LCP（Largest Contentful Paint）

目标：主视口最大内容元素在 2.5s 内渲染完成。

| 影响因素       | 优化手段                                                                    |
| -------------- | --------------------------------------------------------------------------- |
| 服务器响应时间 | 使用 CDN、Edge Rendering、缓存、压缩；减少后端阻塞                          |
| 渲染阻塞资源   | 关键 CSS 内联；非关键 JS/CSS 延迟加载；`<link rel="preload">` 关键字体/图片 |
| 资源加载时间   | 图片使用现代格式（WebP/AVIF）、响应式尺寸、懒加载非首屏图片                 |
| 客户端渲染     | 首屏关键内容优先服务端渲染或静态生成；延迟非关键 hydration                  |

关键动作：

- 识别真实 LCP 元素（DevTools Performance → Timings → LCP）。
- 对 LCP 图片使用 `fetchpriority="high"`。
- 避免在 LCP 元素上覆盖大型背景图或复杂遮罩。

> **Next.js 专项**
> - 使用 `<Image priority />` 自动设置 `fetchpriority="high"` 并优化加载顺序。
> - 对首屏字体使用 `next/font` 自动子集化与预加载，减少字体切换导致的 CLS。
> - 优先使用 App Router 的 RSC 渲染首屏关键内容，避免客户端数据瀑布。

### 2.2 INP（Interaction to Next Paint）

目标：页面生命周期内 75% 的交互响应延迟低于 200ms。

| 问题来源         | 优化手段                                                                            |
| ---------------- | ----------------------------------------------------------------------------------- |
| 长任务阻塞主线程 | 拆分长任务（`scheduler.yield()`、requestIdleCallback）；将非 UI 工作移入 Web Worker |
| 事件处理过重     | 防抖/节流滚动、resize、input；避免在事件中同步执行复杂计算                          |
| 渲染更新过大     | 减少 DOM 操作范围；批量更新样式；避免强制同步布局                                   |
| 第三方脚本       | 延迟加载非关键第三方库；使用 Partytown 将分析脚本移出主线程                         |

关键动作：

- 使用 Chrome DevTools Performance 面板定位长任务。
- 优先优化首页加载后 200ms 内可能触发的交互（按钮、菜单、输入）。
- 复杂 UI 反馈使用 `requestAnimationFrame` 分批处理。

### 2.3 CLS（Cumulative Layout Shift）

目标：页面生命周期内累计布局偏移分数低于 0.1。

| 问题来源                | 优化手段                                                         |
| ----------------------- | ---------------------------------------------------------------- |
| 图片/视频/iframe 无尺寸 | 始终设置 `width`/`height` 或 `aspect-ratio`                      |
| 字体加载导致 FOUT/FOIT  | 使用 `font-display: optional` 或预加载关键字体；预留字体回退空间 |
| 动态注入内容            | 广告、推荐位、toast 预留固定占位；避免在已有内容上方插入         |
| late-loading CSS        | 关键样式优先内联；避免加载后大幅改变布局                         |

关键动作：

- 对所有可能撑开空间的元素使用占位容器。
- 避免使用会改变文档流的动画（width/height/top/left/margin/padding）。
- 交互导致的布局变化应在 500ms 内完成，避免用户预期外偏移。

## 3. 动画性能

> 本节只讨论动画对渲染管线的影响与性能原则，不涉及动画设计本身。

### 3.1 只动画合成属性

浏览器渲染管线包括：JavaScript → Style → Layout → Paint → Composite。只触发 Composite 阶段的属性性能最好。

| 推荐属性（仅触发 Composite）          | 避免属性（触发 Layout/Paint）                       |
| ------------------------------------- | --------------------------------------------------- |
| `transform`（translate/scale/rotate） | `width`、`height`、`top`、`left`、`right`、`bottom` |
| `opacity`                             | `margin`、`padding`                                 |
| 部分 `filter`（谨慎）                 | `background`、`color`（触发 Paint）                 |
|                                       | `box-shadow` 频繁变化                               |
|                                       | 大面积 `backdrop-filter: blur()`                    |

原则：

- 位移动画使用 `transform: translate3d(x, y, 0)` 触发 GPU 加速。
- 缩放动画使用 `transform: scale()`，避免修改 width/height。
- 避免同时动画大量元素，防止合成层（layer）爆炸。

### 3.2 will-change 使用规则

`will-change` 提示浏览器提前准备合成层，但滥用会造成内存与性能浪费。

- 仅在即将发生动画的元素上添加，动画结束后移除。
- 不要对大量元素预加 `will-change`。
- 不要在动画外长期保留 `will-change`。
- 优先通过 `transform`/`opacity` 隐式触发合成，必要时再用 `will-change`。

```css
/* 动画前添加 */
.animated {
  will-change: transform, opacity;
}

/* 动画结束后通过 JS 移除 will-change */
```

### 3.3 减少运动偏好

必须尊重 `prefers-reduced-motion`，具体媒体查询实现见 M04。

性能相关要点：

- 对运动敏感用户，禁用视差、大幅度位移动画、自动播放循环。
- 减少运动后，关键内容仍需立即可见，不能依赖动画完成状态。
- 循环动画在元素离屏时应暂停，减少不必要的 GPU 占用。

## 4. 图片与资源优化

### 4.1 图片优化

| 策略       | 说明                                                                 |
| ---------- | -------------------------------------------------------------------- |
| 格式选择   | 照片用 AVIF/WebP；图标用 SVG；简单图形用 CSS                         |
| 响应式尺寸 | 使用 `srcset` + `sizes`，避免移动端加载桌面大图                      |
| 懒加载     | 首屏外图片使用 `loading="lazy"`；首屏图片使用 `fetchpriority="high"` |
| 解码       | 大图使用 `decoding="async"` 避免阻塞主线程                           |
| 占位       | 使用低质量占位图（LQIP）或纯色占位，防止 CLS                         |
| CDN 转换   | 使用支持格式转换与尺寸裁剪的图像 CDN                                 |

原则：

- 图片尺寸不超过实际显示尺寸的 2x。
- 避免在 CSS 中通过 `background-image` 加载首屏大图。
- 对关键图片使用 `<link rel="preload" as="image">`。

> **Next.js 专项**：使用 `next/image` 自动处理 WebP/AVIF 转换、响应式 `srcset`、懒加载与占位，避免手写 `<img>`。

### 4.2 字体优化

- 限制字体族数量，优先使用系统字体栈作为回退。
- 使用 `font-display: swap` 或 `optional` 控制加载行为。
- 预加载首屏关键字体：`<link rel="preload" as="font" type="font/woff2" crossorigin>`。
- 使用 `size-adjust`、`ascent-override`、`descent-override` 减少字体切换导致的 CLS。
- 子集化字体文件，只包含所需字符。

### 4.3 脚本与样式

- 关键 CSS 内联到 `<head>`，非关键 CSS 异步加载。
- JS 脚本默认 `defer` 或 `type="module"`；非首屏交互脚本使用 `async` 或动态导入。
- 移除未使用的 CSS/JS；定期审计第三方脚本体积。
- 使用 `preload`、`prefetch`、`modulepreload` 合理预加载资源。

## 5. 包体积优化

### 5.1 Tree-shaking

- 使用 ES Modules 导入，避免 CommonJS 全量导入。
- 直接导入具体模块，避免 barrel 文件重新导出整个库。
- 选择 tree-shaking 友好的库；定期检查 `node_modules` 体积。
- 避免在全局导入 polyfill，按浏览器特性按需加载。

```js
// 避免
import { every } from "lodash";

// 推荐
import every from "lodash/every";

// 或改用 tree-shaking 友好的替代库
import { every } from "lodash-es";
```

### 5.2 动态导入与代码分割

- 路由级自动代码分割（Next.js、Vue Router 等默认支持）。
- 对重组件、模态框、图表、编辑器使用动态导入。
- 使用 `next/dynamic`、`React.lazy()`、`import()` 实现按需加载。
- 动态导入配合 Suspense 边界，提供加载状态。

> **Next.js 专项**：对非首屏重组件使用 `next/dynamic` 并配合 `ssr: false`（仅客户端需要时）；对第三方脚本使用 `next/script` 控制加载策略。

```tsx
import dynamic from "next/dynamic";

const HeavyChart = dynamic(() => import("./HeavyChart"), {
  ssr: false,
  loading: () => <ChartSkeleton />,
});
```

### 5.3 依赖审计

- 使用 `webpack-bundle-analyzer`、`@next/bundle-analyzer` 可视化包体积。
- 替换体积过大的库（如 moment.js → date-fns）。
- 避免重复依赖版本，统一锁定版本。
- 将大型纯数据文件（JSON、字典）异步加载或按页拆分。

## 6. 渲染性能

> 本节聚焦运行时渲染效率，不展开 React 组件模式与框架用法。

### 6.1 避免不必要的重渲染

- 使用 React Compiler（React 19）自动记忆化，优先移除不必要的 `useMemo` / `useCallback` / `memo`。
- 将昂贵计算提取到独立 memoized 组件或 hook。
- 条件渲染使用三元表达式，避免 `&&` 造成的意外空值渲染。
- 保持事件处理函数与对象引用的稳定性，避免向下传递新引用导致子组件重渲染。

### 6.2 长列表卡顿根因速查

| 卡顿表现                 | 根因                 | 优化手段                                          |
| ------------------------ | -------------------- | ------------------------------------------------- |
| 滚动时帧率明显降低       | DOM 节点过多         | 使用虚拟化（TanStack Virtual、react-window）      |
| 滚动/输入后延迟响应      | 事件处理过重         | 防抖/节流 scroll/resize/input；使用 scheduler.yield |
| 轻微交互就触发大面积重绘 | 重渲染频繁           | React Compiler / memo / 稳定事件引用              |
| 列表数据量极大且计算复杂 | 主线程被大数据转换阻塞 | 数据转换移入 Web Worker；分页或无限滚动           |

### 6.3 大数据量渲染

- 长列表使用虚拟化（TanStack Virtual、react-window）。
- 表格大数据集使用分页、无限滚动或虚拟滚动。
- 避免一次性渲染大量 SVG、Canvas 节点或复杂 DOM 结构。
- 大数据转换在 Web Worker 中执行，避免阻塞主线程。

### 6.4 DOM 与样式批量更新

- 通过 class 或 `cssText` 一次性批量修改样式，减少重排次数。
- 避免在 `requestAnimationFrame` 中交替读取和写入布局属性（Layout Thrashing）。
- 需要读取布局时，先批量读取，再批量写入。

```js
// 避免：读写交替
for (const el of elements) {
  const height = el.offsetHeight; // 读
  el.style.height = height + 10 + "px"; // 写
}

// 推荐：先读后写
const heights = elements.map((el) => el.offsetHeight);
elements.forEach((el, i) => {
  el.style.height = heights[i] + 10 + "px";
});
```

### 6.5 服务端渲染与 Hydration

- 首屏关键内容优先 SSR/SSG，减少客户端初始渲染压力。
- 对非关键交互组件使用 `React.lazy` 或延迟 hydration。
- 避免在 hydration 阶段执行大量同步计算。

## 6.6 性能优化对可访问性的影响清单

每项性能优化实施前，评估其对可访问性的潜在影响，并记录供 M13 复核：

| 优化手段             | 潜在 a11y 影响                       | 缓解措施                                  |
| -------------------- | ------------------------------------ | ----------------------------------------- |
| 图片懒加载           | 屏幕阅读器可能跳过未加载图片         | 使用原生 `loading="lazy"` + 完整 `alt`     |
| 字体 `font-display: optional` | 自定义字体可能完全不显示       | 确保系统字体回退可读                        |
| 动态导入 / Suspense  | loading 状态未播报                   | 为 loading 区域添加 `aria-busy` / `aria-live` |
| 减少动画 / 低功耗模式 | 关键信息不依赖动画呈现               | 确保静态状态下信息完整可见                  |
| Canvas/WebGL 降级    | 替代内容是否充分                     | 提供等效文本或数据表格                      |

## 7. 测量与监控

### 7.1 Lighthouse

- 使用 Chrome DevTools Lighthouse 面板或 PageSpeed Insights 跑分。
- 关注 Performance、Accessibility、Best Practices、SEO 四项分数。
- 以移动设备分数为主要优化目标。
- 对每项诊断建议按影响与成本排序，优先处理高影响低成本的项。

### 7.2 Web Vitals 库

- 使用 `web-vitals` 库在真实用户环境中采集 LCP、INP、CLS、FCP、TTFB。
- 将数据发送到分析平台（Google Analytics、Sentry、Datadog 等）。

```js
import { onLCP, onINP, onCLS } from "web-vitals";

onLCP(console.log);
onINP(console.log);
onCLS(console.log);
```

### 7.3 运行时性能工具

- Chrome DevTools Performance 面板：定位长任务、重排、重绘、合成层。
- Chrome DevTools Rendering 面板：实时显示 FPS、重排区域、图层边界。
- Memory 面板：排查内存泄漏、Detached DOM、 retained size。
- Network 面板：分析资源加载瀑布、缓存命中率、压缩比例。

### 7.4 性能预算

- 为首页 JS/CSS/图片体积设定上限。
- 在 CI 中集成 Lighthouse CI 或 `bundlesize`，防止回归。
- 将 Core Web Vitals 作为发布门禁之一。
