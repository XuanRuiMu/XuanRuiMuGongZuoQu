# M02 设计系统

> 本模块负责在编码前建立统一、可维护、具备辨识度的视觉语言。所有前端创作必须先完成设计系统，再进入 M03 实现。

---

## 1. 何时激活本模块

| 场景               | 说明                                                            |
| ------------------ | --------------------------------------------------------------- |
| 前端/UI 创作起点   | 用户要求做网页、落地页、仪表盘、React/Vue 组件、HTML/CSS 布局等 |
| 主题/配色/字体需求 | 用户要求“换个风格”“配个色”“选字体”“做品牌样式”                  |
| 跨会话规范         | 需要把当前设计决策持久化，供后续页面或团队复用                  |
| 品牌/预设应用      | 需要直接套用 Anthropic 品牌或 10 个预设主题                     |

**原则**：设计系统不是可选装饰，而是每个网页项目的第一交付物。

### 1.1 入口执行流程

本模块被激活后，按以下顺序执行：

1. **确认需求与资料**：先向用户确认产品类型、目标受众、情感关键词、可参考竞品/品牌资料。若用户未提供，**禁止直接假设生成**，必须用第 2.1 节的 4 个问题澄清。
2. **定义主题锚定**：用一个具体概念（如「瑞士国际主义排版 + 单色调 + 大留白」）锚定整个设计。
3. **生成并验证设计系统**：按第 2-6 节生成色板、字体、质感、signature element、反 AI 同质化规则。
4. **持久化输出**：将结果写入 `design-system/MASTER.md`（模板见 5.1），页面级覆盖写入 `design-system/pages/<page>.md`。
5. **交付确认**：按 AGENTS.md 要求报告完成内容、验证项、遗留问题、置信度、未验证项，并询问是否进入 M03/M04/M07。

## 2. 生成设计系统

### 2.1 查询格式

通过联网搜索、项目上下文或用户提供的品牌资料生成带推理的设计系统推荐。可围绕以下问题展开研究：

- 产品类型与行业调性是什么？
- 目标受众的审美偏好是什么？
- 希望传递的情感关键词是什么？（如优雅、先锋、温暖、冷静）
- 有哪些可参考的视觉风格或竞品？

将研究结论整理为设计系统文档，包含：运动名称、色板、字体、空间原则、反同质化规则。

> **分支规则**：若用户无法回答上述 4 个问题，必须先追问澄清，禁止基于默认值生成设计系统。只有在获得用户确认或明确授权后，才允许基于搜索直接生成。

### 2.2 搜索域

默认并行搜索 5 个域，覆盖美学决策的主要维度：

| 域           | 解决什么问题       | 示例查询                                        |
| ------------ | ------------------ | ----------------------------------------------- |
| `product`    | 产品类型与行业调性 | SaaS、电商、作品集、金融科技                    |
| `style`      | 视觉风格方向       | glassmorphism、brutalist、minimal、editorial    |
| `color`      | 专属色板           | warm earth tone、luxury gold、ocean deep        |
| `landing`    | 页面结构与叙事节奏 | hero social-proof、asymmetric hero              |
| `typography` | 字体搭配与层级     | elegant luxury、playful display、technical mono |

按需补充的细粒度域：

| 域      | 用途                 | 示例                    |
| ------- | -------------------- | ----------------------- |
| `chart` | 数据可视化配色与样式 | real-time dashboard     |
| `ux`    | 交互与可用性最佳实践 | animation accessibility |

### 2.3 输出内容

一次完整的设计系统生成应至少包含：

- **Pattern（视觉模式）**：整体构图倾向，如不对称网格、杂志式分屏、全出血大图等
- **Style（风格关键词）**：brutalist / maximalist / retro-futuristic / luxury / playful / editorial 等
- **Colors（色板）**：主色、强调色、中性色、语义色、双模式映射
- **Typography（字体）**：展示字体、正文字体、数据字体、字号层级
- **Effects（质感）**：纹理、噪点、渐变、玻璃、阴影策略
- **Anti-patterns（需避免的反模式）**：本模块第 6 节列出的 AI 同质化元素

### 2.4 Signature Element（标志性元素）

每个设计系统必须定义至少一个 **signature element**——即用户离开后仍能记住的视觉锚点。它不是通用装饰，而是与该主题强绑定的专属表达。

- 可以是一个形状（如特定圆角半径、裁切路径）、一种纹理、一种排版处理方式、一种交互反馈。
- 必须在整个页面中重复出现 3 次以上，形成节奏，但不能滥用。
- 应与产品气质直接相关：金融科技可用精确的几何切角；生活方式品牌可用有机 blob；科技产品可用单色噪点 + 等宽数据字体。

**示例**：

```
signature element = "16px 大圆角卡片 +  subtle grain 纹理 + 品牌暖橙作为唯一高亮色"
禁止同时出现：蓝紫渐变、发光阴影、通用 emoji 图标
```

### 2.5 两遍工作法（Two-Pass Design）

Anthropic 的设计流程要求至少做两轮：

1. **第一遍：安全但平庸的方案**
   - 使用熟悉的布局、通用色板、标准组件。
   - 目标是快速验证信息架构与功能可行性，不追求美感。
2. **第二遍：有辨识度的方案**
   - 在第一遍基础上，强制应用 signature element、不对称布局、品牌专属色板。
   - 删除所有通用 AI 默认元素（蓝紫渐变、Inter 唯一字体、三卡片 hero 等）。

**执行规则**：

- 不在第一遍停留；第一遍只用于对齐功能与内容。
- 第二遍必须回答：「这个页面如果没有 logo，用户还能认出是我们的产品吗？」
- 若答案为否，继续深化 signature element 与主题锚定。

### 2.6 主题锚定（Theme Anchoring）

用一个具体、可描述的概念锚定整个设计，避免抽象形容词堆砌。

| 抽象描述 | 锚定后 |
| -------- | ------ |
| 现代、简洁 | 瑞士国际主义排版 + 单色调 + 大留白 |
| 科技、未来 | 深色底 + 霓虹青数据高亮 + 等宽字体 + 网格线 |
| 温暖、自然 | 陶土色 + 纸张纹理 + 圆润衬线 + 植物绿点缀 |
| 奢华、精致 | 高对比衬线 + 香槟金 + 大量留白 + 微阴影 |

锚定后，所有后续决策（色板、字体、动效、图标）都用这个概念过滤。不符合锚定的元素即使好看也应删除。

## 3. 色板与主题

### 3.1 强制双模式（浅色 + 深色）

每个网页都必须同时定义浅色模式与深色模式：

- 浅色为默认；深色通过切换启用。
- 两种模式都需要**完整 token 映射**，不能只改背景色。
- 双模式均满足 WCAG 2.1 AA：正文 4.5:1，大文本/UI 组件 3:1。
- 切换偏好应持久化，并尊重 `prefers-color-scheme`。

### 3.2 语义 Token 映射

使用语义 token 描述界面元素，而不是直接写裸颜色值：

| Token          | 浅色模式语义           | 深色模式语义     |
| -------------- | ---------------------- | ---------------- |
| `background`   | 页面背景，偏冷白或米白 | 近黑或深 slate   |
| `surface`      | 卡片、面板、浮层       | 略亮的深色面     |
| `border`       | 分割线、边框           | 低对比度边框     |
| `text-primary` | 主标题、正文           | 高亮文本         |
| `text-muted`   | 辅助说明、元信息       | 降亮度辅助文本   |
| `accent`       | CTA、链接、高亮        | 同色相调整亮度   |
| `glass`        | 半透明毛玻璃背景       | 深色半透明毛玻璃 |
| `hover`        | 悬停背景               | 深色悬停背景     |

### 3.3 2026 现代颜色工作流

优先采用以下现代 CSS 颜色能力生成和维护色板：

- **OKLCH**：用感知均匀的色彩空间定义色阶，避免传统 HSL 的亮度漂移。
- **`color-mix()`**：通过混合主色与白色/黑色/中性色生成衍生色，不依赖手动 hex。
- **相对颜色（Relative Colors）**：使用 `from ...` 语法从基础 token 派生变体，保持单一来源。
- **`light-dark()`**：把同一语义 token 的浅色值与深色值合并为一行声明，减少重复类名。
- **容器感知 Token**：结合容器查询单位（`cqi`、`cqw`）或流体函数，让字号、间距、圆角随组件上下文缩放。

**禁用默认陷阱**：

- 蓝紫渐变背景（`from-indigo-500 to-purple-600`）
- Tailwind 默认紫色/靛蓝作为强调色
- 发光阴影（`shadow-blue-500/20`）作为主要 affordance
- 单一深色页面或没有明确深色映射

## 4. 字体搭配规则

### 4.1 核心原则

- **至少两种字体**：一种负责展示/标题，一种负责正文；数据密集场景可加等宽字体。
- **避免通用默认**：不把 Inter / Roboto / Arial 作为唯一字体。
- **戏剧性层级**：标题与正文尺寸对比明显（例如 72px 标题 vs 16px 正文）。
- **字重克制**：整体使用 2-3 种字重，避免全页面都是 400/500。
- **可读性回退**：自定义字体必须提供系统字体回退栈。

### 4.2 2026 排版增强

| 场景             | 推荐处理                                      |
| ---------------- | --------------------------------------------- |
| 标题             | `text-balance`，避免孤行                      |
| 正文             | `text-pretty`，提升换行美感                   |
| 数据、表格、价格 | `tabular-nums`，保持数字对齐                  |
| 展示标题         | 可考虑可变字体（variable font）实现响应式字重 |

### 4.3 字体与品牌气质

| 气质         | 展示字体方向         | 正文字体方向 |
| ------------ | -------------------- | ------------ |
| 奢华精致     | 高对比衬线、优雅手写 | 温润衬线     |
| 科技创新     | 几何无衬线、等宽     | 人文无衬线   |
| 活泼 playful | 圆润无衬线、装饰体   | 清爽无衬线   |
| 编辑/杂志    | 强个性衬线           | 高可读性衬线 |

## 5. 主题持久化与品牌样式

### 5.1 设计系统持久化

将生成的设计系统写入项目级规范，便于后续模块引用：

- `design-system/MASTER.md`：全局设计规范
- `design-system/pages/<page>.md`：页面级覆盖（如 `dashboard.md`）

`MASTER.md` 推荐包含以下必填字段：

```markdown
# <项目名称> 设计系统

## 1. 主题锚定
<用一句话描述设计概念，如「瑞士国际主义排版 + 单色调 + 大留白」>

## 2. Pattern & Style
- 构图倾向：...
- 风格关键词：...

## 3. Colors
- 主色：...
- 强调色：...
- 中性色：...
- 语义 token（浅色/深色）：...

## 4. Typography
- 展示字体：...
- 正文字体：...
- 数据字体：...
- 字号层级：...

## 5. Effects
- 纹理/噪点：...
- 渐变策略：...
- 阴影策略：...

## 6. Signature Element
- 标志性元素：...
- 使用位置：...

## 7. Anti-patterns
- 禁用模式：...
- 必须应用的差异化技术：...
```

后续实现具体页面时，优先使用对应页面文件覆盖 MASTER。如需跨项目复用，可将 MASTER.md 保存为模板。

## 6. 反 AI 同质化规则

### 6.1 禁用模式（绝不使用）

| 禁用模式                                         | 原因                | 替代方案                                |
| ------------------------------------------------ | ------------------- | --------------------------------------- |
| 蓝紫渐变背景                                     | 第一 AI 识别特征    | 产品专属暖色、大地色或品牌渐变          |
| `bg-gradient-to-r from-indigo-500 to-purple-600` | 最通用 AI 渐变      | 从设计系统获取专属渐变                  |
| 仅深色页面                                       | “AI 开发者工具”审美 | 强制双模式，浅色默认                    |
| Emoji 作为 UI 图标                               | 立即识别为 AI 输出  | SVG 图标库（Lucide、Heroicons、Tabler） |
| Inter/Roboto 作为唯一字体                        | AI 的“安全”默认     | 设计系统推荐的字体搭配                  |
| 套路式 hero + 3 卡片 + CTA                       | 最通用 AI 布局      | 变化章节结构，使用不对称布局            |
| 统一 `rounded-2xl` 到处使用                      | AI 对圆角过度应用   | 区分卡片、头像、弹窗的圆角              |
| 发光阴影（`shadow-blue-500/20`）                 | AI 标志性 glow      | 中性 subtle shadow                      |
| 紫色/紫罗兰强调色                                | AI 最常见强调色     | 产品专属强调色                          |

### 6.2 必须应用的差异化技术（至少 3 项）

1. **独特色板**：所有颜色来自 `--design-system`，而非 Tailwind 默认色盘。
2. **不对称布局**：打破居中 hero + 三卡片套路，使用偏移网格、杂志式排版、分屏。
3. **排版层次**：使用 2-3 种字重与戏剧性尺寸对比。
4. **微交互**：有目的的 hover、focus、active 状态，过渡 150-300ms。
5. **留白设计**：充裕 padding / margin，避免拥挤布局。
6. **纹理/图案**：微妙噪点、网格、点阵、几何图案替代纯色。
7. **多样化章节结构**：全宽、受限宽度、分屏、交错排版交替出现。

### 6.3 识别三种 2026 AI 默认外观

当前生成式前端输出最常见的三种默认审美，设计系统阶段必须主动识别并替换：

| 默认外观 | 特征 | 替换策略 |
| -------- | ---- | -------- |
| **SaaS 蓝紫渐变** | 深蓝到紫的 hero 渐变、发光按钮、白色卡片浮于其上 | 用品牌专属暖色/大地色渐变，或纯色 + 纹理 |
| **深色开发者工具** | 全黑背景、霓虹青/绿强调、等宽字体、终端感 | 若品牌不是开发者工具，改用浅色默认 + 品牌强调色 |
| **极简三栏落地页** | Hero + 3 特性卡片 + 社会证明 + CTA | 打破对称：偏移网格、杂志式分栏、非线性阅读动线 |

识别方法：生成第一遍方案后，截图并问自己「这是否像 2026 年任意一个 AI 生成的落地页？」若是，则进入第二遍，强制应用 signature element 与主题锚定。

## 7. 2026 补充：现代 Token 工作流清单

- [ ] 色板基于 OKLCH 定义，色阶亮度一致
- [ ] 衍生色使用 `color-mix()` 或相对颜色从基础 token 生成
- [ ] 双模式语义 token 优先使用 `light-dark()` 合并
- [ ] 关键 token（字号、间距、圆角）具备容器感知能力
- [ ] 不使用 Tailwind 默认紫色/靛蓝作为强调色
- [ ] 不使用蓝紫渐变或 glow 阴影作为主要 affordance
- [ ] 字体搭配来自设计系统，而非 Inter/Roboto 默认
- [ ] 双模式均通过 WCAG 2.1 AA 对比度要求
