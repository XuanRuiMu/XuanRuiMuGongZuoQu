# M12 应用测试

## 用途

使用 Playwright 对 Web 应用进行功能性、端到端（E2E）和组件级测试，验证用户交互、页面状态流转、数据展示与业务逻辑是否符合预期。

本模块只处理「功能逻辑正确性」，不涉及视觉设计质量审查（见 M11）、可访问性修复（见 M13）、性能指标优化（见 M14）。

> **与通用「软件测试」skill 的边界**：复杂、多模块或需要极限覆盖/测试审计的工程化测试任务，应优先或并行调用通用「软件测试」skill；M12 只负责前端展示型项目的快速功能验证与 Playwright 脚本编写。
>
> 满足以下任一条件时，推荐并行调用通用「软件测试」skill：
> - 需要覆盖 5 个以上独立用户流程。
> - 需要测试覆盖率审计、缺口分析或极限边界覆盖。
> - 涉及后端 API 契约测试、数据库状态验证或多服务集成测试。
> - 用户明确要求 TDD、红绿重构或全量回归测试策略。

## 触发条件

- 用户说：测试网页、前端测试、UI 测试、Playwright 测试、浏览器测试、网页调试、功能验证、E2E 测试、写测试脚本、自动化测试
- 需要验证某个用户流程是否能正常跑通（登录、下单、表单提交、搜索、分页等）
- 需要确认按钮、输入框、弹层、路由切换等业务交互行为正确
- 前端实现完成后，需要补全自动化功能测试

## 测试目标与范围定义

### 确定测试层级

| 层级         | 适用场景                 | 工具/方式                                    |
| ------------ | ------------------------ | -------------------------------------------- |
| E2E 测试     | 完整用户流程，跨多个页面 | Playwright 页面级脚本                        |
| 组件测试     | 单个交互组件的状态与事件 | Playwright Component Tests 或 Storybook 测试 |
| API 契约测试 | 前后端数据接口           | 配合 `page.route` 拦截与 mock                |

### 划定一次测试的最小范围

1. 选择一个**单一用户目标**，例如：用户能完成登录。
2. 列出完成该目标必须经过的**关键步骤**。
3. 为每一步定义**可观察的断言条件**（DOM 元素、URL、文本内容、网络状态）。
4. 明确**前置数据状态**（是否需要已登录用户、是否需要 mock 后端）。
5. 明确**通过/失败标准**。

### 范围检查清单

- [ ] 本次测试覆盖的功能路径不超过一个完整用户目标
- [ ] 关键步骤都有可验证断言
- [ ] 测试不依赖未稳定的外部服务，必要时使用 mock
- [ ] 测试数据与生产数据隔离

## Playwright 功能测试流程

### 第 1 步：确认目标 URL 与入口

询问用户或根据上下文确定被测页面 URL。若 URL 未提供，先用常见占位地址（如 `http://localhost:3000/login`）编写示例脚本，同时明确提示用户替换为实际地址，并停止等待用户确认后再执行。

### 第 2 步：分析关键交互路径

将用户目标拆分为以下元素：

| 元素     | 示例                                      |
| -------- | ----------------------------------------- |
| 入口状态 | 未登录用户访问首页                        |
| 操作序列 | 点击登录按钮 → 输入用户名/密码 → 点击提交 |
| 期望状态 | URL 跳转到 `/dashboard`，页面显示用户名   |
| 异常分支 | 密码错误时显示错误提示                    |

### 第 3 步：编写测试脚本

使用 Playwright Python、Node.js / JavaScript 或 MCP 工具编排脚本。以下示例以 Python 为主，Playwright 的 JS/TS API 完全等价，可根据项目技术栈选择。优先使用语义化定位器，避免依赖脆弱选择器。

#### 推荐定位策略（按稳定性排序）

1. `get_by_role`（role + name）
2. `get_by_text` / `get_by_label`
3. `get_by_test_id`
4. CSS 选择器（仅作为兜底）

#### 脚本结构示例

```python
from playwright.sync_api import sync_playwright, expect

def test_login_flow():
    with sync_playwright() as p:
        browser = p.chromium.launch(headless=True)
        page = browser.new_page()

        # 1. 进入入口
        page.goto("http://localhost:3000/login")

        # 2. 执行操作
        page.get_by_label("用户名").fill("testuser")
        page.get_by_label("密码").fill("correct-password")
        page.get_by_role("button", name="登录").click()

        # 3. 断言结果
        expect(page).to_have_url("http://localhost:3000/dashboard")
        expect(page.get_by_text("欢迎, testuser")).to_be_visible()

        browser.close()
```

#### 异常分支断言示例

```python
def test_login_failure_shows_error():
    with sync_playwright() as p:
        browser = p.chromium.launch(headless=True)
        page = browser.new_page()

        page.goto("http://localhost:3000/login")
        page.get_by_label("用户名").fill("testuser")
        page.get_by_label("密码").fill("wrong-password")
        page.get_by_role("button", name="登录").click()

        # 断言仍停留在登录页
        expect(page).to_have_url("http://localhost:3000/login")
        # 断言错误提示可见（按实际文案调整）
        error = page.get_by_role("alert").or_(page.get_by_text("用户名或密码错误"))
        expect(error).to_be_visible()

        browser.close()
```

#### 方案选择

- 如果项目已有 Python 测试套件或用户指定 pytest，使用 Python 脚本并保存为 `.py` 文件。
- 如果项目使用 JS/TS 或用户已配置 `npx playwright test`，使用 JS/TS 脚本。
- 如果只是单次快速验证且 Trae CN 已启用 `mcp_Playwright`，使用 MCP 工具逐步编排。

#### 使用 mcp_Playwright 编排的等价步骤

1. `playwright_navigate` 打开目标 URL
2. `playwright_fill` 输入表单
3. `playwright_click` 触发按钮
4. `playwright_evaluate` 读取 DOM 状态并断言

### 第 4 步：执行测试

- 单次测试：直接运行脚本或逐个调用 MCP 工具
- 回归测试：将脚本保存为项目测试文件，使用 `pytest`（Python）或 `npx playwright test`（JS/TS）或 CI 执行
- 多浏览器：必要时在 chromium / firefox / webkit 中复现

### 第 5 步：分析结果与报告

- 通过的断言记录为 PASS
- 失败的断言记录为 FAIL，附带：
  - 失败步骤
  - 期望结果
  - 实际结果
  - 相关日志与截图路径

## 编写 UI 行为测试脚本

### 覆盖场景

| 场景 | 测试要点 |
| --- | --- |
| 表单提交 | 输入合法/非法值，验证提交后状态、错误提示、重置行为 |
| 按钮交互 | 点击后触发的事件、loading 状态、禁用状态 |
| 导航与路由 | 链接跳转、浏览器回退、URL 参数变化 |
| 弹层与抽屉 | 打开/关闭、焦点管理、ESC 关闭、点击蒙层关闭 |
| 列表与分页 | 排序、筛选、翻页、空状态 |
| 异步数据 | loading、空状态、错误重试、数据刷新 |
| 本地存储/Cookie | 登录态保持、主题偏好、语言设置 |

### 断言编写原则

- 断言应验证**用户可观察的结果**，而非内部实现细节
- 优先使用 Playwright 内置 `expect`，避免裸 `assert`
- 对异步状态使用 `expect(...).to_be_visible(timeout=...)` 自动重试
- 每个测试用例结束时清理产生的数据状态（logout、清空表单、删除测试记录）

### Mock 外部依赖

当后端服务不稳定或测试需要特定数据时，使用 `page.route` 拦截请求：

```python
page.route("**/api/user/profile", lambda route: route.fulfill(
    status=200,
    content_type="application/json",
    body='{"id":1,"name":"testuser"}'
))
```
## 执行测试并分析结果

### 本地执行

```bash
# 单文件
pytest tests/test_login.py --headed --slowmo 500

# 全部测试
pytest

# 生成报告
pytest --html=report.html --self-contained-html
```
### 结果分析

| 结果 | 处理动作 |
| --- | --- |
| 全部 PASS | 输出测试通过摘要，建议进入下一模块或交付 |
| 部分 FAIL | 定位最早失败的断言，收集截图与日志，进入调试 |
| 测试不稳定 | 检查异步等待、网络超时、测试数据冲突，增加重试或隔离 |
| 脚本本身报错 | 检查定位器是否过期、URL 是否正确、依赖服务是否启动 |

### 失败根因分类

| 类型 | 特征 | 常见修复 |
| --- | --- | --- |
| 定位失败 | 元素未找到或选择器失效 | 换用 role/text/label 定位，增加等待 |
| 时序失败 | 元素存在但状态未就绪 | 使用 expect 重试，显式等待网络空闲 |
| 数据失败 | 断言值与预期不符 | 检查 mock 数据、初始化状态、并发影响 |
| 行为失败 | 页面响应不符合需求 | 报 bug 给开发，附带复现脚本 |

## 浏览器日志与截图收集

### 日志收集

使用 `playwright_console_logs` 获取日志：

```json
{
  "type": "all",
  "limit": 100
}
```
功能测试重点关注：

- JavaScript 运行时错误
- 网络请求失败（4xx/5xx）
- 前端路由异常
- 未捕获的 Promise 拒绝

将日志按来源分类：

| 来源 | 是否优先排查 |
| --- | --- |
| 被测应用自身报错 | 是 |
| 第三方 SDK/埋点报错 | 视是否影响功能而定 |
| 浏览器扩展产生 | 否，可忽略 |

### 截图收集

在关键断言失败时自动截图，便于回溯：

```python
page.screenshot(path=".test-debug/login-fail.png", full_page=True)
```
保存目录建议使用项目根目录下的 `.test-debug/`，命名规则：

```
{test-name}-{step}-{timestamp}.png
```
### 调试流程

1. 复现失败：以相同参数重新运行失败用例
2. 查看失败时截图：确认页面当时状态
3. 查看控制台日志：定位异常来源
4. 最小化复现：删除无关步骤，保留失败最小路径
5. 修复后重跑：确认问题不再出现

## 失败处理

| 问题 | 处理方式 |
| --- | --- |
| mcp_Playwright 未启用 | 提示用户在 Trae CN IDE 设置中启用 mcp_Playwright MCP |
| 本地服务器未启动 | 提示用户提供启动命令或端口，确认后再继续 |
| 测试脚本无法定位元素 | 检查页面是否已渲染完成，改用更稳定的定位方式 |
| 测试用例偶发失败 | 增加显式等待，隔离测试数据，排查并发影响 |
| 后端接口不稳定 | 使用 page.route 进行 mock，待接口稳定后再补充真实请求测试 |
