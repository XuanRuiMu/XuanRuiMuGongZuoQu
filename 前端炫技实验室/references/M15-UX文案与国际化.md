# M15 UX 文案与国际化

> 本模块负责界面中的文字表达与跨语言交付。不讨论视觉 token、布局结构或 ARIA 实现，只解决「 wording 对不对」「不同语言下怎么呈现」的问题。

---

## 1. 何时激活本模块

| 场景 | 说明 |
| --- | --- |
| 用户可见文本需要统一 | 按钮、标签、错误消息、空状态、Toast 等文案出现歧义或不一致 |
| 产品需要支持多语言 | 页面需要中英/多语言切换，或目标用户语言不确定 |
| 需要把文案写成「人话」 | 现有文案充斥技术术语、被动语态、道歉式表达 |
| 交付前文案走查 | 在 M99 交付清单前专门做一轮 wording audit |
| 时间/数字/货币需要本地化 | 日期、价格、百分比的显示方式随地区变化 |

**原则**：文案是界面设计的一部分，不是「随便填几个字」。

### 1.1 制定规范执行流程

当用户要求「制定 UX 文案规范 / i18n 规范」时，按以下顺序执行，避免跳过需求澄清直接起草：

1. **澄清范围**：确认产品类型、目标用户、支持语言、现有文案痛点（错误消息 / 空状态 / 按钮 / 加载 / Toast）。
2. **检查品牌语气**：若涉及品牌语气、视觉语言、字体 token → 先或并行进入 [M02 设计系统](M02-设计系统.md)。
3. **收集样本**：请用户提供 3-5 个当前文案实例（好与坏），或从现有界面截图中提取。
4. **定义术语表**：列出产品中所有核心概念的统一中文/英文说法（登录 / 账户 / 项目等）。
5. **按本模块第 2-5 节起草规范**：文案原则 → 微文案 → 排版细节 → 国际化规则。
6. **走查检查清单**：使用第 7 节快速检查清单复核。
7. **交付规范文件**：输出为 Markdown 规范文档；若需要工程落地（翻译文件结构、i18n 库接入）→ 在规范确认后进入 [M07 组件与框架](M07-组件与框架.md) 或调用通用 skill。

## 2. 文案核心原则

### 2.1 从用户角度出发

用户只关心「我能做什么」「接下来会怎样」。避免系统内部视角。

| 反例 | 正例 |
| --- | --- |
| 提交失败，错误码 5003 | 保存失败，请检查网络后重试 |
| 数据库连接超时 | 数据加载较慢，请稍后再试 |
| 您的请求已被系统拒绝 | 无法完成此操作，原因：余额不足 |

### 2.2 主动语态

主动语态更短、更直接、更容易被屏幕阅读器播报。

| 反例 | 正例 |
| --- | --- |
| 文件已被上传 | 文件已上传 |
| 密码需由用户输入 | 请输入密码 |
| 此操作将被取消 | 将取消此操作 |

### 2.3 按钮用动词命名

按钮描述动作结果，而非功能名称。避免「确定」「好的」这类模糊按钮。

| 反例 | 正例 |
| --- | --- |
| 确定 | 保存更改 |
| 好的 | 我知道了 |
| 取消 | 放弃编辑 |
| 提交 | 发送反馈 |

### 2.4 一致性词汇

同一概念在同一产品内只用一个词。

```
✗ 混合说法
  登录 / 登入 / 登陆
  账户 / 帐号 / 账号
  项目 / 工程 / 计划

✓ 统一说法
  登录、账户、项目
```

### 2.5 错误消息不道歉，给出修复方向

道歉词（抱歉、对不起）增加用户焦虑，且不解决问题。应说明原因和下一步。

| 反例 | 正例 |
| --- | --- |
| 抱歉，出错了 | 无法保存：标题不能为空，请输入标题 |
| 对不起，您没有权限 | 需要管理员权限才能查看此页面 |
| 抱歉，文件太大 | 文件超过 10MB，请压缩后重新上传 |

### 2.6 空状态作为行动邀请

空状态不是「没有数据」，而是引导用户开始下一步。

| 反例 | 正例 |
| --- | --- |
| 暂无数据 | 还没有项目，创建一个新项目开始吧 |
| 列表为空 | 没有收到消息，试试主动联系客户 |
| 404 页面不存在 | 找不到该页面，返回首页或搜索内容 |

## 3. 微文案规范

### 3.1 按钮

- 首字母大写规则需统一（推荐句子式大写：Save changes；不推荐标题式大写：Save Changes）。
- 禁用状态仍要说明原因，或在附近用 helper text 补充。
- 危险操作按钮使用动词 + 对象，让用户确认后果。

```html
<!-- 推荐 -->
<button>创建账户</button>
<button>发送邀请</button>
<button class="danger">删除文件</button>

<!-- 避免 -->
<button>确定</button>
<button>好的</button>
```

### 3.2 标签与占位符

- 标签说明字段含义；占位符给出输入示例，不要重复标签。
- 占位符不要用「请输入」这类无信息前缀。

```html
<!-- 推荐 -->
<label for="email">邮箱</label>
<input id="email" placeholder="name@example.com" />

<!-- 避免 -->
<label for="email">邮箱</label>
<input id="email" placeholder="请输入邮箱" />
```

### 3.3 加载状态

- 说明正在做什么，而不是只显示「加载中」。
- 长任务给出进度或预计时间。

```
✓ 正在保存更改…
✓ 正在生成预览，约需 5 秒
✓ 已上传 3/12 个文件

✗ 加载中
✗ 请稍候
```

### 3.4 Toast / 通知

- 一句话说完结果 + 可执行动作（可选）。
- 成功消息用完成态；错误消息用可修复态。

```
✓ 已保存到草稿箱
✓ 文件已删除，撤销
✓ 发送失败，重试

✗ 操作成功
✗ 发生错误
```

### 3.5 确认对话框

- 标题问具体问题；描述说明后果；按钮给出明确动作。
- 避免用「您确定吗？」这种无信息标题。

```
标题：删除此项目？
描述：删除后，项目内的所有任务和文件都将被移除，无法恢复。
按钮：删除项目 / 取消

避免：
标题：确认
描述：您确定要删除吗？
按钮：确定 / 取消
```

## 4. Typography 细节

### 4.1 省略号

使用真正的省略号 `…`（U+2026），而不是三个句点 `...`。

```
✓ 正在保存…
✗ 正在保存...
```

### 4.2 弯引号

中文内容用弯引号；英文内容按语言习惯使用弯引号或直引号，但需统一。

```
✓ 欢迎来到「前端炫技实验室」
✓ 欢迎来到“前端炫技实验室”
✗ 欢迎来到"前端炫技实验室"
```

### 4.3 不换行空格

防止数字、单位、姓名等被错误断开。使用 `\u00A0` 或 HTML `&nbsp;`。

```
✓ 12&nbsp;GB
✓ 2026&nbsp;年
✓ 张&nbsp;三

✗ 12 GB  （可能被换行断开）
```

### 4.4 数字对齐

价格、数量、日期列使用 `tabular-nums`，保证等宽数字对齐。具体字体与数字样式规则见 [M02 设计系统 - 字体搭配规则](M02-设计系统.md#4-字体搭配规则)。

### 4.5 文本换行

- 标题使用 `text-wrap: balance`，避免最后一行只有一个字。
- 正文使用 `text-wrap: pretty`，提升换行美感。

```css
h1, h2, h3 {
  text-wrap: balance;
}

p, li {
  text-wrap: pretty;
}
```

### 4.6 避免孤字/孤行

中文段落最后一行避免只剩一个字；英文避免短单词单独成行。可通过 `text-wrap` 或手动调整文案解决。

## 5. 国际化（i18n）规则

### 5.1 用 API 格式化日期、数字、货币

不要手动拼接格式；使用 `Intl` 系列 API 或等效库。

```js
// 日期
new Intl.DateTimeFormat('zh-CN', {
  dateStyle: 'medium',
  timeStyle: 'short',
}).format(new Date());
// → 2026年7月6日 14:32

// 数字
new Intl.NumberFormat('zh-CN', {
  notation: 'compact',
}).format(1234567);
// → 123万

// 货币
new Intl.NumberFormat('en-US', {
  style: 'currency',
  currency: 'USD',
}).format(99.9);
// → $99.90
```

### 5.2 品牌名与用户生成内容不翻译

用 `translate="no"` 标记品牌名、代码、用户名、地址等不应被翻译的内容。

```html
<p>
  欢迎使用 <span translate="no">Anthropic</span> 的
  <span translate="no">Claude</span>。
</p>

<pre translate="no"><code>const x = 1;</code></pre>
```

### 5.3 语言检测顺序

1. URL 路径或子域中的语言参数（如 `/zh/`、`/en/`）。
2. 用户已保存的偏好设置。
3. `Accept-Language` 请求头（服务端渲染时）。
4. `navigator.languages` 或 `navigator.language`（客户端兜底）。
5. 默认语言。

```js
// 客户端兜底
const locale =
  localStorage.getItem('locale') ||
  navigator.languages?.[0] ||
  navigator.language ||
  'zh-CN';
```

### 5.4 文案键设计

- 按键名语义化，不要用英文句子当键。
- 支持插值和复数形式。

```json
{
  "saveButton": "保存更改",
  "emptyState": "还没有{entity}，创建一个新的{entity}开始吧",
  "itemCount": {
    "one": "{count} 个项目",
    "other": "{count} 个项目"
  }
}
```

### 5.5 预留文案扩展空间

- 按钮最小宽度应能容纳比中文更长的英文/德文翻译。
- 避免 UI 元素紧贴文案，给翻译留 30% 长度余量。
- 图标 + 文字组合中，图标不应依赖文案语义单独存在。

## 6. 与相邻模块的边界

| 内容 | 归属模块 |
| --- | --- |
| 品牌语气、色板、字体 token、整体视觉语言 | M02 设计系统 |
| 页面结构、布局、hero 叙事、内容层次 | M03 创意实现 |
| ARIA 标签、屏幕阅读器播报、焦点管理 | M13 可访问性修复 |
| 文案措辞、语气、国际化、本地化排版 | **M15 本模块** |

**边界示例**：

- 错误消息文案写什么 → M15；错误消息如何通过 `aria-live` 播报 → M13。
- 按钮文案用动词 → M15；按钮 focus 样式 → M02 / M13。
- 标题是否换行平衡 → M15；标题字号、字重、颜色 → M02。
- 日期格式随语言变化 → M15；日期选择器组件实现 → M07。

## 7. 快速检查清单

- [ ] 所有用户可见文本来自翻译文件或统一文案表，无硬编码。
- [ ] 按钮使用动词，避免「确定」「好的」。
- [ ] 错误消息说明原因和下一步，不道歉。
- [ ] 空状态给出行动邀请，不只是「暂无数据」。
- [ ] 同一概念在全局使用同一词汇。
- [ ] 省略号使用 `…`，引号使用弯引号。
- [ ] 数字列使用 `tabular-nums`；标题/正文使用 `text-wrap: balance / pretty`。
- [ ] 品牌名、代码、用户名标记 `translate="no"`。
- [ ] 日期、数字、货币使用 `Intl.DateTimeFormat` / `Intl.NumberFormat`。
- [ ] 语言检测顺序合理，且 UI 为长文案预留扩展空间。
