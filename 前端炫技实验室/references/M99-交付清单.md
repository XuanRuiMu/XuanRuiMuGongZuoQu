# M99 交付清单

> 用途：在最终交付前执行一次性全面自检，确认所有相关模块的关键要求已被满足。定位：本文件只提供高层级 yes/no 检查项，详细规则见对应模块原文。

---

## 1. 何时激活本模块

在以下任一时机调用 M99：

- 前端页面/组件/应用已完成实现，准备交付前。
- 视觉验证（M11）已执行并修复阻塞性问题后。
- 用户要求“最终检查”“交付前自检”“上线清单”“看看还有没有问题”。

**原则**：M99 是交付前的最后一道闸门，不替代具体模块的实现工作。

**前置条件**：执行 M99 前，应已完成 M11 视觉验证、M12 核心功能测试、M13 可访问性修复、M14 性能优化中的相关项，并保留各自的输出报告/截图/日志路径。M99 只核对结果，不重新执行测试。

---

## 2. 交付自检清单

勾选规则：
- `[x]` 已满足。
- `[ ]` 未满足；未满足项必须升级到对应模块修复，本模块不直接修复。
- `[N/A]` 不适用；需附简短原因，例如「项目未使用 React」「未涉及多语言」。

### 2.1 设计系统（M02）

- [ ] 是否已生成或复用项目级设计系统文件？
- [ ] 是否同时定义了浅色模式与深色模式的完整 token 映射？
- [ ] 颜色是否来自项目专属色板，而非 Tailwind 默认紫色/靛蓝或通用蓝紫渐变？
- [ ] 字体搭配是否至少包含展示字体与正文字体，并提供系统字体回退？
- [ ] 双模式下的正文/UI 对比度是否满足 WCAG 2.1 AA 要求？
- [ ] 是否避免了 Emoji 作为 UI 图标、发光阴影、统一圆角等 AI 同质化元素？

### 2.2 创意实现（M03）

- [ ] 布局是否打破了“居中 hero + 三列卡片 + 底部 CTA”的模板套路？
- [ ] 是否至少应用了 3 项差异化空间处理（不对称、破网格、重叠、出血、杂志式分栏等）？
- [ ] 背景与视觉细节是否与品牌调性一致，且未使用禁用渐变？
- [ ] 主题切换（浅色/深色）是否通过 CSS 变量实现，并已持久化用户偏好？
- [ ] HTML 是否语义化，图片是否有 alt，交互元素是否有可见 focus 状态？
- [ ] 响应式布局是否在 375px、768px、1024px、1440px 四个断点验证通过？
- [ ] 生产代码中是否已清理 console.log、调试样式与未使用资源？

### 2.3 动效与交互（M04）

- [ ] 动画是否服务于引导注意力、确认操作或解释状态变化，而非无意义装饰？
- [ ] 是否优先使用 CSS 动画（含 2026 scroll-driven、anchor positioning 等特性）而非 JS 库？
- [ ] 动画是否主要只涉及 `transform` 与 `opacity`？
- [ ] 微交互（hover/focus/active/pressed/disabled/loading）是否完整且时序合理？
- [ ] 是否已为 `prefers-reduced-motion` 提供全局与局部回退？
- [ ] 关键内容在无动画情况下是否仍然完整可读？
- [ ] 自动播放或循环动画是否在离屏时暂停？

### 2.4 3D / WebGL（M05）

> 仅当项目涉及 Three.js、WebGL、Shader 或粒子效果时检查。

- [ ] 3D 场景是否有明确的性能预算与降级策略？
- [ ] 是否在低性能设备或 `prefers-reduced-motion` 下提供静态回退？
- [ ] 渲染循环是否使用 `requestAnimationFrame` 并在不可见时暂停？
- [ ] 纹理、模型、Shader 资源是否已优化尺寸与加载顺序？
- [ ] WebGL 失败时页面核心内容是否仍可正常访问？

### 2.5 生成艺术（M06）

> 仅当项目涉及程序化视觉、生成图案或算法艺术时检查。

- [ ] 生成逻辑是否有确定性的随机种子或可控参数？
- [ ] 输出是否在不同分辨率与视口下保持视觉稳定？
- [ ] 生成过程是否避免阻塞主线程（如使用 Worker 或分帧）？
- [ ] 最终视觉结果是否仍符合 M02 设计系统的色板与调性？

### 2.6 组件与框架（M07）

> 仅当项目使用 React、Next.js 或 shadcn/ui 时检查。

- [ ] 技术栈选择是否符合“最小够用”原则？
- [ ] 组件拆分是否与状态边界、复用需求相匹配？
- [ ] 是否遵循框架特定的最新稳定模式（如 React 19、Next.js App Router、shadcn/ui 组件约定）？
- [ ] TypeScript 类型是否覆盖主要 props 与 API 返回？
- [ ] 服务端/客户端组件边界是否合理，无不必要的 `use client`？

### 2.7 AI 集成（M08）

> 仅当项目集成 Vercel AI SDK、流式 UI 或模型调用时检查。

- [ ] 是否已处理流式响应的 loading、error、空状态与中断场景？
- [ ] 用户输入是否经过校验与限流，避免滥用？
- [ ] 模型输出渲染是否经过安全处理（如 Markdown/HTML 转义）？
- [ ] 是否向用户明确说明内容由 AI 生成？
- [ ] 失败时是否展示友好的降级界面而非崩溃或暴露内部错误？

### 2.8 视觉验证（M11）

- [ ] 是否已完成桌面端、平板端、移动端三视口截图？
- [ ] 所有 P1 级视觉 BUG（空白页、核心内容不可见、严重错位/重叠）是否已修复？
- [ ] 控制台错误是否已分类处理，影响渲染的错误是否已修复？
- [ ] 截图文件是否已保存到约定目录并可供审查？

### 2.9 应用测试（M12）

> 详细测试规则见 M12。

- [ ] 核心用户流程是否有覆盖的自动化或手工测试用例？
- [ ] 表单提交、状态变化、API 调用是否正常？
- [ ] 错误路径是否返回翻译后的友好提示，而非原始异常信息？
- [ ] 测试是否覆盖边界值、空状态与网络失败场景？
- [ ] 最近一次测试运行是否全部通过？

### 2.10 可访问性（M13）

> 详细 a11y 检查见 M13。

- [ ] 所有图片、图标、控件是否有可访问名称或替代文本？
- [ ] 表单输入是否与 label 正确关联？
- [ ] 颜色是否不是传达状态或错误的唯一方式？
- [ ] 键盘导航顺序是否符合视觉顺序，焦点可见且不外露？
- [ ] 是否通过屏幕阅读器或自动化工具（如 axe）验证关键路径？

### 2.11 性能（M14）

> 详细性能规则见 M14。

- [ ] LCP 是否满足目标阈值？
- [ ] INP 是否满足目标阈值？
- [ ] CLS 是否满足目标阈值？
- [ ] 图片、字体、脚本是否经过压缩与按需加载？
- [ ] 关键渲染路径是否避免阻塞资源？

### 2.12 UX 文案与国际化（M15）

> 仅当项目涉及用户可见文案、按钮 wording、错误信息、空状态、多语言支持时检查。

- [ ] 所有用户可见文本是否来自翻译文件或文案规范，无硬编码？
- [ ] 按钮、错误、空状态文案是否使用主动语态并给出下一步动作？
- [ ] 日期、时间、数字、货币是否使用 `Intl` API 按当前 locale 渲染？
- [ ] 代码、品牌名、人名等不可翻译内容是否设置 `translate="no"`？
- [ ] 本地化排版（省略号、引号、不间断空格）是否符合目标语言习惯？

### 2.13 UX 规则审计（M16）

> 仅当项目已完成 UX 审计或走查时检查。

- [ ] 可访问性、焦点、表单、动画、内容、图片、性能、导航、交互、暗色模式、国际化等维度是否已审计？
- [ ] 审计发现的 P1 项是否已全部修复或明确标注待处理？
- [ ] 修复后是否重新复核相关条目？

### 2.14 React 性能模式（M17）

> 仅当项目使用 React 且涉及并发特性、Suspense、数据获取优化时检查。

- [ ] 是否消除了 async waterfall（串行数据获取）？
- [ ] Suspense 边界是否按组件粒度合理设置？
- [ ] 是否使用 React.cache / useMemo / useCallback 等模式减少重复计算？
- [ ] Server/Client 数据获取边界是否清晰，避免把服务端数据拖到客户端 `useEffect`？

### 2.15 组件组合模式（M18）

> 仅当项目构建可复用组件库或复杂多状态组件时检查。

- [ ] 组件是否避免 boolean prop 爆炸（超过 3 个关联 boolean 时是否改用 Compound Components 或命名变体）？
- [ ] Compound Components 是否通过 Context 共享状态，而不是 `cloneElement`？
- [ ] Context 接口是否拆分为 `state / actions / meta` 三类？
- [ ] 是否在 shadcn/ui 等原语之上做领域封装，而非重新实现基础组件？

## 3. 交付前最终确认

在提交最终产物前，确认以下 5 项全部通过：

1. [ ] 所有适用模块的 yes/no 检查项均已勾选或明确标注“不适用”。
2. [ ] 不存在未修复的 P1 视觉 BUG。
3. [ ] 核心功能测试通过，无阻塞性 defect。
4. [ ] 已清理临时文件、调试代码、重复插件与旧版本资源。
5. [ ] 已告知用户任何已知限制或建议后续优化项。
