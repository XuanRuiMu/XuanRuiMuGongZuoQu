# M13 可访问性修复

## 何时调用本模块

当用户提到「可访问性」「无障碍」「a11y」「WCAG」「ARIA」「键盘导航」「焦点管理」「屏幕阅读器」「读屏」「色盲」「对比度」「焦点陷阱」等关键词，或需要审计并修复现有 UI 的可访问性缺陷时调用本模块。

典型请求示例：

- "帮我审查当前页面的可访问性问题并修复。"
- "审查 `pages/Login.tsx` 的键盘导航与 ARIA 实现，输出 violations 与具体修复。"

本模块只处理可访问性修复，不涉及视觉渲染验证（见 M11）、功能逻辑测试（见 M12）、性能优化（见 M14）。

## 核心原则

1. **优先使用原生 HTML**：能用 `<button>`、`<a>`、`<input>`、`<label>`、`<select>` 等原生元素表达语义时，不添加 ARIA。
2. **翻译文件优先**：`aria-label`、`alt`、表单错误等用户可见文本必须走翻译文件，禁止硬编码。若项目尚未配置 i18n，本次修复可临时使用占位文案并明确标注 `// TODO(i18n)`，后续必须迁移到翻译文件。
3. **外科手术式修复**：只做最小必要修改，不为了 a11y 重写整个 UI。
4. **渐进修复**：先修阻塞性缺陷（无法操作、信息丢失），再修体验类缺陷。
5. **保留现有视觉风格**：修复不强制改变设计，除非设计本身违反对比度等硬性规则。

## 九类规则（按影响优先级排序）

### 1. Accessible Names（可访问名称）

**目标**：所有交互元素和关键非纯装饰元素必须有可被辅助技术读取的名称。

- 图标按钮必须提供 `aria-label` 或通过 `aria-labelledby` 关联可见文本。
- 表单控件优先使用 `<label for="id">`；隐式标签也可以。
- 图片若非纯装饰，必须写 `alt`；装饰性图片用 `alt=""`。
- 链接文本必须能脱离上下文理解目标，避免「点击这里」「更多」。
- 复杂组件（如 tablist、combobox）通过 `aria-label` 或 `aria-labelledby` 说明整体用途。

```html
<!-- 反例 -->
<button><svg>...</svg></button>

<!-- 正例 -->
<button aria-label="关闭对话框"><svg>...</svg></button>
```

### 2. Keyboard Access（键盘访问）

**目标**：所有功能都必须可通过键盘完成，不依赖鼠标、触摸或手势。

- 可点击元素必须是原生可聚焦元素（`<button>`、`<a href>`、`<input>` 等），或显式设置 `tabindex="0"` 并添加键盘事件。
- 禁止把不可聚焦元素（如 `<div>`、`<span>`）伪装成按钮而不处理 `Enter` / `Space`。
- 确保 Tab 顺序符合视觉顺序；不要用正数 `tabindex`。
- 自定义快捷键不要覆盖浏览器/屏幕阅读器默认快捷键。
- 复杂组件（tabs、accordion、menu）必须实现方向键导航。

```html
<!-- 反例 -->
<div class="btn" onclick="submit()">提交</div>

<!-- 正例 -->
<button type="button" onclick="submit()">提交</button>
```

### 3. Focus Management（焦点管理）

**目标**：用户始终知道焦点在哪，焦点不会丢失或被意外困住。

- 所有可聚焦元素必须有可见的焦点指示器（outline、ring、box-shadow 等）。
- 禁止 `outline: none` 而不提供替代焦点样式。
- 打开模态框/对话框时，焦点应移动到对话框内第一个可聚焦元素或标题。
- 关闭模态框时，焦点应返回到触发元素。
- 动态加载内容后，如改变用户上下文，应将焦点移动到相关内容或区域。

```css
/* 反例 */
button:focus {
  outline: none;
}

/* 正例 */
button:focus-visible {
  outline: 2px solid currentColor;
  outline-offset: 2px;
}
```

### 4. Semantics（语义）

**目标**：DOM 结构传达正确的信息层级和区域关系。

- 优先使用语义化标签：`<header>`、`<nav>`、`<main>`、`<section>`、`<article>`、`<aside>`、`<footer>`。
- 页面有且仅有一个 `<main>`。
- 标题层级连续（h1 → h2 → h3），不要跳级，不要用样式冒充标题。
- 列表、表格、表单使用对应原生结构，不要全部用 `<div>` 模拟。
- landmark 区域通过语义标签自然形成；必要时用 `role` 补充，但避免冗余。

```html
<!-- 反例 -->
<div class="header">...</div>
<div class="title">文章标题</div>
<div class="list"><div class="item">...</div></div>

<!-- 正例 -->
<header>...</header>
<h1>文章标题</h1>
<ul>
  <li>...</li>
</ul>
```

### 5. Forms and Errors（表单与错误）

**目标**：表单控件可理解、可预测，错误信息可感知、可纠正。

- 每个输入框都必须有关联标签。
- 必填项通过 `required` 属性和视觉标记共同说明。
- 错误信息通过 `aria-describedby` / `aria-errormessage` 与对应控件关联。
- 提交表单时，将焦点移动到第一个错误字段或汇总错误区域。
- 字段组（如单选组）使用 `<fieldset>` + `<legend>`。
- 输入约束（格式、长度）在标签或提示中预先说明，不要等到报错才告知。

```html
<label for="email">邮箱 <span aria-label="必填">*</span></label>
<input id="email" type="email" required aria-invalid="true" aria-describedby="email-error" />
<span id="email-error">请输入有效的邮箱地址</span>
```

### 6. Announcements（实时通知）

**目标**：关键状态变化必须被屏幕阅读器及时播报，而不依赖视觉。

- 使用 `aria-live` 区域播报异步结果（保存成功、提交失败、加载完成）。
- `aria-live="polite"` 用于非紧急通知；`aria-live="assertive"` 仅用于需要立即打断用户的错误。
- 避免过度播报，不要对每次输入变化都触发 live region。
- 加载状态通过 `aria-busy="true"` 和 `aria-live` 组合告知。
- Toast、Snackbar 等通知组件应自动获得焦点或使用 live region 播报。

```html
<div aria-live="polite" aria-atomic="true" class="sr-only">保存成功</div>
```

### 7. Contrast（对比度）

**目标**：文本和关键图形与背景之间有足够的亮度对比。

- 正文文本与背景对比度至少 4.5:1（WCAG AA）。
- 大文本（18pt 或 14pt 粗体以上）至少 3:1。
- UI 组件边界、图标、焦点指示器等关键图形至少 3:1。
- 不要仅依赖颜色传达状态；必须配合图标、文本标签或形状变化。
- 无法通过代码精确修复时，标记为设计侧问题并提供具体色值建议。

### 8. Media（媒体）

**目标**：非文本内容对所有用户都可感知。

- 视频必须提供字幕（captions）、音频描述或文字稿。
- 音频必须提供文字稿。
- 自动播放的音视频必须有暂停/停止控件，且默认音量可控。
- 图片替换元素必须提供 `alt` 或等效文本。
- 动画内容遵守 `prefers-reduced-motion`，提供关闭或减少动效的选项。

```css
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```

### 9. Tool Boundaries（工具边界）

**目标**：辅助技术不会被错误信息误导，能够理解组件的范围和行为。

- 自定义组件使用正确 `role`，但不要滥用 `role` 覆盖原生语义。
- 状态属性保持同步：`aria-expanded`、`aria-selected`、`aria-checked`、`aria-pressed`、`aria-disabled`。
- 隐藏内容正确区分：完全隐藏用 `display:none` / `visibility:hidden` / `hidden`；仅对辅助技术隐藏用
  `aria-hidden="true"`；视觉隐藏但可访问用 sr-only 样式。
- 弹层、下拉、tooltip 等浮层使用 `aria-haspopup`、`aria-controls` 等属性说明关系。
- 避免在可聚焦元素上设置 `aria-hidden="true"`。

## 常见修复模式

| 缺陷                 | 修复方式                                                                              |
| -------------------- | ------------------------------------------------------------------------------------- |
| 图标按钮无名称       | 添加 `aria-label` 或通过 `aria-labelledby` 引用相邻文本                               |
| `<div>` 冒充按钮     | 改为 `<button>`；若必须保留 div，补 `role="button"`、`tabindex="0"`、Enter/Space 处理 |
| 表单错误未关联       | 错误信息加 `id`，控件加 `aria-describedby` 指向该 id                                  |
| 焦点丢失             | 动态内容加载后将焦点移到相关区域或第一个可聚焦元素                                    |
| 模态框背景仍可操作   | 实现焦点陷阱，并设置 `aria-modal="true"`、背景 `aria-hidden="true"`                   |
| 无焦点样式           | 添加 `:focus-visible` 样式，避免直接去掉 outline                                      |
| 缺少页面区域         | 用 `<header>`、`<nav>`、`<main>`、`<footer>` 替换无意义 div                           |
| 标题层级断裂         | 按 h1-h6 重新映射，不依赖字体大小决定层级                                             |
| 颜色作为唯一状态标识 | 补充图标或文本，如「成功 ✓」「错误 ✗」                                                |
| 动态结果未播报       | 添加 `aria-live="polite"` 区域并在状态变化时更新其文本                                |

## 焦点陷阱与模态框处理

模态框、对话框、侧边抽屉、全屏覆盖层必须实现焦点管理。

### 必需行为

1. 打开时：焦点移动到对话框标题（`tabindex="-1"`）或第一个可聚焦元素。
2. 关闭时：焦点返回到触发该对话框的元素。
3. 焦点循环：当 Tab 到达最后一个可聚焦元素时，下一次 Tab 回到第一个；Shift+Tab 反之。
4. 背景隔离：对话框打开期间，背景内容应 `aria-hidden="true"` 或从 Tab 序列中移除。
5. Esc 关闭：除非业务禁止，否则按 Esc 应关闭对话框。

### 最小实现示例

```html
<button id="open-btn" aria-haspopup="dialog">打开设置</button>

<div role="dialog" aria-modal="true" aria-labelledby="dialog-title" aria-describedby="dialog-desc">
  <h2 id="dialog-title" tabindex="-1">设置</h2>
  <p id="dialog-desc">在此处调整您的偏好。</p>
  <button>保存</button>
  <button id="close-btn">取消</button>
</div>
```

### 焦点陷阱实现要点

- 记录打开前的 `document.activeElement`。
- 监听 `keydown`：Tab / Shift+Tab 时计算容器内可聚焦元素并循环。
- 监听 `keydown`：Esc 时关闭并恢复焦点。
- 使用现有原语（如 Radix Dialog、React Aria Modal、Headless UI Dialog）时，不要重复造轮子，但要验证默认行为是否被覆盖。

## ARIA 最佳实践

### 第一规则：能用 HTML 就不用 ARIA

ARIA 不能赋予元素行为，只能赋予语义。很多 ARIA 模式需要开发者自己实现键盘事件和焦点管理，风险远高于原生元素。

| 需要的行为  | 推荐做法                                           |
| ----------- | -------------------------------------------------- |
| 可点击      | `<button>` / `<a href>`                            |
| 可展开/折叠 | `<details>` / `<summary>`                          |
| 表单输入    | `<input>`、`<textarea>`、`<select>`                |
| 选择框      | `<input type="checkbox">` / `<input type="radio">` |
| 列表选择    | `<select>` 或组合原语                              |

### 第二规则：状态必须实时同步

使用框架时，确保 `aria-*` 属性与组件状态绑定：

- `aria-expanded` ↔ 展开状态
- `aria-selected` ↔ 选中项
- `aria-checked` ↔ 勾选状态
- `aria-pressed` ↔ 切换按钮状态
- `aria-disabled` ↔ 禁用状态

### 第三规则：避免常见反模式

- 不要给 `<button>` 加 `role="button"`。
- 不要给链接加 `role="button"`；如果需要按钮行为，就用 `<button>`。
- 不要在 `<div>` 上滥用 `role="textbox"`，直接用 `<input>` 或 `<textarea>`。
- 不要同时用 `aria-hidden="true"` 和可聚焦子元素。
- 不要把 `aria-label` 放在无角色或不可聚焦元素上期望它被读出。

## 审查与修复流程

1. **明确范围**：询问用户要审查的文件、页面或组件路径。
2. **自动化扫描**（推荐）：先使用 `eslint-plugin-jsx-a11y` 或 `axe-core`（`@axe-core/react` / `axe-playwright`）跑一遍，获取基础 violations 作为人工审查起点。
3. **静态扫描**：检查 HTML 语义、标签层级、交互元素、ARIA 使用、对比度可疑点。
4. **键盘走查**：用 Tab、Shift+Tab、Enter、Space、Esc 测试所有交互。
5. **输出问题清单**：按九类规则分类，每项包含 violation、影响、具体修复位置。
6. **优先修复**：先处理阻塞性缺陷（无法键盘操作、信息丢失、焦点陷阱）。
7. **回归检查**：修复后重新走查键盘路径，并重新运行自动化扫描，确认没有引入新的焦点问题或语义错误。
8. **交付摘要**：列出修改点、未修复项（如需要设计配合的对比度问题）、置信度。

## 边界说明

- 当用户请求同时包含「审计/检查」和「修复」意图时：先由 **M16 UX 规则审计** 输出问题清单，再进入本模块执行 a11y 专项整改与键盘走查回归。若用户直接说「修复可访问性」「改 a11y」则跳过 M16，直接进入本模块。
- 视觉渲染验证由 **M11 视觉验证** 覆盖。
- 功能正确性测试由 **M12 应用测试** 覆盖。
- 性能优化（包括动画性能、渲染性能）由 **M14 性能优化** 覆盖。
- 本模块不替代真实屏幕阅读器测试，只能保证代码层面符合 WCAG 最佳实践。
