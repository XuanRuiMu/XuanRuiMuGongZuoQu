# M18 组件组合模式

> 定位：复杂可复用组件的 API 设计指南。覆盖：何时使用组合模式、避免 boolean prop 爆炸、Compound Components、Generic Context Interface、Explicit Variants、Children over Render Props、常见反模式。边界：不重复 React/Next.js/shadcn 基础架构（M07）、不重复设计 token（M02）、不重复性能优化（M14）。

---

## 1. 何时调用本模块

在以下场景进入 M18：

- 一个组件出现多个 boolean props 控制显隐/行为/样式，状态组合指数级增长。
- 需要为同一底层状态设计多种可互换的 UI 形态。
- 正在构建可复用组件库，或把局部组件抽象为项目级公共 API。
- 需要把复杂配置型组件拆成更易预测、更可读的小块。
- 与 shadcn/ui 等现有库衔接时，需要在其组件之上再包一层领域语义。

若任务只是简单页面或单次使用组件，优先使用 M07 的常规 props 模式；若涉及动画或 3D，分别转交 M04/M05。

---

## 2. 核心原则：避免 boolean prop 爆炸

每个 boolean prop 都会使组件潜在状态翻倍。3 个 boolean 就有 8 种组合，多数组合没有设计意义，只会带来无法测试的隐藏状态。

```tsx
// 反例：boolean prop 爆炸
interface ModalProps {
  isOpen: boolean
  isAlert: boolean
  isFullScreen: boolean
  showCloseButton: boolean
  closeOnOverlayClick: boolean
  // 2^5 = 32 种组合
}
```

优先用显式组合或命名变体替代 boolean flags。目标是让调用点自描述、让非法组合在类型层面不可达。

## 3. Pattern 1：Compound Components（复合组件）

把一个大组件拆成语义化子组件，由内部 Context 共享状态。调用方像搭积木一样组合，而不是阅读一组长参数。

### 3.1 触发条件

- 组件有多个相互依赖的子部分（触发器、面板、标题、关闭按钮）。
- 子部分需要访问同一内部状态，但又不希望把状态泄露给外层。
- 需要支持可选子部分、自定义顺序、重复子项。

### 3.2 Context 共享状态

```tsx
import {
  createContext,
  useContext,
  useId,
  useState,
  useCallback,
  ReactNode,
} from 'react'

interface DialogContextValue {
  open: boolean
  setOpen: (open: boolean) => void
  titleId: string
}

const DialogContext = createContext<DialogContextValue | null>(null)

function useDialog() {
  const ctx = useContext(DialogContext)
  if (!ctx) throw new Error('Dialog 子组件必须在 Dialog.Root 内使用')
  return ctx
}
```

### 3.3 Provider + 子组件示例

```tsx
function Root({ children }: { children: ReactNode }) {
  const [open, setOpen] = useState(false)
  const titleId = `dialog-title-${useId()}`

  return (
    <DialogContext.Provider value={{ open, setOpen, titleId }}>
      {children}
    </DialogContext.Provider>
  )
}

function Trigger({ children }: { children: ReactNode }) {
  const { setOpen } = useDialog()
  return <button onClick={() => setOpen(true)}>{children}</button>
}

function Content({ children }: { children: ReactNode }) {
  const { open, titleId } = useDialog()
  if (!open) return null
  return (
    <div role="dialog" aria-modal="true" aria-labelledby={titleId}>
      {children}
    </div>
  )
}

function Title({ children }: { children: ReactNode }) {
  const { titleId } = useDialog()
  return <h2 id={titleId}>{children}</h2>
}

function Close({ children }: { children: ReactNode }) {
  const { setOpen } = useDialog()
  return <button onClick={() => setOpen(false)}>{children}</button>
}
```

> **可访问性边界**：本示例仅展示组合模式 API。焦点进入/返回触发元素、焦点陷阱、ESC 关闭等完整可访问性行为见 [M13 可访问性修复](M13-可访问性修复.md)。

### 3.4 命名空间导出

通过对象挂载子组件，使 API 与 Radix、shadcn/ui 风格一致。

```tsx
export const Dialog = {
  Root,
  Trigger,
  Content,
  Title,
  Close,
}

// 使用方
<Dialog.Root>
  <Dialog.Trigger>打开</Dialog.Trigger>
  <Dialog.Content>
    <Dialog.Title>确认删除</Dialog.Title>
    <p>此操作不可撤销。</p>
    <Dialog.Close>取消</Dialog.Close>
  </Dialog.Content>
</Dialog.Root>
```

### 3.5 与 shadcn/ui 的衔接

shadcn/ui 组件本身已具备结构（DialogTrigger、DialogContent 等）。在其之上再包一层领域语义时，优先复用 shadcn 原语，只注入业务默认值或约束，不要复制实现。

```tsx
import {
  Dialog,
  DialogTrigger,
  DialogContent,
  DialogTitle,
  DialogClose,
} from '@/components/ui/dialog'

export const ConfirmDialog = {
  Root: Dialog,
  Trigger: DialogTrigger,
  Title: DialogTitle,
  Close: DialogClose,
  // 对 Content 做业务定制
  Content: function ConfirmContent({ children }: { children: ReactNode }) {
    return (
      <DialogContent className="sm:max-w-md">
        {children}
      </DialogContent>
    )
  },
}
```

### Modal 专项示例

把 3.2 的 Dialog 示例映射到常见 Modal 场景。先补充 `Description` 子组件：

```tsx
function Description({ children }: { children: ReactNode }) {
  return <p className="text-sm text-muted-foreground">{children}</p>
}

export const Modal = {
  Root,
  Trigger,
  Content,
  Title,
  Description,
  Close,
}

// 使用方
<Modal.Root>
  <Modal.Trigger>删除账户</Modal.Trigger>
  <Modal.Content>
    <Modal.Title>确认删除账户？</Modal.Title>
    <Modal.Description>此操作不可撤销，所有数据将被清除。</Modal.Description>
    <Modal.Close>取消</Modal.Close>
  </Modal.Content>
</Modal.Root>
```

## 4. Pattern 2：Generic Context Interface

当同一套 UI 需要适配多种数据源或交互行为时，用 Context 约定 `state / actions / meta` 三类接口，而不是为每种场景复制组件。

### 4.1 接口约定

```tsx
interface DataTableContext<T> {
  state: {
    rows: T[]
    selectedIds: Set<string>
    sort: { key: keyof T; direction: 'asc' | 'desc' } | null
  }
  actions: {
    toggleSelect: (id: string) => void
    setSort: (key: keyof T) => void
  }
  meta: {
    rowKey: keyof T & string
    columns: ColumnDef<T>[]
  }
}

const DataTableContext = createContext<DataTableContext<unknown> | null>(null)
```

### 4.2 同一 UI 不同 provider 实现

```tsx
function LocalDataTableProvider<T>({
  rows,
  columns,
  rowKey,
  children,
}: LocalProviderProps<T>) {
  const [selectedIds, setSelectedIds] = useState<Set<string>>(new Set())
  const [sort, setSortState] = useState<DataTableContext<T>['state']['sort']>(null)

  const toggleSelect = useCallback((id: string) => {
    setSelectedIds(prev => {
      const next = new Set(prev)
      if (next.has(id)) next.delete(id)
      else next.add(id)
      return next
    })
  }, [])

  const setSort = useCallback((key: keyof T) => {
    setSortState(prev =>
      prev?.key === key
        ? { key, direction: prev.direction === 'asc' ? 'desc' : 'asc' }
        : { key, direction: 'asc' }
    )
  }, [])

  const value: DataTableContext<T> = {
    state: { rows, selectedIds, sort },
    actions: { toggleSelect, setSort },
    meta: { rowKey, columns },
  }

  return (
    <DataTableContext.Provider value={value as unknown as DataTableContext<unknown>}>
      {children}
    </DataTableContext.Provider>
  )
}

// 服务端/URL 同步版本可替换 provider，UI 组件不变
function ServerDataTableProvider({ children }: { children: ReactNode }) {
  // 从 search params 或 Server Actions 驱动 state/actions
  return <DataTableContext.Provider value={serverValue}>{children}</DataTableContext.Provider>
}
```

## 5. Pattern 3：Explicit Variants（命名变体）

用命名组件替代 boolean modes，让类型系统阻止无意义组合。

```tsx
// 反例：一个 Composer 组件用 isEditing 区分两种模式
<Composer isEditing initialValue={value} />

// 正例：命名变体
<ThreadComposer onSubmit={createPost} />
<EditComposer initialValue={value} onSubmit={updatePost} />
```

实现上仍可共享内部 hooks：

```tsx
function useComposerCore(initialValue = '') {
  const [value, setValue] = useState(initialValue)
  const [error, setError] = useState<string | null>(null)
  return { value, setValue, error, setError }
}

export function ThreadComposer({ onSubmit }: { onSubmit: (text: string) => void }) {
  const core = useComposerCore()
  return <ComposerForm {...core} onSubmit={() => onSubmit(core.value)} />
}

export function EditComposer({
  initialValue,
  onSubmit,
}: {
  initialValue: string
  onSubmit: (text: string) => void
}) {
  const core = useComposerCore(initialValue)
  return <ComposerForm {...core} onSubmit={() => onSubmit(core.value)} />
}
```

## 6. Pattern 4：Children over Render Props

优先用 `children` 组合，而不是把子元素逻辑外包成 render prop。children 更自然、更易读、与 TypeScript/JSX 生态更一致。

```tsx
// 反例：render prop 滥用
<DataTable renderRow={row => <Row row={row} />} />

// 正例：children 组合
<DataTable>
  {rows.map(row => (
    <Row key={row.id} row={row} />
  ))}
</DataTable>
```

仅当子元素需要注入从父组件计算出的隐式数据，且该数据无法通过 Context 简洁表达时，才考虑 render props。

## 7. 反模式

| 反模式 | 表现 | 后果 |
| --- | --- | --- |
| Prop soup | 一个组件接受 10+ 个 props，包含 boolean、style、回调 | 调用点难以阅读，组合爆炸 |
| Render props 滥用 | 所有子元素都通过 render prop 注入 | JSX 嵌套深、类型复杂 |
| 深层嵌套 provider | 组件被 5 层以上 Context Provider 包裹 | 调试困难，渲染边界不清晰 |
| 隐式依赖顺序 | Compound Components 要求子组件必须按固定顺序出现 | 限制组合自由，易出错 |
| 把内部状态当 API | 暴露 open/setOpen 等内部 setter | 调用方承担本应由组件封装的状态复杂度 |

## 9. 快速检查清单

- [ ] 组件是否超过 3 个相互关联的 boolean props？若是，考虑 Compound Components 或命名变体。
- [ ] 是否可以用命名组件（`EditComposer`）替代 `mode="edit"` 或 `isEditing`？
- [ ] Compound Components 内部是否通过 Context 共享状态，而不是通过 cloneElement？
- [ ] Context 接口是否拆分为 `state / actions / meta` 三类？
- [ ] 子组件是否使用命名空间导出（`Dialog.Root` / `Dialog.Trigger`）？
- [ ] 是否优先使用 `children` 而不是 render props？
- [ ] 是否避免深层嵌套 Provider？多个独立 Context 考虑合并或按使用位置局部化。
- [ ] 是否在 shadcn/ui 之上做领域封装，而不是重新实现 Dialog/Tabs 等基础组件？
