## 执行规则

1. **每轮只激活一个模块** 一次用户请求只能进入 `references/`
   中的一个模块。若用户请求跨越多个阶段，先执行当前最核心模块，完成后询问用户是否切换到下一模块。

2. **零冗余** 本 SKILL.md 只保留入口与调度规则，不重复 `references/`
   中的实现细节，也不重复通用 skill（循环工程、软件测试、三轴审查、代码需求实现器、Bug 修复）中的通用流程。

3. **通用 skill 按需调用**
   当任务需要大范围开发、测试、代码审查或 Bug 修复时，可调用通用 skill 协同工作，但本 skill 只负责前端展示/体验相关的专业模块调度，不复制通用 skill 内容。

4. **先选模块，再看参考** 不要一次性加载多个 references/ 文件。先根据调度表确定唯一模块，再读取对应参考文档执行。

5. **高端技术必选**

   当用户在**技术/效果/实现手段**层面表达「尽可能高端 / 尽可能华丽 / 炫技 / 用最先进技术 / 当前最强 / 最炫 / 最顶 / 最顶级体验 / 下一代 / 前沿 / 黑科技 / 惊艳 / 最流畅 / cutting-edge / state-of-the-art / wow factor」等意图时，不要直接回退到默认方案。必须先通过 WebSearch 检索 2026 年最新、最强的相关前端技术（库、CSS 特性、浏览器能力、框架模式、AI 工具链等），整理 3-5 个候选方案，与用户讨论后再决定如何应用到当前项目。

   **判断核心**：「高端/华丽」修饰的是**技术手段或视觉效果的复杂度**，而非品牌定位或风格调性。

   **触发示例**：「高端微交互」「华丽动效」「用最先进的技术实现」「当前最强的 3D 方案」「给我最顶级的交互体验」。
   **不触发示例**：「高端奢侈品网站」「高端大气风格」「高端品牌调性」——这些属于品牌定位或视觉风格描述，进入 M02 设计系统即可。

   **执行顺序**：
   1. 先根据调度表锁定目标模块（如 M04/M05/M07 等）。
   2. 读取对应 reference 文档，建立技术约束（浏览器基线、性能预算、可访问性要求）。
   3. 使用 WebSearch 检索 2026 年相关技术，优先查找官方文档、发布说明、知名团队案例。
   4. 整理 3-5 个候选方案，每个方案必须包含：
      - 技术名称与一句话定位
      - 适用场景
      - 浏览器/环境支持
      - 性能特点
      - 维护风险与学习成本
      - 与本项目需求的契合度
   5. 以对比表形式呈现给用户，等待用户确认或进一步提问。
   6. 用户确认选型后，再回到已锁定的模块执行实现。

   **搜索回退**：若 2026 年资料不足，允许使用 2025 年已成熟并在 2026 年仍为主流的技术，但必须在回复中说明信息时效性。

   **风格 + 实现混合请求**：若请求同时包含风格描述（高端/奢侈品）和实现动作（做落地页/做官网），优先进入 M02 设计系统确立视觉语言，完成后再询问是否进入 M03/M07 实现。

## 模型使用指南

- 收到前端相关请求时，首先识别核心意图属于上表哪一列。
- 向用户简要说明将启用哪个模块，然后读取对应的 `references/Mxx-xxx.md`。
- 若用户同时提到多个模块内容，明确告知将分阶段执行，并询问是否从当前选定的模块开始。
- 执行过程中如需测试、审查或大型工程循环，调用通用 skill，而不是把通用流程写进本 skill 的回复。
