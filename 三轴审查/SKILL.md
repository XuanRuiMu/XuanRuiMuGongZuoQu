# 三轴审查

对 `HEAD` 与用户指定的固定基准点之间的差异进行三轴审查：

- **Standards（规范轴）** — 代码是否符合本仓库文档化的编码规范？
- **Spec（规格轴）** — 代码是否忠实实现了来源需求/PRD/规格？
- **BlindSpot（盲区轴）** — 变更是否存在逻辑漏洞、遗漏维度、过度推断或交互风险？

三条轴作为**并行子代理**运行，互不污染上下文，然后由本技能聚合各自发现。

## 流程

### 1. 固定基准点

用户所说的即为固定基准点——提交SHA、分支名、标签、`main`、`HEAD~5`
等。不要自作主张，直接采用。如果用户未指定，询问："审查对比什么——分支、提交，还是 `main`？"未获得答案前不要继续。

一次性捕获差异命令：`git diff <fixed-point>...HEAD`（三点语法，对比合并基准）。同时记录提交列表：`git log <fixed-point>..HEAD --oneline`。

### 2. 识别规格来源

按以下顺序查找来源规格：

1. 提交信息中的需求引用（`#123`、`Closes #45`、GitLab `!67` 等）——通过 `docs/agents/issue-tracker.md` 中的工作流获取。
2. 用户作为参数传入的路径。
3. `docs/`、`specs/` 或 `.scratch/` 下与分支名或功能匹配的PRD/规格文件。
4. 如果什么都找不到，问用户规格在哪里。如果用户说没有规格，**Spec** 子代理将跳过并报告"无可用规格"。

### 3. 识别标准来源

仓库中任何记录代码应如何编写的文档。常见位置：

- `CLAUDE.md`、`AGENTS.md`
- `CONTRIBUTING.md`
- `CONTEXT.md`、`CONTEXT-MAP.md`、各上下文的 `CONTEXT.md` 文件
- `docs/adr/`（架构决策即标准）
- `.editorconfig`、`eslint.config.*`、`biome.json`、`prettier.config.*`、`tsconfig.json`（机器强制标准——记录它们但不重复检查工具已覆盖的内容）
- 仓库根目录或 `docs/` 下的任何 `STYLE.md`、`STANDARDS.md`、`STYLEGUIDE.md` 或类似文件

收集文件列表。**Standards** 子代理将读取它们。

### 4. 并行启动三个子代理

发送一条包含三个 `Agent` 工具调用的消息。三个都使用 `general-purpose` 子代理。

**Standards 子代理提示词**——包含：

- 完整的差异命令和提交列表。
- 步骤3中找到的标准来源文件列表。
- 任务简述："先读 AGENTS.md。阅读标准文档。然后阅读差异。按文件/代码块报告差异中每一处违反文档化标准的地方。引用标准（文件+规则）。区分硬性违规与判断性建议。跳过工具已强制执行的内容。400字以内。"

**Spec 子代理提示词**——包含：

- 差异命令和提交列表。
- 规格的路径或已获取的内容。
- 任务简述："先读 AGENTS.md。阅读规格。然后阅读差异。报告：(a) 规格要求但缺失或不完整的需求；(b) 差异中未被要求的行为（范围蔓延）；(c) 看似已实现但实现看起来有误的需求。每条发现引用规格原文。400字以内。"

**BlindSpot 子代理提示词**——包含：

- 差异命令和提交列表。
- 步骤3中找到的标准来源文件列表（供其了解项目约定，但审查重点不是规范符合性）。
- 规格的路径或已获取的内容（供其了解意图，但审查重点不是规格符合性）。
- 任务简述："先读 AGENTS.md。你是独立审计者，批判性审视这次变更，找出盲区、逻辑漏洞或过度推断。遵循 AGENTS.md'批判而非全盘否定'原则。审查重点：(1) 逻辑漏洞——推理链是否有跳跃、遗漏或过度推断；(2) 遗漏维度——变更可能遗漏了哪些重要维度（并发安全、异常处理、边界条件、向后兼容、性能影响）；(3) 过度推断——变更的某些假设是否站不住脚；(4) 交互风险——与现有系统的兼容性风险。阅读差异及相关上下文文件。按发现点报告，每条说明薄弱环节所在及为何是盲区。400字以内。"

如果规格缺失，跳过 Spec 子代理并在最终报告中注明。BlindSpot 子代理始终运行——它不依赖规格存在。

### 5. 聚合报告

在 `## Standards`、`## Spec` 和 `## BlindSpot`
标题下分别呈现三份报告，原文照搬或轻度清理。**不要**合并或重新排序发现——三条轴刻意分离，以便用户独立查看，防止一条轴掩盖另一条。

末尾附一行摘要：每条轴的发现总数，以及最严重的单个问题（如有）。

## 为什么是三条轴

一次变更可能通过一条轴但未通过另一条：

- 代码遵循了所有规范但实现了错误的东西 → **Standards 通过，Spec 未通过。**
- 代码完全实现了需求但违反了项目约定 → **Spec 通过，Standards 未通过。**
- 代码既符合规范又实现了需求，但存在逻辑漏洞、遗漏了并发安全/边界条件等重要维度，或基于站不住脚的假设 →
  **Standards 通过，Spec 通过，BlindSpot 未通过。**

第三轴 BlindSpot 之所以必要：Standards 检查"写得对不对"，Spec 检查"做的是不是想要的"，但两者都不审查"开发者没想到的"。一条变更可能规范齐全、需求对齐，却在并发安全、异常路径、向后兼容或与现有系统的交互上存在盲区。BlindSpot 作为独立审计者，专门找出这些前两轴覆盖不到的薄弱环节。

分开报告防止一条轴掩盖另一条。
