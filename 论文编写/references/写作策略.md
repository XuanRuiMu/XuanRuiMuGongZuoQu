# 写作策略与风格指南

本文件融合了Nature级写作规范与中文毕业论文的特殊要求。在需要章节级写作策略、内容来源规则或反AI检测指导时加载。

## 内容来源优先级

所有论文内容必须追溯到授权来源，按以下优先级：

1. **源代码**（最高权威）— 类名、方法签名、数据结构、算法
2. **需求文档**— 功能描述、系统架构、数据流
3. **已有论文文本**— 之前写好并审核通过的部分
4. **领域知识**— 标准术语、公认模式（仅在前三者沉默时使用）

如果某项声明无法追溯到以上任何来源，不要写它。

## 参考资料分类

用户提供参考资料时，写入前必须先分类：

| 类型            | 示例           | 怎么用                       | 怎么不用                     |
| --------------- | -------------- | ---------------------------- | ---------------------------- |
| 格式模版        | .doc/.docx     | 复制结构、字体、页边距、样式 | 不要照抄示例文字             |
| 需求文档        | 需求.md        | 提取功能描述、系统设计       | 不要逐字作为正文             |
| 源代码          | Java/Python    | 描述架构、算法、数据模型     | 不要粘贴代码列表（除非附录） |
| 设计文档        | 架构图、流程图 | 参考技术描述                 | 不要假设读者已看过           |
| 任务书/开题报告 | .doc           | 对齐标题、摘要方向           | 范围可能与最终论文不同       |

## 论文类型与叙事逻辑

不同类型论文使用不同的叙事逻辑：

| 类型        | 读者核心问                                   | 叙事路径                 |
| ----------- | -------------------------------------------- | ------------------------ |
| 研究型      | 为什么重要？做了什么？发现什么？意味着什么？ | 现象→缺口→方法→发现→意义 |
| 系统实现型  | 能不能用？能不能复用？是否更好？             | 需求→设计→实现→测试→结论 |
| 算法/工具型 | 可靠吗？有优势吗？                           | 问题→方案→验证→对比→应用 |

## 技术深度规则

描述到同专业同行能够理解并复现的程度：

- 清楚展示"是什么"和"为什么"
- 展示"怎么做"并给出足够细节
- 省略琐碎样板代码（getter/setter、import语句）
- 包含非平凡的算法逻辑或设计决策

## 反AI写作指南（关键！）

中国高校使用AIGC检测工具（知网AIGC 3.0/4.0、Turnitin、PaperPass）。这些工具检测：

### AI写作的特征指纹

- **句长过于均匀** — 连续句子长度和节奏可预测
- **模板化过渡词** — "首先...其次...最后..."、"一方面...另一方面..."、"综上所述"
- **过度因果连接词** — 连续使用"因此"、"从而"、"进而"
- **零情感中立** — 完全没有个人立场、怀疑或批判声音
- **完美无瑕** — 没有限定语、没有自我修正痕迹
- **抽象概念堆砌** — 段落充满抽象术语但没有具体实例支撑
- **风格完全一致** — 全文一个调子，没有自然波动

### 对策：注入人类写作特征

**认知痕迹** — 在主要段落中包含推理挣扎和决策权衡：

> "在设计属性刷新机制时，曾考虑过事件驱动与轮询两种方案。前者响应及时但调用链复杂，后者实现简单却存在延迟。最终选择事件驱动并辅以防抖策略。"

**句式变化** — 刻意混用长短句，变换从句结构：

- 偶尔用设问句或括号注释
- 有些段落密集，有些段落更松弛
- 过渡可以偶尔"生硬"一点（人类写作就是这样的）

**连接词多样性** — 替换模板化表达：

- "首先/其次/最后" → "其一/再者/更关键的是" 或 直接重构去掉枚举
- "不难发现" → "梳理下来可以看到"
- "综上所述" → 用具体总结替代

**适当限定语**：

- "初步分析表明"、"从现有数据来看"、"在本研究的测试条件下"

**具体与抽象交替** — 陈述抽象原则后立即用具体例子锚定：

> "分层属性模型将属性分为基础层、装备层、天赋层和最终层。以攻击力为例：基础值来自职业模板(如战士150)，装备层累加武器和护甲的加成，天赋层叠加被动技能效果，最终层应用修饰器buff。"

**风格自然波动** — 方法论段更正式，实现段更直接，讨论段更反思性。

**偶见口语痕迹**（在解释性段落中，非正式定义处）：

- "说白了"、"其实"、"简单来讲"
- 不是用在形式化定义中，而是用在帮助读者理解的过渡处

### AIGC自检清单（每写完一段必查）

- [ ] 扫描是否有3个以上连续平行结构句子 — 至少打破一个
- [ ] 检查连接词密度 — 每段不超过2个因果连接词
- [ ] 确认每节都有认知痕迹 — 推理挣扎、选择困境、权衡取舍
- [ ] 确认句长变化超过AI典型阈值
- [ ] 确保没有残留的模板开头/结尾
- [ ] 检查抽象-具体交织 — 不存在纯抽象段落
- [ ] 确认各节风格有差异 — 不是单调统一

## 证据强度分级

选择与证据匹配的动词：

### 强（仅在设计/数据充分支持时使用）

- 表明、证明、确立、揭示、识别

### 中等（解读合理但非决定性时使用）

- 表明、指出、支持...的观点、与...一致、指向

### 推测性（超出直接观察时使用）

- 可能反映、可能源于、似乎是、很可能由于、或许可用...来解释

## 过渡词族

### 对比

- 然而、相比之下、尽管如此、尽管这样、而

### 递进

- 此外、另外、而且、也

### 因果

- 因此、于是、结果是、从而

### 限定

- 在一定程度上、在某种意义上、至少、目前来说

## 大纲构建流程

1. 阅读模版的目录/章节结构
2. 盘点所有参考资料并映射覆盖范围
3. 识别模版要求但无参考资料的缺口
4. 通过AskUserQuestion向用户展示大纲草案
5. 根据反馈修订
6. 锁定大纲后再写正文

大纲格式示例：

```text
  1.1 研究背景与意义 [参考: 需求.md 第1节]
  1.2 国内外研究现状 [参考: 文献资料]
  1.3 主要研究内容 [参考: 技能库.md 概述]
  1.4 论文组织结构 [自行组织]

第2章 相关技术简介
  2.1 Minecraft服务端架构 [参考: 源代码 core模块]
  2.2 Purpur/Paper API [参考: 官方文档]
  ...

第N章 总结与展望
  N.1 工作总结
  N.2 不足与未来工作
```
