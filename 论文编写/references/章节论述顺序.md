# 章节论述顺序与职责

本文件改编自Nature级学术写作规范（Academic Phrasebank + Scientific English
Writing）。仅在章节逻辑已确定、需要具体的论述顺序和短语指导时加载。

## 使用原则

先确定这个段落/章节要完成的**修辞任务**，再从这里选取对应的论述顺序和短语。不要在论证逻辑破碎的地方打磨语句。

---

## 绪论

### 绪论：必须回答的问题

1. 这个主题为什么重要？
2. 已经知道了什么？
3. 还有什么缺失或有争议？
4. 本研究要问什么或做什么？

### 绪论：推荐的论述顺序

1. **建立重要性** — 为什么这个话题值得关注
2. **总结已知** — 前人已经确立了什么
3. **识别缺口** — 指出局限、不足或争议
4. **陈述研究目的** — 本研究要做什么
5. **表明价值或方法** — 这项工作的贡献

### 绪论：常用短语族

- 近年来...越来越受到关注
- X是...领域的核心问题
- 前人研究表明...
- 然而，...背后的机制仍然不甚明了
- 很少有研究关注...
- 在此，我们研究了...
- 本工作提供了...

### 绪论：避坑清单

- 不要写冗长的历史铺垫
- 不要在这里详细罗列结果
- 在定义缺口之前不要夸大创新性

---

## 文献综述

### 文献综述：必须回答的问题

1. 哪些研究方向定义了这个领域？
2. 什么已经被确立？
3. 哪些发现存在分歧或不完整？
4. 哪个缺口对本文最重要？

### 文献综述：推荐的论述顺序

1. 描述现有工作的范围
2. 识别主导方法
3. 陈述已确立的结论
4. 指出分歧或矛盾之处
5. 定位缺失的那一块

### 文献综述：常用短语族

- 大量研究工作集中于...
- 多数研究依赖于...
- 前人工作已确立...
- 关于...的发现存在分歧
- 相比之下，很少关注...
- 尚无研究考察...

### 文献综述：避坑清单

- 不要逐篇引用式总结
- 不要把前人工作一概贬低为薄弱

---

## 方法/系统设计

### 方法/系统设计：核心问题

另一个团队能否仅凭这个描述复现工作？（或：仅凭此描述+明确引用的协议？）

### 方法/系统设计：推荐的论述顺序

1. 设计框架或整体架构
2. 使用的技术/材料/数据来源
3. 具体实现过程/步骤
4. 评估指标或结果度量
5. 分析方法和统计工具
6. 伦理考量（如适用）

对于**系统实现型论文**，还需补充：

1. 核心模块分解
2. 关键算法/数据结构
3. 技术选型理由（为什么选A不选B）
4. 系统部署环境

### 方法/系统设计：常用短语族

- 采用...方法进行了...
- 样本/数据采集自...
- X使用...进行量化
- 我们使用...来评估...
- 差异分析采用...
- 所有分析在...中完成
- 系统采用...架构，分为...层/模块
- 核心模块负责...，通过...实现...

### 方法/系统设计：避坑清单

以下模糊表述**绝对禁止**出现，必须替换为具体可复现的信息：

| 禁止用语         | 应替换为                         |
| ---------------- | -------------------------------- |
| 在标准条件下     | 具体的温度/压力/环境参数         |
| 使用常规方法     | 方法的具体名称和版本             |
| 数据经过统计分析 | 统计检验名称、软件版本、参数设置 |
| 差异显著         | 具体的统计量、p值、效应量        |
| 样本随机分配     | 随机化方案的具体描述             |
| 该方法已验证     | 验证实验的具体设计和结果         |

## 结果/实现效果

### 结果：核心问题

在什么条件下观察到了什么？有什么量化支撑？

结果写作应：

- 主要使用过去时态
- 报告观察到什么、在什么条件下、有什么量化支持
- 正确且节制地使用统计数据
- 节制地使用补充材料

**结果回答"发生了什么"，而非"最终意味着什么"。**

### 结果：推荐的论述顺序

1. 引导读者看向图表/表格/实验
2. 陈述主要发现
3. 补充定量细节
4. 指出预期内或意外的模式
5. 仅在有助于澄清结果时才与前人工作对比

### 结果：常用短语族

- 图1展示了...
- 如表1所示...
- 最显著的发现是...
- 与预期相反的是...
- 未观察到显著差异...
- 这些结果与...一致
- 与早期报告相比...

### 结果：避坑清单

- 不要写讨论长度的机制解释
- 不要重复图表中的每个视觉细节

## 讨论

### 讨论：必须回答的问题

1. 主要发现意味着什么？
2. 它如何融入更广阔的领域？
3. 哪些解释是合理的？
4. 什么局限性约束了解释？
5. 从中发现什么？什么不能推出？

### 讨论：推荐的论述顺序

1. 重述主要发现
2. 解释可能的原因
3. 与前人工作比较
4. 指出局限性
5. 陈述意义/启示
6. 如需要，指出未来方向

### 简短规则

- **结果 = 我们观察到了什么**
- **讨论 = 我们如何理解它，以及何时可能失效**

### 讨论：常用短语族

- 综合来看，这些发现表明...
- 一种可能的解释是...
- 这种差异可能反映了...
- 这些结果应谨慎解读，因为...
- 这意味着...
- 进一步的研究需要确定是否...

### 句型区分（重要！）

结果段的句子通常**报告**：

- 检测到... / 增加... / 显示... / 使能... / 实现...

讨论段的句子通常**解读**：

- 可能反映... / 表明... / 可能指示... / 很可能是由于... / 或许促进...

不要让结果段自然滑入讨论句法，除非这种转换是有意的。

---

## 结论

采用三段式收尾：

1. 重申核心贡献
2. 总结关键证据或成果
3. 陈述意义并给出边界

**不要在结论中引入新数据。** 始终在此执行"过度声明检查"。

### 结论：常用短语族

- 本研究旨在...
- 本研究发现...
- 这些结果扩展了我们对...的理解
- 尽管存在这些局限...
- 未来研究需要...

### 结论：避坑清单

- 不要引入新实验
- 不要以空泛的赞美结尾

---

## 摘要

摘要是迷你论文：

`背景/问题 → 缺口/目标 → 方法 → 关键结果 → 意义`

必须回答：

1. 解决了什么问题？
2. 如何解决的？
3. 发现了什么？
4. 为什么读者应该关心？

保持选择性。如果一个细节不影响编辑初审，它大概不属于摘要。

---

## 标题

好的标题应该：

- 让读者知道期待什么
- 避免不必要的技术术语
- 易于搜索
- 有数据支撑
- 在不失可信度的前提下引发兴趣

**用"引人好奇且有可信度"的方式，而非空洞的小聪明。**
