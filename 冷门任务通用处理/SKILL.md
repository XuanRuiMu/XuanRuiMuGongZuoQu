# 冷门任务通用处理

处理AI缺乏专业知识的冷门任务。核心思路：**先研究后执行**——通过联网搜索和GitHub工具发现，弥补知识盲区，重新规划任务方案，然后自动执行。

## 铁律

| 规则                      | 内容                                                                              |
| ------------------------- | --------------------------------------------------------------------------------- |
| 禁止跳过研究阶段          | 任何冷门任务必须先完成联网研究和工具发现，禁止凭已有知识直接执行                  |
| 禁止假设                  | 需求不清立即用AskUserQuestion询问；工具/API可用性必验证；不造mock                 |
| 禁止幻觉                  | 所有事实必须来自联网搜索结果或GitHub真实项目；引用必真实存在                      |
| 必须多源验证              | 关键信息至少2个独立来源交叉验证；单一来源的信息标注"未验证"                       |
| 必须用AskUserQuestion交付 | 研究完成、执行方案确定、遇到关键决策点、任务完成时必须用AskUserQuestion向用户报告 |
| 必须先规划后执行          | 输出结构化执行方案并获用户确认后，方可进入自动执行阶段                            |
| 遇到关键决策点必须暂停    | 执行中遇到不可逆操作、破坏性操作、需要用户判断的分支时，必须暂停询问              |

## 阶段跳过规则

| 情况                       | 可跳过的阶段 | 条件                                         |
| -------------------------- | ------------ | -------------------------------------------- |
| 任务已有成熟skill匹配      | 阶段2-3      | 用户明确指定使用某skill，且该skill能覆盖任务 |
| 用户已提供完整研究资料     | 阶段2        | 用户提供了足够的文档/链接/教程               |
| 纯执行类任务（方案已明确） | 阶段4        | 用户已给出完整步骤，只需执行                 |

**跳过前必须用AskUserQuestion向用户确认**："检测到[情况]，建议跳过[阶段名]直接进入[下一阶段]，是否同意？"

## 完整工作流

````text
阶段1：需求理解与澄清 ──→ 阶段2：联网研究 ──→ 阶段3：GitHub工具发现 ──→ 阶段4：重新规划 ──→ 阶段5：自动执行 ──→ 阶段6：交付确认
    ↓ AskUserQuestion        ↓ WebSearch+WebFetch     ↓ npx skills find+GitHub搜索    ↓ 综合研究发现        ↓ 按方案执行         ↓ AskUserQuestion
    ↓ 查看文件和文件夹        ↓ 多波探索+差距评估       ↓ 评估工具质量                ↓ 结构化方案          ↓ 关键决策点暂停
```text

### 阶段1：需求理解与澄清

**目标**：完全理解用户的冷门任务需求，包括所有提及的文件和文件夹

1. **收集所有需求信息**：
   - 仔细阅读用户描述，提取核心任务意图
   - 识别用户提及的所有文件路径和文件夹路径
   - 识别任务涉及的工具、平台、格式、环境

2. **查看提及的文件和文件夹**：
   - 对用户提及的文件，用Read工具读取内容
   - 对用户提及的文件夹，用LS工具查看结构
   - 理解文件/文件夹的用途和当前状态

3. **多轮扫描策略**（优化点#6，适用于文件/磁盘类任务）：
   - **轮次1（顶层）**：扫描目标路径顶层目录，找出>500MB的大目录
   - **轮次2（AppData深层）**：深入`AppData\Local`、`AppData\Roaming`子目录，找应用缓存
   - **轮次3（系统级）**：扫描`ProgramData`、`Windows\WinSxS`、`Windows\Installer`等系统目录
   - **轮次4（程序文件）**：扫描`Program Files`、`Program Files (x86)`中的工具缓存
   - 每轮扫描结果按大小降序排列，标注类型（缓存/日志/数据/安装文件）
   - 使用`Get-ChildItem -Recurse`时注意`Measure-Object`对空目录报错（非致命，忽略即可）

4. **磁盘扫描效率优化**（优化点#4）：
   - 大目录（>10GB）避免用`Get-ChildItem -Recurse`，改用`robocopy /L /S /BYTES`快速统计
   - 或使用WizTree/TreeSize命令行版本（若已安装）
   - 缓存扫描结果，避免对同一目录重复计算

5. **识别需求歧义**：
   - 任务目标是否明确？（要达成什么结果）
   - 输入输出是否清晰？（从什么到什么）
   - 约束条件是否清楚？（不能做什么、必须做什么）
   - 环境信息是否完整？（操作系统、工具版本、权限等）

6. **有歧义则用AskUserQuestion询问**：
   - 每次只问最关键的1-4个问题
   - 每个问题提供推荐选项
   - 等待用户反馈后再继续

7. **输出**：清晰、无歧义的任务需求陈述

---

### 阶段2：联网研究

**目标**：通过大量联网搜索，收集处理该冷门任务所需的知识和方法

**详细方法见** [references/联网研究方法.md](references/联网研究方法.md)

1. **分解研究子问题**：将任务拆解为3-7个研究子问题
   - 这个任务是什么？属于什么领域？
   - 业界标准做法是什么？
   - 有哪些工具/方法可以完成？
   - 有哪些注意事项和坑？
   - 成功标准和验证方法？

2. **多波探索搜索**（借鉴research-web）：
   - **波次1**：对每个子问题用WebSearch搜索（2-3个关键词变体）
   - **收集发现**：记录关键信息、URL、数据点
   - **评估差距**：是否有子问题未充分覆盖？
   - **波次2+**：针对差距启动新波次，直到关键信息充分

3. **深度阅读关键源**：
   - 对最有价值的3-5个URL，用WebFetch获取完整内容
   - 不依赖搜索摘要，获取一手信息

4. **多源交叉验证**：
   - 关键信息至少2个独立来源验证
   - 标注信息置信度（高/中/低）

5. **输出**：结构化研究发现报告（含来源URL）

---

### 阶段3：GitHub工具发现

**目标**：在GitHub生态中搜索可用的skill、MCP、脚本等工具

**详细方法见** [references/GitHub工具发现方法.md](references/GitHub工具发现方法.md)

1. **搜索相关skill**：
   - `npx skills find "<关键词>"` 搜索skills生态
   - GitHub搜索：`https://github.com/search?q=skill+<关键词>+SKILL.md&type=code`
   - 浏览 awesome-claude-skills 等聚合仓库

2. **搜索相关MCP服务器**：
   - GitHub搜索：`mcp server <关键词>`
   - 查阅 MCP 官方目录和社区目录
   - 评估MCP的安装量和维护状态

3. **搜索相关脚本/工具/库**：
   - GitHub搜索：`<关键词> tool`、`<关键词> script`、`<关键词> cli`
   - 搜索 npm/pip/brew 等包管理器
   - 评估工具的stars、最近更新、文档质量

4. **评估工具质量**（借鉴skill-discovery）：
   - 安装量/星数（优先1K+，100以下谨慎）
   - 来源信誉（官方>知名组织>未知作者）
   - 维护状态（最近6个月有更新）
   - 内容审查（读取SKILL.md/README判断合理性）
   - 兼容性（与当前环境兼容）

5. **输出**：可用工具清单（含安装方式、质量评估、适用场景）

---

### 阶段4：重新规划

**目标**：综合研究发现和工具发现，制定结构化执行方案

**详细模板见** [references/任务规划模板.md](references/任务规划模板.md)

1. **综合分析**：
   - 阶段2的研究发现告诉了我们什么方法？
   - 阶段3的工具发现提供了什么可用工具？
   - 哪些工具可以直接使用？哪些需要安装？哪些需要替代方案？

2. **权限与环境预规划**（优化点#3）：
   - 标注每个步骤的权限要求（普通用户/管理员/系统级）
   - 标注需要关闭的进程（列出进程名、对应应用、关闭影响）
   - 标注不可逆操作（需用户二次确认）
   - 需要管理员权限的操作 → 生成独立`.bat`/`.ps1`脚本供用户手动以管理员身份运行
   - 需要关闭进程的操作 → 提供"关闭→清理→重启"完整流程

3. **制定执行方案**（结构化文档）：

    ```markdown
    # [任务名] 执行方案

    ## 任务目标
    [清晰的目标陈述]

    ## 研究发现摘要
    - 关键方法1：[描述]（来源：URL）
    - 关键方法2：[描述]（来源：URL）

    ## 可用工具
    | 工具 | 类型 | 安装方式 | 适用场景 | 质量评估 |
    |---|---|---|---|---|

    ## 执行步骤
    | 步骤 | 操作 | 权限要求 | 需关闭进程 | 可逆性 | 验证方法 |
    |---|---|---|---|---|---|
    | 1 | [操作] | 普通/管理员 | [进程名] | 可逆/不可逆 | [方法] |

    ## 关键决策点（需用户确认）
    - [决策点1]：[描述] — 影响：[什么]

    ## 风险与注意事项
    - [风险1]：[描述] — 缓解：[措施]

    ## 成功标准
    - [标准1]
    - [标准2]
    ```

4. **用AskUserQuestion向用户确认方案**：
   - 展示方案摘要
   - 询问是否同意执行，或需要调整

---

### 阶段5：自动执行

**目标**：按确认的方案自动执行，遇到关键决策点暂停询问

1. **按步骤执行**：
   - 严格按阶段4的执行步骤顺序操作
   - 每完成一步，验证成功标准
   - 验证失败则暂停，用AskUserQuestion报告问题

2. **文件删除方法选择**（优化点#1，沙箱环境适配）：
   - **禁止使用** `Remove-Item`（在trae-sandbox等沙箱环境中可能假成功，不真正删除文件）
   - **必须使用** .NET方法确保真实删除：
     - 删除文件：`[System.IO.File]::Delete($path)`
     - 删除目录：`[System.IO.Directory]::Delete($path, $true)`（递归删除）
   - **回退策略**：若.NET方法因进程占用失败，逐文件删除：

     ```powershell
     Get-ChildItem $path -Recurse -File -Force | ForEach-Object {
         try { [System.IO.File]::Delete($_.FullName) } catch {}
     }
     [System.IO.Directory]::Delete($path, $true)
     ```

   - **禁止使用** `cmd /c`（在沙箱中被安全策略阻止）

3. **进程占用预检测**（优化点#2）：
   - 清理前用`Get-Process`检测目标应用是否运行
   - 若运行中，用AskUserQuestion询问用户：
     - 选项A：自动关闭应用→清理→提示用户重启应用
     - 选项B：跳过此项，记录到"未清理清单"
     - 选项C：生成脚本供用户后续手动执行
   - 常见应用进程映射：WPS→wps.exe, 微信→WeChat.exe, Edge→msedge.exe, Docker→Docker Desktop.exe

4. **清理验证机制**（优化点#5）：
   - 每批清理前后对比`Win32_LogicalDisk.FreeSpace`
   - 不依赖删除函数返回值，以磁盘空间实际变化为准
   - 验证公式：`实际释放 = 清理后FreeSpace - 清理前FreeSpace`
   - 若实际释放≈0但报告删除成功 → 判定为沙箱假删除，切换.NET方法重试

5. **关键决策点暂停**（必须用AskUserQuestion询问）：
   - 不可逆操作（删除、覆盖、格式化等）
   - 破坏性操作（修改系统文件、注册表等）
   - 需要用户判断的分支（多种方案选择）
   - 执行结果与预期不符
   - 发现方案中未预见的问题

6. **异常处理**：
   - 工具失败：尝试替代方案，失败则暂停报告
   - 权限不足：生成管理员脚本供用户手动执行
   - 结果异常：暂停报告，不继续执行
   - 进程占用：按预检测流程处理

---

### 阶段6：交付确认

**目标**：向用户正式交付任务结果

1. **验证最终结果**：
   - 对照阶段4的成功标准逐项验证
   - 记录验证结果

2. **用AskUserQuestion向用户交付**，包含：
   - 完成了什么任务
   - 使用了哪些工具和方法
   - 执行结果（成功/部分成功/失败）
   - 验证结果
   - 遗留问题（如有）
   - 产生的文件/变更

3. **询问用户下一步计划**
````
