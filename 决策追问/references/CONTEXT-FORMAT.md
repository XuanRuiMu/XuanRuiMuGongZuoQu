# CONTEXT.md格式

## 结构

```md
# {上下文名称}

{一两句话描述这个上下文是什么以及为什么存在。}

## 语言

**订单**: {术语的简洁描述} _避免_：购买、交易

**发票**: 交付后发送给客户的付款请求。 _避免_：账单、付款请求

**客户**: 下订单的个人或组织。 _避免_：客户方、买方、账户

## 关系

- 一个**订单**产生一个或多个**发票**
- 一个**发票**恰好属于一个**客户**

## 示例对话

> **开发者：**"当一个**客户**下**订单**时，我们立即创建**发票**吗？"
> **领域专家：**"不——**发票**只有在**履约**确认后才生成。"

## 标记的歧义

- "账户"被用来同时表示**客户**和**用户**——已解决：这是两个不同的概念。
```

## 规则

- **要有主见。**当同一概念存在多个词时，选择最好的一个，将其他列为应避免的别名。
- **显式标记冲突。**如果术语被模糊使用，在"标记的歧义"中指出并给出明确的解决方案。
- **保持定义紧凑。**最多一句话。定义它是什么，不是它做什么。
- **展示关系。**使用粗体术语名称，在显而易见处表达基数。
- **只包含此项目上下文特有的术语。**通用编程概念（超时、错误类型、工具模式）不属于这里，即使项目大量使用它们。添加术语前先问：这是此上下文独有的概念，还是通用编程概念？只有前者属于这里。
- **在自然分组出现时将术语归入子标题。**如果所有术语属于同一个连贯领域，平铺列表即可。
- **写一段示例对话。**开发者和领域专家之间的对话，展示术语如何自然交互，并澄清相关概念之间的边界。

## 单上下文 vs 多上下文仓库

**单上下文（大多数仓库）：**仓库根目录一个 `CONTEXT.md`。

**多上下文：**仓库根目录的 `CONTEXT-MAP.md` 列出上下文、它们的位置以及它们之间的关系：

```md
# 上下文映射

## 上下文

- [订单](./src/ordering/CONTEXT.md) — 接收和跟踪客户订单
- [账单](./src/billing/CONTEXT.md) — 生成发票和处理付款
- [履约](./src/fulfillment/CONTEXT.md) — 管理仓库拣货和发货

## 关系

- **订单 → 履约**：订单发出 `OrderPlaced` 事件；履约消费它们以开始拣货
- **履约 → 账单**：履约发出 `ShipmentDispatched` 事件；账单消费它们以生成发票
- **订单 ↔ 账单**：共享 `CustomerId` 和 `Money` 类型
```

技能推断适用哪种结构：

- 如果 `CONTEXT-MAP.md` 存在，读取它以找到上下文
- 如果只有根目录 `CONTEXT.md` 存在，单上下文
- 如果两者都不存在，在第一个术语被确定时延迟创建根目录 `CONTEXT.md`

当存在多个上下文时，推断当前话题与哪个上下文相关。如果不清楚，询问。
