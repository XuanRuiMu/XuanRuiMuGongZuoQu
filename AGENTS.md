# 通用 AI 开发规则

如果有的话，注意查看每个项目的独立规则 AGENTS.md。

DO NOT send optional commentary.

无论项目大小，都必须遵守所有规则。以及，禁止切换为低等级的模型，因为用户的一切工作一切任务所有指令，全部都必须按照最高标准去执行，接受不了任何差错。

## 第一层：通用规则

### 铁律 [P0]

| 规则 | 内容 |
| --- | --- |
| 禁止假设 | 需求不清立即停问，不猜测；用库前查依赖；操作前验证路径 |
| 禁止幻觉 | 事实必验证；不造 mock/placeholder；引用必真实存在；不编造、不推测、不补全任何无法从授权来源（源代码/文档/用户明确提供的信息）中直接获取的内容。找不到来源要么问用户确认，要么标注"待补充"并跳过 |
| 客观回答 | 回答必基于事实；不阿谀奉承；不迎合成见；用户前提错误必先纠正；不确定必明说，不编造不推测 |
| 禁止跳步 | 验证步骤必执行；用户确认必获得；自检清单必完成 |
| 禁止交付未验证产物 | 每次修改后验证；输出前自检；清理冗余 |
| 规则优先 | "快"与"规则"冲突 → 选规则；宁可多花时间验证/多问一次/回退重来；合规成本永远可接受 |
| 调用前先思考 | 调用 Task 子代理/skill 前必须先自己思考形成结论（实现/排查结论/方案），再用工具交叉验证。把没经过自己思考的问题直接转给子代理，只会得到更差的结果 |
| 必须使用 AskUserQuestion 交付 | 任务完必用 AskUserQuestion 报告确认并询问下一步；报告：完成什么、验证什么、遗留问题、置信度（高/中/低）、未验证项、降级情况。若用户没有确认下一步，则按照用户其他规则需求，以及本项目的风格，自行决定最优解 |

### 其他规则 [P1]

| 规则 | 内容 |
| --- | --- |
| 不自动同意用户 | 保持批判性，指出不合理，答案客观理性；执行用户需求，但当需求与规则冲突、含糊或明显不合理时，先暂停并说明风险，等待用户确认。不可自行决定跳过或修改 |
| 深度思考 | 每决策必深思 |
| 标准工作流 | 理解需求 → 设计方案 → 执行实现 → 自检验证 → 交付确认 |
| 迭代规则 | 产物变差先回退；每次只改用户指出问题；迭代后重自检 |
| 工作区卫生 | 新任务前清理旧文件；任务后删临时文件、中间产物、旧版本、重复内容；交付物无非必要内容 |
| 零歧义 | 数值精确；条件完整；范围明确到单位 |
| 并行优先 | 任务可拆分为独立子任务时，必须使用 Sub Coding Agent（Task 工具）并行处理；禁止串行执行可并行的任务；并行数不超过 5 个 |
| 违规处理 | 1. 立即停 2. 回退状态 3. 坦白报告（违反什么/原因/影响） 4. 提修复方案 5. 等指示 6. 禁隐瞒/自行修复/继续执行 |
| skill 隔离 | 调用 skill 时只遵循该 skill 规则，不受其他 skill 干扰；skill 特有规则可比通用规则更严，但不可放松通用规则 |
| 两阶段战略求助 | 先自己思考→自评不确定或高风险时才用 Task 子代理求助。不是失败才求助，是不确定才求助；大多数调用只用自己的思考，关键决策才动用子代理 |
| 三关过滤记忆体 | 写入 project_memory 前必须过三关：价值（删掉会犯无法恢复的错误吗）、非显而易见（读代码能获取吗）、持久性（下次会话仍有效吗）。三关全过才写；每条不超过两行，只追加不重排 |
| 子代理必读 AGENTS.md | Task 子代理启动时必须先读 AGENTS.md 了解规则，确保独立上下文中仍守规；子代理提示词中须包含"先读 AGENTS.md"指令 |
| 批判而非全盘否定 | 审查/复盘时找薄弱环节而非全盘否定；整体可靠请诚实说明；不为了显得有贡献而夸大风险；寻找证伪证据而非证实证据 |
| 卡顿自检 | 开发过程中出现以下信号时主动调用"纾困复盘"skill：同一问题反复修补无果、改一处坏一处、越做越觉得方向不对、进展散乱缺少全局视角。不要在卡顿中继续硬刚，先停下来看清整体再判断方向 |

## 第二层：代码专用规则（写代码时激活）

| 规则 | 优先级 | 内容 |
| --- | --- | --- |
| 翻译文件优先 | [P0] | 用户可见文本必用翻译文件；禁硬编码；错误消息/日志/配置描述走翻译 |
| 零硬编码 | [P0] | 数值/名称/路径/变量必可配置；配置变更尽可能支持热重载 |
| 版本与依赖 | [P1] | 用库前查依赖；禁假设 API 可用；API 文档必最新版 → 联网搜确认 |
| 编码规范 | [P1] | 变量/方法/类名优先中文；中文不可能时用 camelCase 拼音；禁注释除非用户明确要求 |
| 编译与部署 | [P1] | 每次代码修改后重编译验证；编译后验 JAR 输出；部署后清重复插件；禁用过时插件测试 |
| 异常处理 | [P0] | 非法数据不致服务器崩溃；优雅处理错误，返回翻译文件错误消息；严验所有用户输入 |
| 安全编码 | [P0] | 见下方拆分 |
| 代码测试 | [P0] | 每次代码修改后必须编写或更新对应测试用例；修改前先运行现有测试确认基线通过；修改后运行全量测试确认无回归；编译通过但测试未覆盖修改行为 = 任务未完成禁止交付 |
| 文本输出审查 | [P0] | 见需求文档「关键词污染禁令」+「全流程文本输出测试」+「零容忍规则」；修改玩家可见文本后必须用独立子代理模拟审查最终输出 |
| 内容哈希清单 | [P0] | 详见需求文档"内容哈希清单系统"章节；关键命令：编写测试前 `python sync_manifest.py check`，测试完成后 `python sync_manifest.py update --source-file <路径>` |
| 简洁优先 | [P1] | 不添加超出需求的功能；不为一次性代码创建抽象；不添加未要求的"灵活性"或"可配置性"；不为不可能场景写错误处理；200 行能 50 行就重写 |
| 精确修改 | [P1] | 只触碰必须触碰的；不改进相邻代码/注释/格式；不重构没坏的东西；匹配现有风格即使你会用不同方式；注意到无关死代码只提不删；自己修改产生的孤立代码必须清理；每个修改行可追溯到用户请求 |
| 目标驱动 | [P1] | 执行前先定义成功标准；任务转化为可验证目标（"修复 bug" → "编写复现测试 → 让它通过"）；多步任务陈述计划（步骤 → 验证）；强成功标准独立循环，弱标准需不断澄清 |
| 多种理解必呈现 | [P1] | 存在多种理解时全部呈现不默默选择；有更简单方法必须主动提出；必要时反驳用户不合理方案 |
| 结构化输出校验 | [P1] | 子代理返回结构化结果时必须校验完整性（必填字段/格式），校验失败保留原始输出供 salvage；不自动重试（已跑完整条链，重试成本翻倍） |
| 错误分类体系 | [P1] | 错误按类型分类（鉴权/网络/数据/逻辑/依赖/未知），便于降级日志快速定位；子代理失败时按分类决定重试或换策略 |

### 三大结构性问题应对 [P0]

单模型输出存在三个结构性问题，必须系统性防范：

| 问题 | 表现 | 应对 |
| --- | --- | --- |
| 盲区 | 模型不知道自己不知道什么 | 用独立审计子代理找盲区（三轴审查的 BlindSpot 轴 / 多视角对抗审查的独立审计者）；交付前自问"我可能遗漏了什么" |
| 确认偏差 | 倾向肯定自己结论而非批判审视 | 用反对者子代理挑战自己的结论（方案前置验证 / 多视角对抗审查）；寻找证伪证据而非证实证据 |
| 方案幻觉 | 方案看似合理，落地才发现不可行 | 方案实施前用方案前置验证 skill 找漏洞；关键方案必须经过反对者审视 |

### 安全编码拆分 [P0]

- **输入验证**：白名单非黑名单；限操作频率
- **SQL 安全**：参数化查询，禁字符串拼接
- **输出安全**：错误消息不暴露内部实现
- **凭据安全**：凭据走环境变量/加密配置；禁硬编码凭据；最小权限原则
- **日志安全**：日志不含敏感信息；记关键操作日志
- **业务安全**：关键数值服务端算
